## Unified ReactiveView: Compiler-Driven Migration

**Context**: The current `BlockRefView` and `ReactiveCollection` split creates a DAG cycle in `wire_collection_drivers` and causes ~370 rebuilds/sec. This plan unifies them into a single `ReactiveView` type with an inner enum encoding the mode (Block, Collection, Static), eliminating the cycle and making failures visible rather than partial.

**Approach**: Delete old code first in a worktree, let compiler errors guide the rewrite. No phases, no strangler fig — the non-compiling state is isolated and the compiler becomes the migration checklist.

### Step 0: Create Worktree
Branch to `federated-whistling-falcon-rewrite` worktree. Isolates non-compiling state. Other sessions unaffected.

### Step 1: Delete Old Code (Pure Removal)
Delete:
- `holon-frontend/src/reactive_view/block_ref.rs` (entire file)
- `holon-frontend/src/reactive_view/collection.rs` (entire file)  
- `holon-frontend/src/reactive_view/child_entity_manager.rs` (entire file)
- `watch_collection_drivers()` and `wire_collection_drivers()` from `holon-frontend/src/reactive_view/mod.rs`
- `ReactiveViewKind::Collection { ... }` and `ReactiveViewKind::BlockRef { ... }` variants

Compiler will emit ~50+ errors. This is the migration checklist — each error points to code that needs adapting.
