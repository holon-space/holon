## Diagnosed Materialized View Panic in holon-live Startup

The app was crashing on startup with no data shown — investigation revealed a database panic caused by stale materialized views that Turso couldn't resolve.

**What happened:**
- Examined `/tmp/holon.log` and found panic at line 11280: `PANIC: Failed to open database: DatabaseError("Internal error: Cannot resolve materialized view dependencies (possible circular dependency): watch_view_2206789c2d25be5a")`
- Traced the issue to `TursoBackend::open_database()` in `lifecycle.rs:62`, which attempts to resolve materialized view dependencies during DB open
- Discovered secondary issue: `watch_view_15d1b245264ba81d` had incompatible version/missing DBSP state table, indicating a schema mismatch between sessions
- Confirmed that the existing `drop_stale_views()` cleanup (which queries `sqlite_master WHERE type='view'`) runs **after** the panic — too late to help

**Challenges:**
- Turso materialized views aren't regular SQLite views — they're tracked differently and not visible to the `sqlite_master` query used in cleanup
- The panic occurs during `open_database()` initialization, before any app-level cleanup routines execute
- Stale views from previous schema versions accumulate in the persistent database file

**Decisions:**
- Two solutions proposed: (1) delete the database + WAL files (`holon.db` and `holon.db-wal` in `~/.config/holon/`) to force recreation, or (2) implement pre-open cleanup that drops stale matviews before Turso tries to resolve dependencies
- Option 1 is immediate; option 2 requires code-level fix in `open_database()` or a new pre-open step

**Status:** Root cause identified; awaiting decision on whether to delete stale DB or implement permanent code-level fix.
