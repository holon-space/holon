## Virtual Child Implementation Status Investigation

**Context**: The virtual child feature (injectable synthetic row for creating new collection items) was partially implemented in a previous session but never made visible in the UI. Investigation revealed the infrastructure is complete but disconnected from the render path.

**What happened**:

- Located the prior session (`0aa5979a-046c-4068-b806-97e223fd9894`) and verified all backend pieces exist: `VirtualChildRowProvider`, `VirtualChildSlot`, `virtual_child_slot_from_arg`, and `resolve_virtual_parent`
- Traced the render path through `block_domain.rs:collection_render_from_profile` and discovered `virtual_parent: true` is never injected into collection expressions
- Found that `resolve_virtual_parent` (in holon-frontend) handles the live_query path but the live_block path has no equivalent substitution
- Identified crate dependency constraint: holon-frontend depends on holon, so `block_domain.rs` cannot call `resolve_virtual_parent` directly
- Confirmed the original design decision: inject `virtual_parent: true` into render DSL (not programmatic injection), then substitute `true → entity_uri` at interpretation time

**Challenges**:

- **Architectural split**: The `true → entity_uri` substitution logic exists in holon-frontend but needs to happen in holon's block_domain.rs — requires re-implementation or refactoring
- **Selective enablement**: Only editable collection variants should get virtual_parent, requiring careful configuration in `collection_profile.yaml`

**Status**: Full picture documented in `devlog/virtual_child_status.md` with file locations and line numbers. Ready for design discussion on how to resolve the crate dependency constraint.
