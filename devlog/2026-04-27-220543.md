## Board Widget Integration and Drag-State Bug Fix

Added render functions to integrate the `Sortable` board visualization from the gpui-component fork into the holon-frontend shadow and GPUI builder pipelines, then debugged and fixed a critical drag-state persistence issue.

**What happened:**
- Created shadow builder (`crates/holon-frontend/src/shadow_builders/board.rs` and `board_lane.rs`) with auto-registration via `builder_registry!`
- Created GPUI builder (`frontends/gpui/src/render/builders/board.rs`) mounting `gpui_component::sortable::Sortable<BoardCard>` per lane with `SortableState` caching
- Added `Mode::Board` tab to design_gallery with demo expression and fallback three-lane UI (To Do / In Progress / Done)
- Gallery build succeeded and app launched cleanly with no GUI-thread errors
- **Bug discovered:** Drag-and-drop was inconsistent—items flickered and disappeared when dragging
- **Root cause:** `SortableState` was keyed by the `ReactiveViewModel` pointer, which changed on every render, allocating fresh state mid-gesture and losing drag tracking
- **Fix:** Switched cache key to stable string `"design-gallery-board:lane:N"` so state persists across render passes

**Challenges:**
- Initial screenshot verification blocked by Mac sleep/screensaver, but app health confirmed via process inspection and log analysis
- Window location difficult to pinpoint in multi-display setup; validated app state through logs and PID inspection instead

**Status:** Board widget wired through shadow→GPUI pipeline with three-lane demo rendering cleanly. Drag-state caching stabilized; ready for manual validation of drag-and-drop interactions and transition to real `board_lane(...)` children with `card(...)` payloads once confirmed working.
