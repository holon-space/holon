## Evaluating Salsa as a DI/Reactivity Replacement

The session explored whether Salsa could replace both FluxDI and futures-signals in the holon backend architecture.

**Current architecture:**
- FluxDI handles startup object graph wiring (TursoBackend, DbHandle, OperationDispatcher, etc.)
- futures-signals powers the reactive middle layer (CDC events → UiEvent → ReactiveQueryResults → Signal<ViewModel>)

**Why Salsa seems attractive:**
- Unifies DI and reactivity into one model: tracked functions derive services from configuration inputs, automatic invalidation on config changes
- Pull-based evaluation avoids glitch problems and enables fine-grained memoization

**Critical mismatches:**
- Salsa is synchronous and pull-based; holon's pipeline is async and push-based (Turso pushes CDC events to downstream subscribers)
- No equivalent to `SignalVec` for incremental list diffing—Salsa tracks value equality, not structural changes
- Salsa's `Database` uses `&mut self` for writes; holon uses `Arc<Mutex<_>>` for multi-writer concurrent access
- Designed for infrequent input changes with expensive derivations; holon has high-frequency small updates where signal propagation overhead is lower

**Better alternative identified:**
Keep futures-signals for the data layer and add reactive service reconstruction by replacing FluxDI's static registration with `Mutable<Arc<Config>>` signals. This is a small, targeted change using the reactive system already embedded in the codebase.

**Status:** Analysis complete—Salsa is unsuitable; current dual-system approach (FluxDI + futures-signals) is architecturally appropriate for holon's event-driven, push-based workload.
