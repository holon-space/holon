## Multi-Frontend Theming, Settings, and Collapsible Sidebars Architecture

We need to add theming, a settings screen, and collapsible sidebars to all `@frontends/` while maximizing code reuse in `@crates/holon-frontend/`.

**What happened**

- Analyzed current architecture: Flutter has basic theme assets (`assets/themes/`), Blinc has `ThemeState` + `ColorTokens`, GPUI uses `Rgba`. Each frontend implements columns/sidebars differently.
- Designed a shared `ThemeColors` trait in `holon-frontend` with direct `impl` blocks producing frontend-specific color types (Blinc's `ColorTokens`, GPUI's `Rgba`, Flutter's `Color`) — avoiding intermediate wrapper structs.
- Planned Phase 1 (Theming): Define `ThemeColors` + `ThemeBundle` in core, each frontend implements theme loading from `assets/themes/*.yaml`.
- Planned Phase 2 (Settings): Generic `HashMap<String, WidgetSettings>` keyed by block ID to track widget state (collapsed, width). Persisted in `holon.toml`. Settings screen is frontend-specific UI.
- Planned Phase 3 (Collapsible Sidebars): Extended `partition_screen_columns` to return block IDs of sidebar regions, so columns builders can look up collapse state from the widget settings map.

**Decisions**

- Use trait impls over wrappers: `ThemeColors` produces native frontend types directly, reducing indirection.
- Generic widget state by block ID: Allows any collapsible widget (sidebar or custom) to store/retrieve state, not just hardcoded sidebars.
- Keep theme assets in `assets/themes/` at workspace root for shared access across frontends.

**Status**

Architecture designed and validated. Ready to begin Phase 1 implementation (core `ThemeColors` trait and theme loading in each frontend).
