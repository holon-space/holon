## Code Review Critique & Plan Refinement: `view_mode` Refactor

The session performed a rigorous senior dev code review on a plan to refactor `view_mode` state management in a large React/GPUI codebase. The user read a high-level plan, then systematically validated it against 108+ actual call sites.

### What happened

- **Traced the full `view_mode` dependency graph** across `ui_generation`, `collection_profile`, predicate system, and shadow builders (tree/table/list)
- **Discovered critical state-loss bug**: original plan stored `Mutable<bool>` directly on view variants, which get recreated on every reactive signal fire. Expanding an item then editing a child block would silently collapse the item via re-interpretation
- **Found performance cliff**: plan's always-interpret-content gating would spawn unnecessary live queries and DB round-trips for collapsed sections in `claude-history.yaml`
- **Identified 7 major gaps**: collection profile dead code, PBT reference render integration, predicate system caching, race conditions, keyboard shortcut removal, and timing between expand + interpret
- **Executed quick validations** on the most dangerous assumptions (state loss, performance impact) by reading actual YAML configs and FDW queries

### Challenges

- **State persistence across re-interpretation**: Solved by moving `Mutable` caches to `BuilderServices`/`ReactiveEngine` (keyed by entity ID), decoupling storage from view node generation—mirroring the existing `UiState` pattern
- **Lazy content interpretation trade-off**: Confirmed gating should remain (don't always interpret), avoiding 50+ unnecessary DB queries per collapsed section
- **Implicit assumptions hidden across multiple systems**: The plan only mentioned ~10 files but `view_mode` touches 108; required reading `collection_render_from_profile`, predicate context flow, and shadow builder variant switching to find all edge cases

### Decisions

- **Keyed Mutable caches** live on the reactive engine, not view nodes, to survive re-interpretation cycles
- **`is_expanded` predicate preserved** for future FDW lazy-load variants—the cache surfaces it via `context_for()`
- **Documented 7 explicit hypotheses** with validation status (quick checks done; longer integrations marked for follow-up tasks)

**Status**: Critique complete and plan rewritten to address state loss, performance cliff, PBT integration, predicate caching, and timing races. Ready for implementation.
