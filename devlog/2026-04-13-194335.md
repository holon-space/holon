## Moving `UserDriver` trait to `holon-frontend` and implementing dual drivers

The test harness needed scrolling support, and there was tension between two use cases: PBTs (want OS-level input for full pipeline coverage) and MCP/live-app inspection (want off-screen safety and no cursor hijacking). Rather than force one solution, the session moved the `UserDriver` trait to a shared location and built two concrete implementations.

### What happened

- **Moved** `UserDriver` trait + `ReactiveEngineDriver` + `HeadlessShadowRouter` from test-only crate into `crates/holon-frontend/src/user_driver.rs`, re-exported for backcompat
- **Refactored** the old `GpuiUserDriver` (enigo-based) â†’ `EnigoUserDriver` in `holon-frontend`, made it take `Arc<dyn GeometryProvider>` instead of GPUI-specific `BoundsRegistry`, gated behind `enigo-driver` feature
- **Created** a real GPUI-native `GpuiUserDriver` in `frontends/gpui/src/user_driver.rs` that dispatches `InteractionEvent` commands on a channel â€” works off-screen, doesn't move the host cursor
- **Added** `InteractionEvent::ScrollWheel` variant + handler in `interaction_event_to_platform_inputs` that produces `PlatformInput::ScrollWheel(ScrollWheelEvent {...})`
- **Wired** DI: `DebugServices` now holds `user_driver: OnceLock<Arc<dyn UserDriver>>`, instantiated in `setup_interaction_pump` with the channel-based GPUI driver
- **Added** MCP `scroll` tool with `ScrollParams { entity_id?, x, y, dx, dy }` that calls `user_driver.scroll_at()` or `scroll_entity()`

### Challenges

**OS-level input (enigo) cannot drive off-screen windows.** `CGEventPost` on macOS is routed by screen coordinate and window focus â€” minimized / hidden / occluded windows don't receive events. The channel-based synthetic dispatch (`PlatformInput::*` directly into the window) bypasses the OS entirely and works regardless of window state. Two drivers, one trait, pick at DI time based on the use case.

**`apply_ui_mutation` still bypasses the window.** Generic `(entity, op, params)` operations have no corresponding real user gesture â€” there's no keybinding to "set focus on block X with cursor at byte Y". This is marked `#[deprecated]` in the trait and only reachable from edge cases (missing geometry, extra params in key chord).

### Decisions

- **Keep `EnigoUserDriver` as a cross-frontend option** rather than GPUI-specific, so any frontend (WaterUI, Dioxus) can opt into it via the `enigo-driver` feature
- **Use channel dispatch for the GPUI impl** rather than trying to wrap GPUI's `VisualTestContext::simulate_event` â€” existing MCP `click` tool already proved the pattern works, and it's simpler than thread-local test context plumbing
