## Fixed Click Handlers in Sidebar Document Items

The sidebar was rendering documents but clicking them produced no effect or logs. Two interconnected bugs in the shadow builder and GPUI renderer prevented click handlers from being wired.

**What happened:**

- Debugged empty click behavior by tracing through `holon-live` logs and the render DSL code
- Discovered shadow builders for `selectable`, `focusable`, `draggable`, and `pie_menu` were pass-through — they returned the child's ViewModel directly instead of wrapping it in the corresponding `NodeKind` variant
- Updated all four shadow builders to create proper wrapper `NodeKind`s (e.g., `NodeKind::Selectable { child: Box::new(child) }`)
- Found that the `action` template arg in `selectable(child, #{action: navigation_focus(...)})` was never extracted or converted to operations
- Modified the `selectable` shadow builder to resolve the `action` RenderExpr against the current row and store it in the entity map using `__action_*` keys
- Updated the GPUI `selectable` renderer to read `__action_*` from entity data, parse the dot-notation name (`navigation.focus` → entity + operation), and dispatch on click

**Challenges:**

- The `OperationWiring` system in ViewModel is always empty — the DSL parser doesn't populate it. Had to use entity map storage instead.
- The `action` arg is a RenderExpr (FunctionCall) that needs row-context resolution. Used `resolve_args` to substitute column references with actual row values.
- Dot-notation parsing: `navigation.focus` splits to entity `"navigation"` and operation `"focus"`, matching the expected `execute_operation` signature.

**Status:** All builds pass, integration tests pass, click handlers now fire and navigate correctly.
