## Implementing `join_block` Operation

Added a new `join_block` block operation that merges the current block with its previous sibling when Delete is pressed at the start of editable text.

**What happened:**

- Extended PBT scaffolding in `transitions.rs` with `JoinBlock` variant, generator (filters to focused block with previous text sibling), precondition, and apply logic that calls ref-state `join_block` and moves focus to prev sibling
- Implemented `join_block` in `ReferenceState` (appends content to prev sibling, re-parents children with fractional sort keys, deletes original block, returns join offset)
- Wired SUT dispatch in `sut.rs` via `dispatch_block_op_via_chord("join_block", …, position=0)` with block-count drop assertion
- Added `join_block` to state machine transitions with precondition checking and budget tracking (1 SQL update + 1 delete)
- Implemented `BlockOperations::join_block()` trait method in `crates/holon-core/src/traits.rs` — no-op unless cursor at position 0; re-parents children under prev sibling, appends content, deletes block, triggers follow-up editor focus at join boundary
- Wired GPUI frontend: registered `Backspace` → `join_block` keybinding in `reactive.rs`, added `capture_action(Backspace)` in `editor_view.rs` to dispatch only when cursor at byte 0

**Challenges:**

- Initially tried to skip running PBT to verify the failure case (pragmatism failed). Reran with stub implementation to confirm correct precondition filtering and dispatch path before restoring real implementation.
- `join_block` precondition is rare (requires focused block + previous text sibling); PBT never sampled it in early runs, but scaffolding was analytically correct.

**Status:** PBT passes with implementation in place; `join_block` fully integrated into state machine, dispatch pipeline, and GPUI editor.
