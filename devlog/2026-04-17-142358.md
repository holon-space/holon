## Code Review: Radiant Dreaming Taco Plan — Refined & Hardened

**Context**: A comprehensive review and refinement of the subtree-sharing feature plan, treating it as a senior code review where critical assumptions, edge cases, and implicit design decisions are surfaced and made explicit.

**What happened**:
- Conducted a severity-ordered critique covering dispatcher collision behavior, authentication gaps, serialization inexpressibility, and construction-order entanglement
- Verified three critical issues against source (`operation_dispatcher.rs:345-375`, `shared_tree.rs:157`) and confirmed silent failures in provider registration and peer-id generation
- Identified nine categories of missing edge-case handling: idempotency under re-share/re-accept, nested mount extraction, peer-id collision detection, and partial-failure cleanup
- Restructured the plan into Phase 1 (MVP with documented gaps) and Phase 2 (security + lifecycle), plus a "decisions to resolve before coding" section (D1–D5)
- Rewrote the accepter-loop task from fictional "factor out" to concrete `IrohAdvertiser` with `CancellationToken`, graceful shutdown, and error semantics
- Updated ticket format to include `v: 1` versioning; restricted Phase 1 `HistoryRetention` to `full`/`none` only (dropped unprintable `Since(Frontiers)`)
- Flagged dispatcher silent-drop as a separate architectural issue (affects all multi-provider scenarios, not just this feature)

**Challenges**:
- **Loro peer-id collision under concurrent joins**: No built-in detection; fixed by deriving peer-id from iroh's persistent `SecretKey` rather than `rand::random()`, making device identity the source of truth
- **`Arc<LoroBackend>` isn't a wiring refactor**: Current seeding is async and order-sensitive; exposing as DI singleton requires rethinking construction order and ensuring all consumers (controller, operation provider, existing doc-store) see the same post-seeded instance
- **Accepter loop doesn't exist**: Test's `sync_pair` is a one-shot helper; had to design a real persistent loop with lifecycle, cancellation, and connection-error recovery
- **Ticket = bearer token with no auth layer**: `iroh_sync_adapter` has no peer allowlist or revocation; added explicit threat-model documentation and Phase 2 pre-shared-secret + ALPN-derivation tasks

**Decisions**:
- Dispatch ops to new entity `"shared_tree"` rather than `"block"` to avoid silent collision with existing `SqlOperationProvider` block operations
- Phase 1 ships only `"full"` and `"none"` retention modes (exclude `"since"` which requires unprintable `Frontiers` serialization)
- Derive peer-ids from iroh `SecretKey` to ensure device-scoped uniqueness and enable peer allowlist/revocation in Phase 2

**Status**: Plan rewritten and hardened with 5 blocking decisions identified, 9 edge-case tasks added, and 3 verified critical issues fixed in-design; ready for implementation task breakdown and architecture review before coding begins.
