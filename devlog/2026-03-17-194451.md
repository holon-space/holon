## Migrated sync_token_store and credential_store to QueryableCache

Replaced raw SQL in two key-value store implementations with type-safe `QueryableCache` wrappers to eliminate boilerplate and improve maintainability.

### What happened

- **Audited all DbHandle usage** across the codebase and identified two refactoring candidates: `sync_token_store.rs` and `credential_store.rs` — both were hand-rolling DDL + raw SQL for simple CRUD operations that `QueryableCache` handles automatically
- **Defined two new Entity types** with `#[derive(Entity)]`: `SyncState` (provider_name → sync_token) and `OAuthCredential` (server_uri → credentials_json)
- **Rewrote sync_token_store.rs**: replaced `DbHandle` + raw SQL with `QueryableCache<SyncState>`; `load_token()` now uses `get_by_id()`, `save_token()` uses `upsert_to_cache()`, `clear_all_tokens()` uses `cache.clear()`
- **Rewrote credential_store.rs** in holon-mcp-client similarly, eliminating `initialize_table()` since `QueryableCache::for_entity()` handles DDL auto-creation
- **Updated DI registration** to handle the new async constructors
- **Added `Clone` impl for QueryableCache** — needed because `TursoCredentialStore` derives Clone
- **Updated sync_states.sql** with `_change_origin TEXT` column to match QueryableCache's generated DDL
- **Verified compilation and tests**: all dependent crates compile cleanly, QueryableCache unit tests pass

### Challenges

- **Async constructor propagation**: `DatabaseSyncTokenStore::new()` became async (returns `Result`), requiring DI registration to use `block_on()` — handled cleanly in one place
- **QueryableCache wasn't Clone**: needed to implement it manually since `TursoCredentialStore` was deriving Clone; verified all fields were Clone-compatible
- **Schema module vs. auto-DDL**: kept `SyncStateSchemaModule` but synchronized its DDL with QueryableCache's expectations (`_change_origin` column, NOT NULL constraints)

### Decisions

- **Kept SyncStateSchemaModule** rather than removing it — provides explicit schema versioning alongside the Entity-driven DDL
- **No schema module for OAuthCredential** — credential store is application-scoped, not core infrastructure, so auto-DDL via QueryableCache is sufficient
- **Plain String fields instead of EntityUri** for entity keys — cleaner and matches the original key-value semantics

**Status**: Both stores fully migrated and compiling. No call sites needed updating (credential_store's OAuth mode isn't yet instantiated in any production code). Ready for integration testing.
