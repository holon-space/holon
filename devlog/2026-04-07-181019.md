## Investigating `INSERT OR IGNORE` Migration in Document Deduplication

The codebase had previously migrated from `INSERT OR REPLACE` to `INSERT OR IGNORE` for certain database operations, but the reasoning wasn't immediately clear. This session focused on understanding the root cause and design decisions behind that change.

**What happened:**

- Searched `claude-history` MCP for context on the `INSERT OR REPLACE` → `INSERT OR IGNORE` migration but got no direct matches
- Checked git history and code directly to find the source
- Located the full explanation in `handoff-pbt-document-dedup.md` devlog
- Identified the root cause: **race condition during concurrent document creation**
- Documented the complete fix chain, including a follow-up bug that emerged from the initial solution

**Root cause:**

Two concurrent `on_file_changed` calls would race through `get_or_create_by_name_chain`, each creating document blocks with `name="index"` but different UUIDs. With `INSERT OR REPLACE`, the second caller silently overwrote the first document, breaking references to the original UUID—a data-destroying operation.

**Challenges:**

- **Race condition masking**: The original `INSERT OR REPLACE` behavior silently corrupted data rather than failing loudly, making the bug hard to detect
- **Follow-up bug (bug #4)**: After switching to `INSERT OR IGNORE`, `LiveDocumentManager::create` still returned the ignored doc's UUID instead of querying the actual row. Fixed by adding a `SELECT` after `INSERT OR IGNORE` in `prepare_create` to return the actual DB row

**Decisions:**

- Added `UNIQUE` index on `(parent_id, name) WHERE name IS NOT NULL` in `blocks.sql` to enforce constraint
- Changed to `INSERT OR IGNORE` so first caller wins; second insert fails silently without destroying data

**Status:** Migration rationale fully documented and understood; both bug #3 (race condition) and #4 (UUID return) resolved through constraint + query approach.
