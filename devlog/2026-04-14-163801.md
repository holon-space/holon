## Decoupling Layout Property Tests from GPUI Using UserDriver

The goal was to make the layout property tests in `frontends/gpui/tests/layout_proptest.rs` reusable across multiple UI frameworks (GPUI, Dioxus, etc.) by leveraging the `UserDriver` abstraction.

**What happened**

- Analyzed the current `layout_proptest.rs` test structure and identified tight coupling to GPUI-specific rendering and assertion mechanisms
- Examined `UserDriver` interface to understand its current capabilities and limitations for cross-framework testing
- Identified key abstraction points: component rendering, layout measurement, and property assertions
- Discussed extending `UserDriver` with layout-specific methods (`measure_layout`, `get_element_bounds`, `assert_layout_property`) to enable framework-agnostic layout testing
- Sketched a generalized property test harness that accepts a UI framework adapter implementing a common layout testing interface
- Proposed separating test logic (property generation and validation) from UI-specific implementation details (rendering and measurement)

**Challenges**

- **Framework differences**: GPUI, Dioxus, and other frameworks have different layout engines and measurement APIs. Resolved by defining a minimal common interface (`LayoutDriver` trait) that each framework implements
- **State management**: Property tests need to recreate components with varying props repeatedly. Required `UserDriver` to support stateless component creation or explicit state reset between iterations

**Decisions**

- Create a new `LayoutDriver` trait abstracting layout measurement across frameworks, implemented per-UI library
- Keep property test logic in a shared, framework-agnostic module; move GPUI-specific details into a GPUI adapter
- Extend `UserDriver` minimally—add methods for querying layout rather than embedding layout logic directly

**Status**

Architecture and interface design discussed; implementation of `LayoutDriver` trait and refactoring of tests into framework-agnostic form ready to begin.
