## Fixed inv16 spurious CDC events in PBT test

The `inv16` invariant (CDC quiescence after settlement) was failing due to redundant writes and Loro echo loops triggering spurious IVM CDC events.

**What happened:**

- Ran `general_e2e_pbt.rs` and identified 1 spurious CDC event after every settlement
- Traced the root cause to `CacheEventSubscriber` redundantly writing to the same `block` table as `SqlOperationProvider`, triggering a second IVM CDC
- Removed the redundant `apply_batch` call in `flush_block_batch` — events are now marked processed without re-writing to SQL
- Added a diff guard (`WHERE col1 IS NOT val1 OR ...`) to `prepare_update` in `sql_operation_provider.rs` to skip no-op UPDATEs from Loro echoes
- Excluded timestamp columns (`updated_at`, `created_at`) from diff guard since `block_to_params` always sets them to `now`
- Fixed spurious CDC from JSON key ordering differences by canonicalizing `properties` with `BTreeMap` in both `prepare_create` and `prepare_update`
- Verified all 3 PBT variants pass: `general_e2e_pbt`, `general_e2e_pbt_cross_executor`, `general_e2e_pbt_sql_only`

**Challenges:**

- The Loro echo loop was subtle: inbound events write to Loro, then Loro's outbound diff produces UPDATEs back to SQL with identical values, triggering unnecessary IVM CDC
- Properties JSON serialization was non-deterministic across code paths, causing the diff guard's string comparison to fail

**Status:** All PBT tests passing; inv16 invariant now holds across all settlement transitions.
