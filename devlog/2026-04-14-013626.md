## V1–V4 validation follow-ups for fail-loud geometry refactor

**Context:** After the fail-loud geometry + `wait_for_element_bounds` refactor shipped, four validation tasks were deferred. This session closed them.

**What happened:**

- **V1 — MCP `scroll` audit.** Disclosure only, see devlog `2026-04-14-010301.md`. Single caller at `frontends/mcp/src/tools.rs:1684-1722` already surfaces driver errors. Zero in-tree tests rely on silent-Ok semantics. Behavior change: `scroll { entity_id: "bogus" }` now returns `internal_error` instead of `Ok`.

- **V3 — `EnigoUserDriver` deletion.** Confirmed zero `::new(` call sites in-tree. Deleted `crates/holon-frontend/src/enigo_user_driver.rs`, the `enigo-driver` feature flag, the `enigo` dep, and all doc references in `user_driver.rs` + `gpui/src/user_driver.rs` + `polling.rs`. `grep -rn EnigoUserDriver` returns zero matches afterwards. `cargo check -p holon-frontend` clean.

- **V4 — `dispatch_intent_sync` + sleep removal.** Added `BuilderServices::dispatch_intent_sync(&self, intent) -> Pin<Box<dyn Future<Output = Result<()>> + Send + '_>>` to the trait, with a default impl that delegates to fire-and-forget `dispatch_intent` (for test stubs where the operation has no observable effect). Overrode it on `ReactiveEngine` to inline-`await` `session.execute_operation(...)`. Migrated `ReactiveEngineDriver::synthetic_dispatch` and `GpuiUserDriver::synthetic_dispatch` to call the new method; dropped both `tokio::time::sleep(Duration::from_millis(10))` fire-and-forget barriers. `general_e2e_pbt` runs post-change: `_sql_only` PASSED (95.8s vs 184.7s baseline — **≈2× faster**), the two known-failing cases fail with the same `"effort"` custom-property fingerprint as before (no new failure modes). Zero `no bounds recorded` or `dispatch_intent_sync` panics.

- **V2 — `gpui_ui_pbt` bounds-miss baseline.** Ran `PROPTEST_CASES=20 cargo test -p holon-gpui --features pbt --test gpui_ui_pbt`. Proptest's first case hit a pre-existing invariant failure at `crates/holon-integration-tests/src/pbt/sut.rs:3470` (`[inv14c/B3] Element 'spacer#11' has zero-area bounds`) after one `ApplyMutation::Update` transition. Coverage is therefore limited. Within that coverage: **zero `no bounds recorded` errors**, i.e. the fail-loud geometry path never fired. The V4 runs against `general_e2e_pbt` (which uses the headless reactive driver) also logged zero bounds-miss errors. No `wait_for_element_bounds` sprinkling is needed for the currently-exercised transitions; whether additional sprinkling would be needed for structural Create/Split/Merge transitions against the GPUI driver remains unanswered until the pre-existing `inv14c/B3` bug is fixed.

**Challenges:**

- The parallel F10 rename (apply_ui_mutation → synthetic_dispatch) had already landed before V4 started, so the 10ms sleep-removal had to happen on top of it without re-adding deprecation scaffolding.
- Mid-session the user called out that plan-file references in source comments are not durable ("will not be available anywhere else than on my machine"); future work should avoid citing local plan filenames in code.

**Decisions & findings:**

- `dispatch_intent` stays fire-and-forget (correct for GPUI click handlers that must not block the UI thread); `dispatch_intent_sync` is the completion-aware variant for tests, MCP tool handlers, and the reactive driver's synthetic path. Preference sets (which run inline without going through `execute_operation`) short-circuit in both methods.
- The ~2× speedup of `general_e2e_pbt_sql_only` from 184.7s to 95.8s is not purely from removing the 10ms sleep — the parallel F10/F2 refactor's quiescence-based `wait_for_index_quiescence` replacing the old `sleep(30ms)` in `send_key_chord` probably contributes most of it. Treat it as a compound improvement over both refactors.
- `EnigoUserDriver` deletion is a pure burn-down; if OS-level input synthesis is ever wanted again, `git log` has the full implementation at the delete commit.

**Deferred / not in scope:**

- The `inv14c/B3` spacer zero-height failure in `gpui_ui_pbt` is a pre-existing layout bug that blocks deep V2 coverage. Separate investigation needed.
- The `"effort"` custom-property round-trip failure in `general_e2e_pbt` / `_cross_executor` predates this work and is not in the scope of the UserDriver refactor.
