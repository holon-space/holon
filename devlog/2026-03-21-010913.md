## Fuzzer Mid-Transaction Matview Verification

Added detection for IVM bugs that only manifest during open transactions, which the existing fuzzer missed by verifying only after COMMIT.

**What happened**

- Added matview verification inside the batch transaction loop in `runner.rs` (lines 503–505), immediately after each statement executes. This catches divergences between Turso matviews and SQLite views while the transaction is open.
- Set default `batch_probability` to 0.3 in `main.rs` (lines 254–258) when `--matview` flag is used without explicit `--batch-probability`. This ensures mid-transaction checks are actually exercised.
- Verified both changes compile cleanly without regressions.

**Why this works**

The existing `verify_matviews_match` oracle already handles mid-transaction state correctly—SQLite's regular views re-evaluate on every SELECT and see uncommitted data on the same connection. No new generators or test patterns needed; just better assertion placement.

**Decisions**

- Reused existing `verify_matviews_match` rather than building separate mid-transaction logic. The oracle already does the right thing.
- Chose 0.3 as default batch probability because it's high enough to trigger multi-transaction sequences (which expose DDL + IVM ordering bugs) without slowing fuzzer runs excessively.

**Status**

Both changes implemented and building. Ready to run fuzzer with `--matview` to catch previously undetected recursive CTE matview bugs occurring mid-transaction.
