## Cross-block cursor navigation (Up/Down at block boundary)

**Context**: The GPUI frontend had full navigation infrastructure but MoveUp/MoveDown actions would stop at block boundaries instead of moving to the adjacent block. Goal: Up at first line → last line of block above; Down at last line → first line of block below, both preserving cursor column.

**What happened**:

- Patched the `gpui-component` fork (`crates/ui/src/input/movement.rs`): Added Zed-style `cx.propagate()` to `up()` and `down()` — when the cursor can't move vertically, the action now propagates to parent views instead of being silently swallowed.

- Added `set_cursor_offset(offset, cx)` public method to `InputState` — allows setting cursor position from `Entity::update()` without needing `&mut Window`.

- Removed boundary-blocking logic from `EditorView::capture_action()` — deleted the checks that stopped MoveUp/MoveDown at block edges. Now actions propagate naturally to parent handlers.

- Extended `BoundsRegistry` in `geometry.rs` with `shadow_dom` and `editor_inputs` fields. Added methods: `set_shadow_dom()`, `editor_input_for()`, `focused_editor_row_id()`, `bubble_input()`, `focused_cursor_column()`.

- Built `ShadowDom` during `AppModel::rebuild()` and registered each editor's `InputState` entity by `row_id` in BoundsRegistry.

- Added parent `on_action` handlers for MoveUp/MoveDown in `HolonApp::render()`. When action bubbles up from a blocked editor, the handler uses `ShadowDom::bubble_input()` to find the target block, computes the cursor offset via `placement_to_offset()`, and transfers focus with correct column preservation.

- Updated `Cargo.toml` to pull the patched `gpui-component` from the remote `holon` branch at commit `5ce62689`.

**Status**: Implementation complete and compiling cleanly. All 67 holon-frontend tests pass. Ready for manual testing with the running app.
