## SQL Reproducer Minimization & Bug Investigation

I investigated and minimized a Turso btree panic (`current_page=-1 is negative`) in holon's sync layer.

**What happened:**

- Located panic in Turso's btree code (triggered by `CREATE MATERIALIZED VIEW` + concurrent block inserts)
- Fixed `extract-sql-trace.py` to handle ANSI escape codes and added `--stop-at`/`--stop-pattern` flags for targeted extraction
- Improved `holon-orgmode` file watcher to respect `.gitignore` and skip `.git/`/`.jj/` using the `ignore` crate
- Created `turso_sql_replay.rs` example to reproduce the panic with SQL statements (1189 → 433 directives, 64% reduction)
- **Identified minimizer bug**: crash detection treats any non-zero exit as "still crashes," so DDL removals with orphaned DML statements falsely appear necessary

**Challenges:**

- `catch_unwind` doesn't cross FFI boundaries; switched to subprocess-based crash detection
- Panic text was interleaved with continuation lines in log output; fixed regex to recognize `thread 'X' panicked` patterns
- Original minimizer was too conservative due to SQL error semantics (removing `CREATE TABLE` while `INSERT` remains looks like a crash)

**Decisions:**

- Implemented two-phase minimization: remove DML statements first, then DDL, to avoid false-positive errors masking unnecessary schema
- Added crash pattern detection to distinguish btree panics from SQL errors (syntax, constraint violations, etc.)

**Status:** Reproducer minimized but still contains unused table definitions. Next: implement chunked removal phase (delta debugging) before 1-minimality pass to reach true minimal test case.
