## Fixing Profile-Based Rendering in GPUI Tree Builder

The main panel was rendering as a flat, non-editable list instead of a tree with editable blocks. The root cause: entity profiles' render expressions were being stripped during data conversion, causing all blocks to fall back to plain text rendering.

**What happened:**

- Investigated why `describe_ui(__root__)` returned empty while backend data was present and correctly structured
- Traced the rendering pipeline through GPUI's `tree` builder, backend shadow builders, and the `BlockRenderCache`
- Identified that `BlockRenderCache::get_or_watch()` at line 61 strips profile information when converting `ResolvedRow` to render context data: `cached.data.iter().map(|r| r.data.clone()).collect()` only kept the `HashMap<String, Value>`, discarding `ResolvedRow.profile`
- Found that `shared_render_block_build()` (lines 384-411) hardcodes rendering logic and never resolves entity profiles, causing all text blocks to render as plain `text` nodes instead of the profile's default render expression
- Confirmed profiles ARE resolved upstream on `ResolvedRow.profile` but lost in the data conversion step

**Decisions:**

- Inject profile render expressions into row data as a special `_profile_render` key rather than refactoring the `RenderContext` API, allowing `render_block` to consume profiles without breaking existing code
- Create a helper function `resolved_rows_to_data()` to centralize the profile injection logic at conversion points

**Status:** Root cause identified; ready to implement profile injection into `BlockRenderCache::get_or_watch()` and update `shared_render_block_build()` to use `_profile_render` expressions.
