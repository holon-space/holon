## ARCHITECTURE.md Enhanced with Three-Tier Event Model

**Context:** The ViewModel layer needed clearer documentation on how input events flow from platform-specific frontends back into shared Rust logic without reimplementing text editing across 7 platforms.

**What happened:**

- Added new "Three-Tier Event Model" subsection after the existing ViewModel description in ARCHITECTURE.md
- Documented the three event handling tiers: Native (platform text stack), Trigger (local O(1) checks), and Sync (debounced backend updates)
- Introduced `InputTrigger` enum struct as part of ViewModel nodes, with concrete examples (`PrefixAtCursor`, `KeyChord`)
- Included a performance table showing that normal typing incurs zero ViewModel overhead
- Walked through the `/` command menu flow as a concrete example of trigger→ViewModel→new subtree
- Clarified the ownership boundary: ViewModel owns mode transitions and semantic actions; View owns text editing and platform interaction
- Listed what does NOT belong in ViewModel (cursor position, IME state, scroll, focus, animation)

**Decisions:**

- **Triggers declared in render DSL** — natural fit alongside existing `OperationWiring`; avoids duplicating input logic across 7 frontends
- **Deferred text sync (Tier 3)** — keeps ViewModel round-trips rare; text content syncs async on blur/timer for persistence, not UI logic
- **Local trigger matching** — View checks triggers O(1) per keystroke; only matching triggers cross the Rust↔frontend boundary

**Status:** ARCHITECTURE.md now documents the complete input event model. Ready for implementation in the next session.
