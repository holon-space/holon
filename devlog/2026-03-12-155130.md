## Fix: CreateStaleLoro + render mutation sync timeout

**Context:** PBT seed `6459ec2` and others failed when a `CreateStaleLoro` transition was followed by a render source mutation (`set_field`). The org file was never synced to disk. Additionally, `BulkExternalAdd` transitions were overwriting ingested file content with stale cached data.

**What happened:**

- Added `warn!` logging at three silent `continue` points in `re_render_all_tracked()` to diagnose where the sync chain breaks
- Discovered root cause: `on_file_changed()` reads from CDC-driven block cache immediately after writing disk changes via `execute_batch()`, before CDC events propagate. This causes the cache to return stale data
- Implemented cache-sync wait using `yield_now()` loop in `on_file_changed()` after `execute_batch()` to allow CDC events to populate the cache
- Added parent_id to `FieldsChanged` event payloads so render mutations route through `on_block_changed()` instead of the slower `re_render_all_tracked()` path
- Updated `extract_doc_ids_from_event()` in `di.rs:770` to extract doc_id from FieldsChanged events

**Challenges:**

- **CDC race between write and read:** After inserting blocks via SQL in `execute_batch()`, the immediate `render_file_by_doc_id()` call read from the CDC-driven cache which hadn't received the new blocks yet. Fixed by waiting for cache population via polling loop with timeout.
- **Event routing complexity:** FieldsChanged events were falling through to `re_render_all_tracked()` which uses the slower name-chain lookup. Resolved by enriching the event payload with parent_id and extending `extract_doc_ids_from_event()` to handle this case.
- **Unrelated compile errors:** WIP code in holon-frontend broke builds; required fixing `RenderContext<Ext>` references in `operations.rs:70` after removing generic parameter.

**Status:** All regression seeds now pass (including `6459ec2`, `BulkExternalAdd` cases, and render mutation scenarios). `wait_for_org_file_sync` completes successfully.
