## Validated Turso IVM Fix: Change Counts Regression

### Context
A Turso IVM fix for change count tracking (commit `f5e02fac`) was suspected to resolve an `IdxDelete` index corruption bug. Testing revealed the fix introduces a severe regression—an infinite loop during IVM processing.

### What happened
- Extracted full SQL trace (1383 statements) from holon crash log (`/tmp/holon-gpui.log`) up to the `IdxDelete` failure
- Initial replays with old Turso (`45cf64e8`) completed successfully in ~1 minute
- After `cargo update` fetched the new commit (`f5e02fac`), the replay **hangs indefinitely** with 100% CPU
- Confirmed the hang is specific to the new commit by testing both versions
- Turso-side `test_ivm_upsert_delete` tests compile and pass, but the regression appears in realistic end-to-end traces

### Challenges
- **Non-deterministic bug**: The original `IdxDelete` error doesn't reproduce with serial SQL replay—requires actor crash or concurrent DDL/DML interleaving
- **Regression detection**: The new commit passes unit tests but introduces a hang in the complex 1383-statement trace, suggesting the infinite loop only triggers under specific IVM state conditions
- **Cargo caching**: Initial replays used cached old Turso; required explicit `cargo update` to fetch new changes

### Decisions
- **Reverted holon to `45cf64e8`**: The old Turso version is stable; the new commit (`f5e02fac`) needs debugging before use
- **Documented the regression path**: Changes in `core/incremental/persistence.rs` (sought state tracking) and `core/vdbe/mod.rs` are the investigation focus—likely a loop condition or cursor lifecycle bug

### Status
Old Turso (`45cf64e8`) is stable and validated against the 1383-statement replay. New commit `f5e02fac` must be debugged in the Turso repo to resolve the infinite loop before re-merging to holon.
