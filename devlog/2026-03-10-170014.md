## Screenshot Tool for MCP Frontend

Added a screenshot capability to the Holon MCP server, enabling Claude to see the running GPUI frontend during UI testing via MCP tool calls.

**What happened:**

- Added `screenshot` MCP tool to `@frontends/mcp/` that captures the currently running GPUI window and returns it as base64-encoded PNG
- Initial implementation used `xcap` crate but hit a blocker: `xcap` hardcodes `OptionOnScreenOnly` in its macOS window enumeration, making it unable to see windows on other macOS Spaces/desktops
- Switched to direct Core Graphics API calls via `objc2_core_foundation` and `objc2_core_graphics` to use `OptionAll`, enabling capture from any desktop
- Tool defaults to PID-based matching (captures the window owned by the MCP server process itself) for reliability, with optional `window_title` parameter for explicit targeting
- Migrated back to `xcap` using the community fork (`nightscape/xcap#feat/macos-offscreen-windows`) once the off-screen window support was implemented

**Challenges:**

- **False positive matching**: Early case-insensitive title/app-name matching captured Zed editor instead of GPUI because Holon project was open in Zed. Solved by using PID matching — since the MCP server runs embedded in the GPUI process, `std::process::id()` reliably identifies the correct window.
- **xcap's hardcoded on-screen-only filtering**: xcap locked windows to the currently visible macOS desktop via `OptionOnScreenOnly`. Worked around by hand-rolling Core Graphics calls with `OptionAll`, then switched to a community fork that fixed this upstream.

**Status:** Screenshot tool is functional — `screenshot {}` captures the GPUI frontend from any macOS desktop and returns it as viewable image content to Claude via MCP.
