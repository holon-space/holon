## Slash Command Menu Integration for GPUI

**Context**: The slash command system (Operation-based command menu with entity parameter collection) was designed and partially implemented in `holon-frontend`. Now it needs to be wired into the GPUI `editable_text` builder, which has evolved significantly and has a TODO comment for this integration.

**What happened**:

- Reviewed `operation_matcher.rs` and `command_menu.rs` — already implemented with unit tests covering operation matching, filtering, and parameter resolution
- Analyzed the GPUI InputState management pattern in `ensure_input_states()` — InputState entities are created per editor, with subscriptions to `InputEvent::Blur` for save-on-blur
- Identified the core challenge: `CommandMenuController` needs persistent per-editor state (filter text, selection index, search results) that survives GPUI re-renders, but the builder function only receives immutable `&ViewModel`
- Checked the TODO on line 44 of `editable_text.rs` and the InputEvent subscription pattern used for Blur events

**Challenges**:

- **State persistence across re-renders**: `CommandMenuController` tracks mutable state (current filter, selection, search results) as the user types. GPUI builders run on every keystroke, so state can't live in the builder closure. Solution: promote `EditInfo` from a local throwaway struct to a persistent entity-like state object stored alongside `InputState`, keyed by `entity_id + field_name`.
- **Subscription wiring location**: The builder can't create subscriptions (no `Window` access), but `ensure_input_states()` has full context. Solution: subscribe to `InputEvent::Change` in `ensure_input_states()` where `InputEvent::Blur` is already subscribed, and route text changes to the controller.
- **Menu rendering**: The menu overlay (filtered operation list + param collection UI) must be positioned near the cursor and re-render as state changes. Solution: render the menu in the builder using filtered matches from controller state, positioned via `absolute` positioned div.

**Decisions**:

- Store `CommandMenuController` state in a new `CommandMenuState` struct alongside `InputState` in the `editor_states` map. Keyed by `(entity_id, field_name)` tuple for isolation across multiple editors on the same page.
- Wire `InputEvent::Change` subscription in `ensure_input_states()` to call a new action `update_command_menu()` which updates the controller and triggers a re-render via `ctx.notify()`.
- Render the menu overlay conditionally in `editable_text.rs` using the controller's current `MenuState` — show operation list when filtering, show entity search UI when collecting a parameter.

**Status**: Ready to implement the subscription wiring in `ensure_input_states()` and the menu rendering in `editable_text.rs`. The controller logic is complete; this session focuses on GPUI integration plumbing.
