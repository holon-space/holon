## jj-rebase Skill Documentation and Workflow

Comprehensive documentation for safely rebasing all user-authored branches onto the latest upstream main with automatic conflict resolution in an isolated clone.

**Context**: The jj-rebase skill automates the complex, error-prone process of rebasing multiple feature branches in jujutsu, handling divergent commits, pre-existing conflicts, and large upstream gaps with both bulk and incremental strategies.

**What happened**:
- Documented complete skill workflow with 12 sequential steps from clone creation through validation and cleanup
- Defined critical revset knowledge: correct branch detection using `roots(author(...) ~ ::main)` instead of `roots(main..)`, and divergent commit identification via immutability/parent properties (never `/N` offsets)
- Specified four bundled scripts with distinct purposes: `jj-rebase-clone.sh` (isolated setup), `jj-cleanup-divergent.sh` (stale copy removal), `jj-imerge2.sh` (incremental rebase with auto-resolve), `jj-rebase-validate.sh` (cross-diff validation)
- Provided detailed pre-rebase conflict resolution strategy: resolve root conflicts first, use `jj restore` only for files the descendant doesn't modify, leverage `jj new + edit + jj squash -u` pattern
- Included bulk vs. incremental rebase decision logic based on first-parent merge commit count (not total DAG commits)
- Documented mandatory validation step 6c (`PRE_REBASE_OP` recording) and step 9 (cross-diff validation to detect lost code)
- Covered edge cases: divergent commits, mega-merges, immutable copies, empty commits, and conflict cascade behavior

**Challenges**:
- Revset correctness: `roots(main..)` fails when main has advanced past fork point; required `~ ::main` exclusion operator
- Divergent commit ambiguity: `/N` offset numbering is unstable and context-dependent; solution is checking `immutable` property and parent commit IDs
- Pre-existing conflict cascades: conflicts baked into commits before rebase require independent resolution per commit; skipping this causes imerge2 to stall repeatedly on identical conflicts
- First-parent vs. total DAG ambiguity: 370 total commits might be only ~78 merge commits; misidentifying gap size leads to choosing wrong rebase strategy

**Decisions**:
- Mandatory `PRE_REBASE_OP` recording before rebase and cross-diff validation after: catches code loss during conflict resolution before transferring results back
- Incremental rebase (`jj-imerge2.sh`) as preferred method for gaps >80 merges: keeps conflicts small and traceable to specific upstream changes, vs. bulk rebase producing monolithic hard-to-understand conflicts
- Clone-based workflow: all operations in `/tmp/jj-rebase-clone` with `git.abandon-unreachable-commits = false` prevents accidental loss of original repo state and allows safe iteration

**Status**: Complete skill documentation and workflow reference ready for use; all 12 steps, edge cases, and script behaviors documented with examples and decision criteria.
