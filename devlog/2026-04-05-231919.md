## Optimizing Build Performance and Eliminating Slow Crypto Dependencies

**Context:** Compilation was intermittently slow (1 crate/few seconds vs. multiple per second), fans were spinning despite low observed CPU, and builds seemed to bottleneck on disk I/O rather than computation.

**What happened:**

- Killed an orphaned nushell process (PID 24700) that had been running for 2+ days accumulating 1669 minutes of CPU time with revoked file descriptors, spinning in a busy loop and starving the build of a full core
- Identified that `aws-lc-sys` (a slow C crypto library compiled via cmake) was being pulled in via `reqwest → hyper-rustls → rustls(aws_lc_rs)` while `ring` was already in the dependency tree via iroh
- Updated workspace `Cargo.toml` to force reqwest 0.13 to use `rustls-no-provider` and configured rustls at the workspace level to use the ring crypto backend instead of aws-lc-rs
- Updated `holon-mcp-client`'s reqwest 0.12 dependency to use `rustls-tls-ring` feature
- Removed `default-tls` feature requests across crates to stop pulling in native-tls (OpenSSL/SecureTransport)
- Discovered system was using 6.7 GB swap despite 93% RAM free — caused by pages swapped during past memory pressure (2-day nushell CPU hog + 15 concurrent rustc processes competing for memory) that were never reclaimed
- Ran `cargo clean` to remove 131 GB of stale artifacts across 863K files in `target/`

**Challenges:**

- **Reqwest feature fragmentation:** Reqwest 0.13 moved from `rustls-tls` to `rustls` (which defaults to aws-lc-rs) and removed the built-in ring feature. Solved by using `rustls-no-provider` at workspace level and configuring ring via rustls dependency features.
- **I/O bound builds:** Initial diagnosis looked like CPU contention, but `ps` and `iostat` revealed the actual bottleneck was disk IOPS (25K/sec) from 15 concurrent rustc processes with `-Z threads=8` each, fighting for memory and causing swap thrashing.

**Decisions:**

- Chose ring over aws-lc-rs not because it's inherently better, but because ring is already unavoidable in the tree (iroh needs it), so aligning all crypto to ring eliminates aws-lc-sys entirely rather than duplicating work
- Decided to run `cargo clean` despite the 2-hour runtime, since stale artifacts from old dependency trees were adding metadata overhead to incremental builds

**Status:** aws-lc-sys removed from dependency tree, swap usage will reset after next reboot, awaiting cargo clean completion before verifying the build performance improvement.
