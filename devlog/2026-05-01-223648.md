## Resolved JJ Rebase Conflicts in Holon Integration Tests

A feature branch with property-based testing was rebased onto a newer main that included significant refactors (method renames, let-chain syntax). Batch auto-resolve created divergent commits, requiring manual resolution of 3-sided and 2-sided conflicts.

**What happened:**

- Ran batch auto-resolve script; it created divergent commits instead of resolving cleanly
- Identified 4 conflicted commits: 3-sided conflicts in `state_machine.rs` and 2-sided in `sut.rs`
- Manually resolved all 3 conflicts in `state_machine.rs` by combining upstream API changes (`is_document() → is_page()`) with feature branch refactors (`.is_some_and()` chains and delegation to `transition.precondition()`)
- Resolved 1 conflict in `sut.rs` by preserving upstream's SQL query changes while adopting feature branch's let-chain syntax
- Cleaned up 4 stale divergent commits using `jj-cleanup-divergent.sh`
- Verified: `jj log -r 'conflicts()'` empty, `cargo check --workspace --exclude rust_lib_holon` passes, `cargo check -p holon-integration-tests --tests` passes

**Challenges:**

- Batch auto-resolve created divergent commits (3-sided conflicts can't be auto-merged); manually resolved instead
- 3-sided conflicts in `state_machine.rs` required understanding both upstream refactors AND feature branch intent; used `snapshot` conflict format for clarity
- File balance tracking showed empty descendants weren't data loss — their content was already in the rebased main merge commit

**Decisions:**

- Prioritized preserving upstream's API changes (`is_page()`) over feature branch's original method names
- For `sut.rs`, dropped the feature branch's `doc_block.name = ...` assignment since upstream SQL no longer fetches `name`

**Status:** All conflicts resolved and verified. Workspace builds cleanly. Separate unrelated issue: local main (`mxkyqnqv/0`) is divergent from origin main (`mxkyqnqv/2`).
