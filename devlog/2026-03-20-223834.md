## Share ReactiveEngine Between GPUI and MCP via Dependency Injection

The MCP `describe_ui` tool was returning empty data because `HeadlessBuilderServices` doesn't have access to the real render expressions and data. GPUI's `ReactiveEngine` has this data, but MCP couldn't access it. The fix: create the engine early in GPUI and share it with MCP via DI.

### What happened

- **MCP server** (`server.rs`): Added `builder_services: Option<Arc<dyn BuilderServices>>` field to `HolonMcpServer` and updated `new()` to accept it
- **Threaded through DI** (`di.rs`): Updated `run_http_server()` signature to accept `builder_services`, passed through `McpServerHandle`, with DI module passing `None`
- **Rewrote `describe_ui`** (`tools.rs:1205`): When `builder_services` is `Some`, calls `svc.get_block_data(&block_uri)` to get real render expressions and data rows instead of empty tables. Extracted `format_display_tree()` helper to avoid duplication
- **GPUI creates engine early** (`main.rs`): Creates `ReactiveEngine` and `OnceLock` services before MCP starts, passes `Some(services)` to `run_http_server` and engine to `launch_holon_window_with_engine`
- **GPUI reuses engine** (`lib.rs`): Made `make_interpret_fn` public, added `launch_holon_window_with_engine()` accepting pre-created `Arc<ReactiveEngine>`, refactored shared `launch_holon_window_impl()` 
- **Other frontends** (`blinc`, `ply`, `flutter`, `mcp` standalone): Pass `None` for `builder_services` — no behavior change

### Status

All target packages build clean (`holon-gpui`, `holon-mcp`, core libs). `describe_ui` now returns real block data from GPUI's ReactiveEngine when available, falling back to headless interpretation when not.
