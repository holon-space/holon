## Code Review: Refined Arrow-Key + Click Navigation PBT Plan

**Context**: A senior dev's critical review of the arrow-key navigation PBT plan identified nine significant gaps: underspecified navigators, magic timing, flaky preconditions, and noise from the "50% walk pre-step" feature. Validation revealed several were actionable and the plan was refined accordingly.

**What happened:**

- **Validated each criticism** against actual codebase: PBT content IS single-line (InputState handles boundary detection), ReferenceState already has render expressions, 100ms sleeps are unobservable, and non-focusable blocks can't be clicked.
- **Removed the 50% walk pre-step entirely** — it contaminated every edit transition with silent fallback logic that defeated test signal.
- **Replaced magic 100ms sleeps** with observable signals: exponential backoff polling of FocusRegistry after clicks, single read of actual focused element to catch layout-shift misses.
- **Tightened ClickBlock preconditions** to `block_is_focusable() && block_is_rendered()` (not just "exists in DB"). Added helper that builds ViewModel and finds only EditableText nodes, excluding source/query/render blocks.
- **Specified Left/Right semantics per navigator type**: List returns None, Tree does parent(L)/child(R), Table does column movement. Direction generation now restricted by layout.
- **Clarified boundary detection pipeline**: EditorController always propagates → InputState handles internally → emits MoveUp/MoveDown at boundary → parent on_action calls handle_cross_block_nav.
- **Added investigation tasks**: Verify InputState boundary behavior under nested editors, and API design for focus read-back after enigo clicks.

**Status**: Plan restructured to eliminate noise, tighten preconditions, and be honest about what each mode tests (headless: f(x)==f(x); GPUI: real InputState pipeline). Ready for implementation with two open investigations.
