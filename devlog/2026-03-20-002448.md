## Cross-Block Cursor Navigation Infrastructure

We're wiring up cursor movement between blocks — specifically moving from the first line of a block to the last line of the block above when pressing up at a boundary.

**What happened**

- Explored existing GPUI cursor/focus infrastructure in both `@frontends/gpui/` and `@crates/holon-frontend/`, including `InputState`, `FocusHandle`, and action propagation
- Investigated GPUI's `Entity::update()` and `Window` API constraints for cross-entity cursor manipulation
- Deep-dived into focus handling: `subscribe_in` callbacks provide `(&mut Window, &mut Context<T>)`, but `Entity::update()` only provides `(&mut App)` without `Window` access
- Researched Zed's approach: editors call `cx.propagate()` when navigation hits a boundary, letting parent views handle focus transfer
- Confirmed we can capture `&mut Window` from outer scope in `entity.update()` callbacks due to separate borrow domains

**Challenges**

- **Cross-entity cursor setting**: `InputState::set_cursor_position` requires `(&mut Window, &mut Context<InputState>)`, but `Entity::update()` only provides `&mut App`. Solved by using a two-phase approach: (1) `entity.update(cx, ...)` for cursor position calculation, (2) `window.focus(&handle, cx)` separately for focus
- **Borrow checker complexity**: `entity.update(cx, ...)` mutably borrows `cx`, while `&mut Window` is separate. Confirmed this works because they're different reference types with independent lifetimes

**Decisions**

- Adopt Zed's simpler propagation model: `EditorView` calls `cx.propagate()` on boundary, letting parent handle focus logic rather than attempting direct cursor manipulation
- Store pending cursor placement in `BoundsRegistry` (shared state), with target editor applying it on `FocusIn` event where both `Window` and proper `Context<InputState>` are available

**Status**

Architecture decisions finalized; ready to implement cursor position bubbling in EditorView's `MoveUp` handler and focus event subscription.
