## Fixed self-healing `_routing_doc_uri` appearing in org file properties

The `_routing_doc_uri` routing metadata was being written to the org file's `:PROPERTIES:` drawers and source block header args, and persisting even after manual deletion.

### What happened

- Searched historian for context on `_routing_doc_uri` — found it's routing metadata added at the event payload top level, not in `data`
- Traced the data flow: org file → parser → DB → event → cache → OrgRenderer → org file
- Discovered `_routing_doc_uri` appearing in both `:PROPERTIES:` drawers and `#+BEGIN_SRC` header args in `index.org`
- Found the root cause: `drawer_properties()` in `models.rs:658` was returning `_routing_doc_uri` because it wasn't in the `INTERNAL_KEYS` allowlist
- Identified the self-reinforcing cycle: parser reads it from the file → `drawer_properties()` returns it → renderer writes it back

### Root cause

The codebase has a convention that `_`-prefixed keys are routing metadata (used in `partition_params()` line 213 and `build_event_payload()` line 571). However, `drawer_properties()` used an allowlist (`INTERNAL_KEYS`) instead, missing `_routing_doc_uri`. Once the key appeared in the file, the cycle kept it alive.

### Fix

Modified `drawer_properties()` in `crates/holon-orgmode/src/models.rs` to exclude ALL `_`-prefixed keys (matching the convention elsewhere), not just those in `INTERNAL_KEYS`. Added `!k.starts_with('_')` filters in three extraction paths within the method.

**Status**: Fixed. On app restart, existing `_routing_doc_uri` entries will be cleaned up on next render cycle as `drawer_properties()` no longer returns them.
