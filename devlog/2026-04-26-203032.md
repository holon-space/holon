## Correcting ReadOnlyMutable Access Control Architecture

The codebase was attempting to pass mutable `DataRow` references downstream, which violated the core architectural constraint: only `ReactiveLiveQuery` should hold `Mutable` access, with all downstream consumers restricted to `ReadOnlyMutable` to prevent unsynced writes to Turso.

**What happened:**

- Reviewed the handoff task and identified that initial implementation was passing `Mutable<DataRow>` downstream instead of `ReadOnlyMutable<DataRow>`
- Caught mid-implementation when user highlighted the architectural violation: writes outside `ReactiveLiveQuery` would silently fail to sync with Turso
- Pivoted the approach: `RenderContext` now holds `ReadOnlyMutable` via `current_row_mutable` field
- Added `data_mutable()` accessor and `with_row_mutable()` constructor to `RenderContext` with correct type signatures
- Updated downstream consumer sites to work with `ReadOnlyMutable` instead of `Mutable`

**Challenges:**

- **Type system alignment**: The constructor needed `ReadOnlyMutable<DataRow>` type parameter, not `Mutable<DataRow>`. Compiler errors in `with_row_mutable()` were due to passing the wrong type variant.
- **Architectural validation**: The TypeScript-style type system prevented the bug from silently shipping—attempting mutable writes on `ReadOnlyMutable` becomes a compile error rather than a runtime sync failure.

**Decisions:**

- `ReactiveRowSet` remains the sole holder of `Mutable<DataRow>` and is responsible for all Turso writes
- All other code paths receive `ReadOnlyMutable` to make invalid mutation attempts compile errors

**Status:** Architecture corrected mid-implementation; `RenderContext` plumbing updated with `ReadOnlyMutable` type signatures, remaining type errors being resolved.
