## Extending PBT with ViewModel/Shadow DOM assertions

The PBT currently tests down to the SQL/data layer (invariant 11 calls `render_entity()` but discards output). We're adding ViewModel structural validation to close the gap between "data is correct" and "the UI tree frontends build is correct."

**What happened**

- Analyzed the test infrastructure: `E2ESut` persists across transitions, `check_invariants()` is called after each mutation
- Explored the real frontend path: `watch_ui()` → `WidgetSpec` → `create_shadow_interpreter().interpret()` → `ViewModel` (same as GPUI/Blinc)
- Identified that `render_entity()` (one-shot) differs from `watch_ui()` (streaming with CDC); using `watch_ui()` tests production code paths
- Designed persistent `WatchHandle` stored on `E2ESut` via `RefCell<Option<_>>` to survive across mutations
- Planned lazy initialization: watch created on first invariant check where `has_valid_index_org()` is true

**Challenges**

- `WatchHandle` is `Send` but not `Clone`, with `mpsc::Receiver` that can't be cloned — solved by storing `RefCell<Option<WatchHandle>>` on SUT and mutating in-place
- `check_invariants(&self)` takes immutable reference, but `WatchHandle::drain()` needs `&mut` — solved with interior mutability (`RefCell`)
- `try_recv()` doesn't exist on `WatchHandle` yet — adding it to `streaming.rs` to drain pending events without blocking after first event

**Decisions**

- Use `watch_ui_first_structure()` instead of `render_entity()` — tests the exact code path frontends use
- Drain pattern: block-wait only on first call, then `try_recv()` all pending events — avoids test hangs while catching all structure updates
- Keep assertions framework-agnostic: check entity ID presence, widget types (no error roots), ordering — no pixel-level or Flutter-specific checks

**Status**: Implementation plan complete; ready to code `streaming.rs` (add `try_recv`), `sut.rs` (persistent watch + shadow interpret), and `display_assertions.rs` (helper function).
