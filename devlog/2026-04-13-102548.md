## Fixed state_toggle operations missing in list item_template

A PBT regression surfaced when mutations rewrote render blocks to use `state_toggle(col("task_state"))` inside list item templates: the widget showed empty state with no operations attached, even for valid rows.

**What happened**

- Identified root cause: three `interpret_row` call sites in `reactive_view.rs` built row contexts with `RenderContext::default().with_row(row)` but never resolved the row's profile to attach operations
- Production code bypasses this via `block_ref()` → `render_entity` which resolves operations later; inline PBT mutations bypass that path
- Added `row_render_context()` helper to resolve profile ops per row and updated three sites: `create_tree_driver`, `create_flat_driver`, `create_sorted_flat_driver`
- Updated `Collection` macro Static arm in `widget_builder.rs:256` to resolve profile ops per row during `interpret_pure` (used by PBT snapshots)
- Test progressed from `[inv10h] StateToggle … has no operations!` → `[inv10h] StateToggle current '' != reference 'NEXT'`

**Challenges**

- Root cause turned out to be context threading, not widget rendering — the state_toggle builder was correct, but had no operations to work with
- Discovered the actual remaining failure is a **generator contract mismatch**, not a renderer bug: the GQL/PRQL queries return only `id, content` but the mutation generates `state_toggle(col("task_state"))`, which doesn't exist in the row data

**Decisions**

- Left generator queries unchanged (updating them exceeds scope) and filed task #2 with two clean options: (1) update queries to return `properties`/`task_state`, or (2) filter mutations to only expressions whose visible columns match current schema
- Avoided filtering peer-modified blocks from mutation targets to prevent concurrent-edit scenarios, which the reference model can't simulate with overwrite semantics

**Status**

Operations now correctly resolve for inline widget builders in all contexts. Full variant fails at a later invariant due to missing query columns — a known generator limitation, not a renderer defect.
