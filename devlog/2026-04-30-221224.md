## Refactoring LiveData to Use MutableBTreeMap

**Context**: The `LiveData` struct was using a manual `items` field alongside a `version_tx` channel to track data changes. The goal was to simplify this by leveraging `MutableBTreeMap` from the `futures-signals` crate, which provides change notification semantics out of the box.

**What happened**:

- Analyzed the `LiveData` struct and identified all callers to understand the refactoring scope
- Reviewed the `futures-signals` `MutableBTreeMap` API and trait bounds requirements (`Eq + Ord + Clone` for keys, `Clone` for values)
- Studied the initialization semantics to correctly model startup rebuild behavior with in-place state
- Replaced the `items: BTreeMap<K, V>` field with `items: MutableBTreeMap<K, V>`
- Removed the `version_tx` channel since `MutableBTreeMap` emits signals automatically on mutations
- Updated all mutation sites (insertions, removals, clears) to use the `MutableBTreeMap` API
- Adjusted signal subscriptions and change detection logic to consume the new signal stream

**Challenges**:

- **Initial state handling**: The startup rebuild needs to populate the `MutableBTreeMap` in-place before subscribers attach, requiring careful ordering to avoid race conditions between initialization and signal emission
- **API differences**: `MutableBTreeMap` has a different mutation interface than standard `BTreeMap`; required mapping operations like `insert()` and `remove()` to the reactive equivalent

**Decisions**:

- Kept the public interface of `LiveData` unchanged to minimize cascading changes
- Chose `MutableBTreeMap` over manual versioning for cleaner, more idiomatic reactive state management

**Status**: Refactoring plan complete with clear migration path identified; ready for implementation.
