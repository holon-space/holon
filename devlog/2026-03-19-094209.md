## Fixed GQL Schema Registration & Standardized Node Label Casing

The GQL compiler was using EAV fallback tables instead of direct entity tables, and GQL node labels didn't match SQL/PRQL table names, creating a confusing inconsistency for users.

**What happened:**

- Diagnosed that `BackendEngine` factory in `registration.rs` called `try_resolve_all()` (sync) to resolve `EntitySchemaProvider` instances, but the MCP integration provider was registered with `Provider::root_async`. The sync resolve couldn't handle async providers and likely returned empty or incomplete results.
- Changed both `BackendEngine` factory sites (lines 224, 442) to use `try_resolve_all_async()` — confirmed 3 providers now resolve (Core Block/Document, Filesystem File/Directory, MCP integrations).
- Discovered GQL label matching is case-sensitive: `(n:block)` fell through to EAV, while `(n:Block)` worked. This created confusion — users must write `block` in SQL/PRQL but `Block` in GQL.
- Standardized all `graph_label` values to lowercase table names (`block`, `document`, `file`, `directory`, `todoist_task`, `todoist_project`, `current_focus`, `focus_root`) for consistency across all query languages.
- Updated all GQL queries in org files, tests, and schema tests to use lowercase labels.

**Status:** Both issues fixed and tested. App correctly resolves all 3 entity schema providers and GQL queries now use consistent lowercase node labels matching SQL/PRQL table names.
