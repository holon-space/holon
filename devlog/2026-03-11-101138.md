## Eliminating Redundant Document Path Resolution Queries with LiveData

During startup, Holon was executing 1,340 identical SQL queries (`SELECT * FROM document WHERE parent_id = $parent_id AND name = $name`) in a tight loop, hitting Turso 100+ times for the same 5 documents.

### What happened

- Analyzed logs to identify the query pattern and call chain: `find_by_name_chain` → `find_by_parent_and_name` (SQL) for each path segment
- Traced the root cause: CDC block events fired one-at-a-time during startup, each triggering document path resolution via SQL
- Discovered the documents table is effectively immutable after initial load, making repeated SQL lookups unnecessary
- Evaluated caching strategies: in-memory HashMap, `QueryableCache<Document>`, and `LiveData<Document>` backed by CDC
- Implemented `LiveDocumentManager` using `LiveData<Document>` with Turso CDC stream:
  - Created `MatviewManager::watch("SELECT * FROM document")` to establish CDC subscription
  - Loads all documents into memory at startup; CDC stream keeps HashMap updated automatically
  - Replaced `DocumentManagerAdapter` in orgmode DI wiring with the new `LiveDocumentManager`
  - Made `DocumentOperations::row_to_document` public for reuse

### Decisions

- **LiveData over simple HashMap**: Ensures in-memory cache stays synchronized with database mutations via CDC, not just during startup
- **CDC-driven updates over write-through**: `create()` delegates to `DocumentOperations`; CDC automatically updates the cache, maintaining consistency across distributed components
- **Scan-based `find_by_parent_and_name`**: With only ~5-10 documents, an O(n) in-memory scan is faster than SQL roundtrips and avoids secondary index maintenance

**Status**: Implemented and compiling. Document path resolution now hits in-memory cache instead of Turso; eliminates all 1,340 redundant queries during startup.
