## Diagnosed and fixed stale materialized view crash

The holon app was failing to start with a cryptic "circular dependency" error in the database layer. Investigation revealed the actual problem: a persisted materialized view referenced a column that no longer exists in the schema.

**What happened**

- User reported app crash at startup with panic in `holon/src/di/lifecycle.rs:62`
- Examined `/tmp/holon.log` and found error: `Cannot resolve materialized view dependencies (possible circular dependency): watch_view_2206789c2d25be5a`
- Used `tursodb` CLI to query the DB and extracted the problematic view SQL
- Found the view queries `block WHERE is_document = 1` but the `block` table has no `is_document` column
- Traced through Turso's `populate_materialized_views` logic in `core/schema.rs:1370-1397` and identified the root cause

**Root cause**

The DB schema was updated to remove the `is_document` column from the `block` table (documents are now identified by non-empty `name` field). However, the persisted materialized view still referenced this dropped column. Turso's view resolution loop defers views with missing dependencies ("not found in schema") expecting they'll be available in future iterations. Since `is_document` never reappears, the view defers forever, and the loop eventually reports this as a "circular dependency" rather than a schema mismatch.

**Resolution**

- Updated `/Users/martin/Workspaces/pkm/holon-pkm/index.org` line 24: removed redundant `filter is_document == 1` since name filters already identify documents
- Deleted stale DB at `~/.config/holon/holon.db` to allow clean recreation
- Created detailed hand-off prompt for Turso maintainers documenting the bug and suggesting a fix: distinguish between "dependency not ready" (defer) vs. "permanently invalid view with dropped columns" (drop)

**Status**: App can now restart cleanly with recreated DB. Turso needs enhancement to handle stale views more gracefully during `populate_materialized_views`.
