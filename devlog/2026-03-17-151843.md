## Jujutsu Branch Rebase & Conflict Resolution Skill

**Context:**
Documented a comprehensive skill for safely rebasing multiple user-authored branches onto upstream main in an isolated clone, with automatic conflict resolution and validation.

**What happened:**

- Created `jj-rebase-clone.sh` to establish isolated temporary clones with auto-rebase disabled, allowing safe fetch/rebase without triggering unwanted cascading operations
- Built `jj-cleanup-divergent.sh` to identify and abandon stale divergent commit copies before rebasing, preventing conflicts from cascading through both chains
- Implemented `jj-imerge2.sh` as the preferred incremental rebase method—walks the first-parent chain one merge commit at a time with automatic mergiraf resolution, producing small traceable conflicts
- Wrote `jj-auto-resolve.sh` to run mergiraf on conflicted commits root-first with looping until no progress, cascading resolutions to descendants
- Created `jj-rebase-validate.sh` for mandatory cross-diff validation using `--at-op` to detect lost code after conflict resolution
- Developed `jj-rebase-finish.sh` to transfer results back and clean up temporary bookmarks and clones
- Documented critical revset patterns (e.g., `roots(author("User") ~ ::main)`) and divergent commit identification via `immutable` property and `parents()`, never `/N` offsets

**Challenges:**

- **Divergent commit instability**: The `/0` vs `/1` assignment depends on internal ordering, not which copy is "original." Solution: Always check `immutable` property and parent to identify the mutable copy parented on latest main.
- **Pre-existing conflict cascades**: Conflicts baked into commits before rebase don't auto-resolve when parents are fixed. Solution: Resolve pre-rebase conflicts in step 3b using `jj restore` for files the descendant doesn't modify, manual edit for files it does.
- **Lost operation context**: Cross-diff validation requires pre-rebase operation ID. Solution: Record `PRE_REBASE_OP` before rebasing (step 6c), allowing `--at-op` lookups even if the rebase command is interrupted.

**Decisions:**

- **`-s` (source) only, never `-b` (branch)**: Source-based rebase is safer and more predictable for preserving merge topology with mega-merges.
- **Incremental vs. bulk strategy**: Use first-parent merge commit count (not total DAG commits) to decide. >80 merges → incremental; <20 → bulk. Incremental produces smaller, traceable conflicts tied to specific upstream commits.
- **Task checklist upfront**: Create task list with all 11 steps before starting to prevent skipping critical steps like step 6c (record op) and step 9 (validation).

**Status:**
Skill fully documented with 12-step workflow, 6 bundled scripts, comprehensive edge case handling, and critical revset patterns for safe multi-branch rebase in isolation.
