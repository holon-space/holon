## Minimized Turso Recursive CTE IVM Bug to 89 Statements

The recursive CTE materialized view (`block_with_path`) was accumulating 3 phantom rows during incremental maintenance via INSERT OR REPLACE, while fresh evaluation was correct. Needed to identify the minimal set of SQL statements that reliably reproduces the bug for an upstream report.

**What happened:**

- Ran the ddmin minimizer on the full 1135-directive SQL trace, which iteratively removed statements and table definitions while checking if the "IVM BUG REPRODUCED" pattern still appeared
- Minimizer completed in 6 phases, reducing from 1135 to **89 directives** (92% reduction)
- Verified the minimal reproducer: 1 CREATE TABLE, 1 CREATE MATERIALIZED VIEW (recursive CTE), 43 initial INSERTs, 44 INSERT OR REPLACEs
- Confirmed bug still manifests: matview has 87 rows, fresh evaluation gives 84 (3 phantom rows)
- Traced the pattern: three children of block `3fd58f88` ("Cross-Device Sync") become phantom rows, triggered when an unrelated sibling subtree is re-inserted via INSERT OR REPLACE at statement 89

**Challenges:**

- Attempted synthetic reproducer with fewer blocks didn't trigger the bug—the issue is data-dependent on the specific tree structure and re-insertion order
- Phantom rows only appear at the final statement, but ddmin confirmed all 87 DML statements are individually necessary; removing any one makes the bug disappear
- This suggests the bug requires the exact sequence of ancestor-before-child re-insertions to manifest

**Status:** Minimal reproducer isolated and saved to `devlog/turso-ivm-rcte-bug-minimal.sql`. Ready for upstream bug report to Turso fork.
