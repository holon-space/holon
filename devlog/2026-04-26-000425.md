## Devlog: Code Review & Plan Refinement for Rich-Text Inline Marks

**Context:**  
A phased plan to add inline rich-text marks (bold, italic, etc.) to Holon blocks existed but was built on unvalidated assumptions. The goal was to read it as a senior dev doing a hostile code review, verify its claims against the actual codebase, expose gaps, and rebuild it correctly.

**What happened:**

- **Read the original plan** and identified 23 implicit hypotheses (Loro version pinning, storage architecture, orgize stability, frontend coverage, etc.)
- **Validated against codebase**: Loro is `^1.0` not pinned; `BlockContent` is constructor-only, not storage (marks have nowhere to live in SQL); `config_text_style` is document-scoped not per-text; two parallel orgmode crates exist (not one); seven frontends exist (not three)
- **Ran fast spot-checks**: confirmed orgize is `0.10.0-alpha.10`, `Block.content` is `String` in SQL, Loro's mark API is document-level, FRB boundary crosses `BlockContent` already
- **Documented root causes**: The plan's core error was treating `BlockContent::RichText { text, marks }` as storage when it's only a constructor arm; actual storage is `Block.content: String` flattened to `blocks.content TEXT`, leaving marks with no durable home
- **Built a 23-row hypothesis table** mapping each assumption to verification status and evidence
- **Rewrote the plan** from first principles, adding: a sibling `marks` column/field for denormalized Loro projection; explicit `LoroDoc` init hook; decision gate on dual-maintain vs merge for the two orgmode crates; Phase 0 validation spikes for Loro conflict semantics, editor replacement viability, and orgize audit; clarified mark-only CDC propagation as open; decomposed Phase 4 parser statelessness myth into a `CaretCtx` trait design; gated Phase 3 on Phase 0.2 editor spike with fallback-wrapper strategy

**Challenges:**

- **Architecture invisibility**: `BlockContent` is not storage; the plan was written by someone who'd never traced a block write through PRQL/SQL. Required reading `block.rs:275`, `loro_backend.rs:213`, the CDC path, and FRB bridge to surface the gap.
- **Forked codebase blindness**: Two parallel orgmode crates with the same parser/models/renderer were mentioned once in passing. Original plan touched only one crate inconsistently — required a merge-vs-dual-maintain decision gate before Phase 1.1.
- **API-level assumptions**: Loro's `config_text_style` is `LoroDoc` not `LoroText`, with a "key always same expand type" contract. Plan called it at `LoroText` init inside a hot path (`meta.get_or_create_container` on every block update). Spike required reading `loro-1.10.8/src/lib.rs:314` and the contract docs.

**Decisions:**

- **Add marks as denormalized column**: `marks: Option<Vec<MarkSpan>>` lives as sibling field to `content: String` — keeps text matchers working, matches Loro's projection model
- **Three Phase 0 validation spikes** before committing to architecture: Loro conflict semantics
