## Fixed false-positive duplicate block detection in org sync

The sync controller was incorrectly flagging nested blocks within the same document as "foreign" (belonging to different files), causing false duplicate warnings on startup.

**What happened:**

- Investigated why `ClaudeCode.org` reported 15 duplicate block ID conflicts despite no actual duplicates existing on disk
- Traced the issue to `find_foreign_blocks()` in `di.rs:234`, which checked only the **direct parent** of each block (`b.parent_id != expected_doc_uri`)
- Nested blocks like `cc-tasks` (parent: `block:cc-history-root`) were falsely flagged as foreign because their parent wasn't the document entity itself
- The real ClaudeCode.org structure has 16 blocks total: 1 root + 5 headings + 10 render blocks, all correctly nested — but 15 were misidentified as cross-file conflicts

**Root cause:**

The comparison was flat and didn't walk the ancestor chain. Only the direct child of a document matched; deeper descendants always failed the check.

**Solution:**

- Added a correct **default implementation** of `find_foreign_blocks()` on the `BlockReader` trait in `traits.rs` that reuses `iter_documents_with_blocks()` — which already performs proper BFS traversal to group all descendants under their document root
- Removed broken overrides from both `CacheBlockReader` (real implementation) and the test mock
- Both now correctly walk the full ancestor chain

**Tests added** (4 cases in `sync_controller_mutation_pbt.rs`):
- Nested blocks under same document not flagged as foreign
- Actual cross-document duplicates detected correctly  
- Deeply nested (3-level) foreign blocks detected
- Edge case: empty query returns empty

**Status:** Fixed and tested — nested blocks are now correctly identified as belonging to their document, eliminating the spurious 15-conflict warnings.
