## Critique & Redesign: Query-Driven Actions for Journal Auto-Creation

**Context**: Adding a declarative action system where queries trigger CDC-based mutations — first use case is journaling, but architected for broader patterns like email, webhooks, shared-tree constraints.

**What happened**:
- Explored existing CDC, render block, and operation bus infrastructure
- Designed `action_watcher.rs`: streaming discovery of query+action block pairs, Rhai DSL parser, column resolution via `resolve_args_with()`, routing through `execute_operation`
- Implemented journal auto-creation as `#+BEGIN_SRC holon_sql` trigger + `#+BEGIN_SRC action` blocks in `Journals.org`
- Added `start_action_watchers()` to `frontend_module.rs` before OrgMode readiness, with wasm guard
- Fixed `OrgSyncController` to `create_dir_all` before file writes
- Extended `render_dsl.rs` exports for Rhai DSL reuse

**Challenges**:
- **Async settling**: Action watchers execute after CDC fires; PBT needs drain/settle logic (reused from existing render watch pattern)
- **Ordering independence**: Query+action blocks can appear in any org file; solved via streaming discovery loop that polls for new pairs every 5s
- **Empty document edge case**: rendering empty doc produces `null`; handled gracefully by action's `block.create()` call

**Decisions**:
- Route actions through existing `execute_operation` bus rather than custom handler — all operations (UI, MCP, actions) use same validation/audit pipeline
- Use Rhai for action DSL — lightweight, sandboxed, column args resolved before eval
- **Defer: Loro sync & security model** — architecture supports it (actions identified by stable block IDs) but needs threat model: arbitrary actions + shared trees = malicious triggers; email actions + device sync = unwanted duplicates. Requires per-action sync policy + deduplication framework (post-MVP)
