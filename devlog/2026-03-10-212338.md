## Resilient MCP Proxy for holon-live Backend Restarts

**Context**: The Claude Code MCP client required manual reconnection every time the holon-live app restarted. We needed a proxy that survives backend restarts and auto-reconnects without losing the session.

**What happened**:
- Evaluated `mcp-proxy` and `mcp-rust-proxy` — both designed to spawn/manage servers, not proxy to already-running endpoints
- Forked `mcp-rust-proxy` and added support for external server proxying (no process spawning required)
- Added `streamableHttp` transport type to connect to holon-live's HTTP-POST MCP protocol on `http://127.0.0.1:8520/mcp`
- Configured the proxy to run as a stdio MCP server that Claude Code spawns, with `restartOnFailure` and retry logic
- Updated `~/.claude.json` to point Claude Code at the proxy instead of direct holon-live HTTP
- Verified tools are proxied through with proper namespacing (`mcp__holon-live__holon_live__{tool}`)

**Challenges**:
- **Tool name prefixing**: The proxy adds server name prefix, but Claude Code also prefixes — resulted in doubled names like `mcp__holon-live__holon_live__`. Solved by granting fresh permissions on new tool names.
- **Stale session after backend restart**: The proxy stays alive and stdio connection persists, but the streamable HTTP session becomes invalid when the backend restarts. Tool calls return HTTP 404 because the session ID no longer exists on the new backend instance.

**Status**: Proxy is installed and wired up, tools are callable, but **reconnect-on-error logic is needed** — when a tool call fails with 404/connection error, the proxy should re-initialize the backend connection and retry the request.
