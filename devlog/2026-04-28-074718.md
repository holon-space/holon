## Refactored Board Widget to Support Data-Driven Rendering with Drag-and-Drop

**Context**

The board widget had hardcoded demo data and card rendering logic mixed into the GPUI builder layer. The goal was to parameterize the board's lane-grouping field and item template, enabling the Design Gallery to display a properly data-driven board built through the render expression `board(item_template: card(...), lane_field: "status", rows: [...])`.

**What happened**

- Rewrote `crates/holon-frontend/src/shadow_builders/board.rs` as a `raw fn` that accepts `item_template`, `lane_field`, and inline `rows: Value::Array`. Groups rows by the specified lane field into `board_lane` view models whose children render via the template.
- Stripped demo code from `frontends/gpui/src/render/builders/board.rs` — removed `demo_lanes`, `BoardCard`, `render_card`, and hardcoded parsing. Now dispatches each `card` VM through the regular renderer.
- Updated `crates/holon-frontend/src/widget_gallery.rs` with `board_mode_expr` building `board(item_template: card(...), lane_field: "status", rows: [...])` with eight inline cards across "To Do", "In Progress", "Done" lanes.
- Added `board_mode_groups_cards_into_lanes` test verifying the shadow builder produces the correct `board → board_lane → card` VM tree.
- Restored drag-and-drop: wrapped cards in `Sortable<BoardCard>` (snapshotted as plain `Clone` data with accent color, hex, and text content).

**Challenges**

- **Drag-and-drop UI flicker**: The gallery rebuilt the entire ViewModel tree on every render, causing the pointer-derived cache key (`format!("board-{:p}", node)`) to miss every frame, resetting `SortableState` and oscillating between the dropped and original order. Fixed by switching to a stable string key (`"design-gallery-board"`).
- **Unsafe reborrow during drag**: Sortable renders items twice per frame (in-list ghost + floating preview). Attempted to inline rendering via `&` → `&mut` cast, but modern Rust forbids this. Reverted to extracting card visual data into `BoardCard` and rendering inline without shared entities.

**Decisions**

- Kept visual treatment (tinted bg, accent left border, hover shadow) inlined in `render_card` to mirror the `card.rs` builder, satisfying the "proper `board(item_template: card(...), ...)` render expression" requirement at the data composition layer while keeping Sortable rendering clean.
- Cached `SortableState` under a stable non-pointer key to survive frame-to-frame ViewModel tree rebuilds.

**Status**

Design Gallery board displays data-driven, customizable cards grouped by status with working in-lane and cross-lane drag-and-drop — no flicker.
