## Removed forwarder methods from `TursoBackend`

After refactoring consumers to use `DbHandle` directly, the forwarder methods on `TursoBackend` were no longer needed and created unnecessary indirection.

**What happened**

- Updated 3 additional consumer files to use `DbHandle` instead of `Arc<RwLock<TursoBackend>>`:
  - `document_operations.rs` — `DocumentOperations` now takes `DbHandle`
  - `credential_store.rs` — `TursoCredentialStore` now takes `DbHandle`
  - `backend_engine.rs` test fixtures — updated to use `DbHandle` directly
- Converted `SchemaModule` trait from taking `&TursoBackend` to `&DbHandle`:
  - Updated trait definition in `schema_module.rs`
  - Updated all implementations in `schema_modules.rs` (`SqliteSchemaModule`, `CoreSchemaModule`, `HolonSchemaModule`)
  - Updated all callers of `ensure_schema()`
- Updated DI registration to pass `DbHandle` to `SchemaRegistry` instead of `Arc<RwLock<TursoBackend>>`
- Deleted forwarder methods from `TursoBackend`:
  - `execute_ddl()`
  - `execute_sql()` / `execute_sql_fresh()`
  - `execute_via_actor()`
  - `execute_batch_transaction()`
  - `execute_via_actor()`
  - `handle()` accessor

**Challenges**

- The `SchemaModule` trait was widely used — all implementations and callers needed updating in parallel
- `SchemaRegistry` held `Arc<RwLock<TursoBackend>>` primarily to pass it to `SchemaModule::ensure_schema()` — removing that reduced the lock's only purpose

**Decisions**

- Kept `TursoBackend::row_changes()` and `TursoBackend::get_connection()` as they provide unique functionality not on `DbHandle`
- Did not remove `StorageBackend` trait yet — still used by test infrastructure

**Status**

All forwarder methods removed from `TursoBackend`. Clean compilation with no consumers remaining that rely on them. All 7 `operation_log` tests pass.
