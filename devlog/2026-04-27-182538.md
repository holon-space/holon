## Landed MarkdownFormatAdapter for Obsidian Vault Support

Implemented Phase 1's second FileFormatAdapter to validate the seam and ship the F1 wedge—an Obsidian-compatible markdown parser paired with `OrgFormatAdapter` under the same trait surface.

**What happened:**

- Created new `holon-markdown` crate with parser (via `markdown` 1.0 crate), renderer, and YAML frontmatter handling
- Designed wikilink extraction: `[[wikilink]]` text preserved in `block.content` for round-trip fidelity; bare names extracted into a `wikilinks` JSON-array property for graph queries
- Mapped markdown structure to Block hierarchy: headings → Blocks, paragraphs/lists between headings → parent content, code fences → child Source blocks (mirroring org `#+BEGIN_SRC`)
- Block IDs via Obsidian `^id` markers; frontmatter parsed into block properties
- Implemented `FileFormatAdapter` trait, allowing both adapters to operate through the same `Arc<dyn FileFormatAdapter>` surface without trait modification
- All 37 unit tests + 2 integration tests pass (round-trip full Obsidian file through trait); workspace compiles clean

**Challenges:**

- **Wikilink resolution scope**: initially considered resolving `[[Other Note]]` → `EntityUri::file(...)` inside the adapter, but single-file boundary lacks vault-wide filename index. Pushed to higher layer (sync controller or dedicated resolver).
- **Frontmatter parsing**: stray `?` in `split_frontmatter` was returning `Option` into non-Option function—fixed by removing unused hash helper and simplifying logic.

**Decisions:**

- **Single crate, not split**: `holon-markdown` avoids the complexity of separate parser/sync crates until justified by runtime needs.
- **Deferred decisions now resolved**: image handling and post-write hooks remain in controller—format-agnostic and already produce consistent intermediate structures.

**Status:** MarkdownFormatAdapter complete, seam validated with dual-impl integration test, spec updated (0006 step 1.3, Sync.md); ready for vault-mode runtime wiring in next phase.
