## Fixed Bug C: Ref-state trailing-whitespace divergence

Ref-model `Mutation::Create`/`Update::apply_to` stored content with
trailing whitespace on the first line intact, but the org file
round-trip (serialize → write → parse) always strips it because the
title line becomes the org headline and both the renderer and parser
normalize it. Result: backend SQL ends up with `"nL q\npAWaeG..."`
while ref_state keeps `"nL q \npAWaeG..."`, and the UI spot-check at
`sut.rs:4241` flags the divergence on the next mutation that touches
the same block.

**What happened:**
- Identified that `content.trim_end()` in `pbt/types.rs` only trims
  the END of the whole string, not the first line's trailing
  whitespace — which is the exact normalization the org renderer and
  parser perform on the title line
- Added `normalize_content_for_org_roundtrip(content, content_type)`
  that splits on the first newline, trims the title, preserves body
  verbatim; source blocks pass through unchanged
- Applied normalization in both Create and Update paths
- `sql_only` variant now passes 50/50 cases

**Challenges:**
- Took a while to spot because the spot-check failure surfaced on a UI
  `task_state` mutation, but the actual divergence came from an
  earlier External content update. Had to trace through the
  serializer to realize `serialize_block_recursive` does `trim_end`
  only on the title line, not the whole content.

**Status:** Bug C fixed and verified for `sql_only`. `general_e2e_pbt`
