## Fixed GPUI Deadlock and `block_ref()` Argument Fallback

The GPUI frontend was frozen showing only a gray bar due to a deadlock in the render pipeline, compounded by a panic in block reference resolution.

**What happened:**

- Investigated `/tmp/holon-gpui.log` and found two distinct issues blocking the UI
- **Issue 1**: `block_ref()` used as `item_template` in the root layout panicked because it expected a positional `block_id` argument but received none
  - Root cause: when `block_ref()` is used in an `item_template`, it's called per-row with the row bound to context, but had no fallback to extract `id` from that context
  - Fixed in `render_interpreter.rs`: added fallback to `ctx.row().get("id").and_then(|v| v.as_string())` when no positional arg provided
  
- **Issue 2**: Even after fix 1, the GPUI remained frozen (main thread deadlocked)
  - Root cause: `shared_block_ref_build` called `render_block()` directly with synchronous `handle.block_on()` during the GPUI render pass, blocking the main thread
  - If the database actor was busy (e.g., org sync queries), the render would hang indefinitely
  - Fixed in `render_context.rs`: modified `BlockRenderCache::get_or_watch()` to be non-blocking on first access
    - Returns a placeholder `spacer()` immediately 
    - Spawns the `watch_ui` fetch as a background tokio task
    - Content populates on next render frame (~2 seconds later)

**Decisions:**

- Used `BlockRenderCache` instead of direct `render_block` calls — it's the designed pattern for cached renders
- Made cache non-blocking by deferring first fetch to background task rather than blocking on initial render

**Status:** Both fixes compiled cleanly. Ready to restart the GPUI binary and test non-blocking render with fallback block_ref argument resolution.
