## Fixed Loro content overwrite race condition in cross-executor variant

The PBT test's `cross_executor` variant was failing because Loro sync operations were overwriting SQL mutations with stale data. A content update would succeed in SQL, but then a concurrent Loro bulk-add would trigger a full-row UPDATE that clobbered the fresher SQL value.

**What happened:**

- Ran `general_e2e_pbt.rs` and found `cross_executor` variant failing with block content divergence
- Traced the root cause: `ApplyMutation::Update` wrote `"GMkjxI1C"` to SQL, but then a `BulkExternalAdd` Loro sync ran with stale snapshot data and overwrote it back to `"Kg4 GN"`
- Identified the bug in `diff_snapshots_to_ops`: when building UPDATE params, it called `block_to_params` (full-row snapshot) even though only some fields changed in Loro
- The `_expected_content` guard was intentionally skipped when content didn't change in Loro, but `block_to_params` still included the old content value in the SET clause, allowing it to bypass the diff guard
- Implemented `block_diff_params` function to only include fields that actually differ between old and new snapshots
- Modified `diff_snapshots_to_ops` to use delta params for UPDATEs instead of full-row snapshots

**Challenges:**

- Race condition wasn't obvious: the SQL update succeeded, but a concurrent async Loro reconciliation clobbered it with stale data
- The diff guard logic was counterintuitively allowing the overwrite because excluded fields were still in the SET clause without guards
- New test failures emerged post-fix: `sql_only` now shows CDC churning on no-op property updates due to JSON serialization order differences in the diff guard

**Status:** Cross-executor variant now passes; however, `sql_only` variant reveals a secondary issue with properties column diff detection that needs investigation.
