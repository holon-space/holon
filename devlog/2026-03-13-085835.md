## Fix: editable_text nodes missing entity/operations from render_block

**Context**: The PBT test is now detecting the bug we set out to catch — `EditableText` nodes created by `render_block.rs` lack entity and operations metadata, so they can't be found in the display tree when the blur handler tries to fire `set_field`.

**What happened**:

- PBT generated an `EditViaDisplayTree` transition for `block:-p7-29ei--q12-1` with new content `"so  w ra  zk  z"`
- The SUT walked the display tree and asserted that an `EditableText` node with `entity["id"] == "block:-p7-29ei--q12-1"` exists
- **Assertion failed**: No such node was found — only 20 `table_row` nodes exist in the tree, all with different block IDs
- This confirms the bug: `render_block` calls `ViewModel::element(...)` which creates nodes with empty `entity: HashMap::new()`

**Root cause**: The `block:-p7-29ei--q12-1` block is being rendered as a table_row (via one of the main panel/sidebar queries) rather than as an editable text block. The display tree shows it's part of a PRQL query result set, not a standalone text block.

**Analysis**: The transition is trying to edit a block that's being rendered as a *table cell* (in a query result), not as an editable text field. The reference model treats all text blocks as independently editable via UI, but in the actual shadow tree, some blocks are embedded in query result tables where they appear as `table_row {content: "...", id: "..."}` — not `EditableText` nodes.

**Decision**: The precondition for `EditViaDisplayTree` needs to be tightened. We should only allow this transition on blocks that:
1. Are `ContentType::Text`
2. Are **not** children of query result tables (i.e., not sourced via `::src::0` or `::render::0` blocks)

Or alternatively: walk the display tree and check that an `EditableText` node actually exists before generating the transition.

**Status**: Bug surfaced correctly; transition generation needs precondition refinement to avoid trying to edit blocks embedded in query results.
