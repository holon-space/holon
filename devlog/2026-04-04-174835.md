## Enhanced Log Analysis: Instrumentation & JSON Format Support

**Context:** The log analysis tools revealed that sync cycles take ~15s per cycle and span multiple components, making it hard to trace end-to-end. Additionally, regex-based log parsing is fragile. Added structured tracing spans and JSON log format support to make logs machine-parseable and correlation trivial.

**What happened:**

- Instrumented 4 key sync functions (`mcp_full_sync`, `sync_entity`, `resource_fetch`, `subscription_resync`) with `#[instrument]` macros, giving each operation a unique span ID that threads through all child logs
- Migrated log messages from string interpolation (`"[McpSyncEngine] Got X records..."`) to structured fields (`records=X, new=Y, updated=Z`), making them queryable in JSON format
- Added `:json` suffix support to `HOLON_LOG` destinations (e.g., `HOLON_LOG=stderr,file:///tmp/holon.json:json`)
- Uses `tracing-subscriber`'s JSON formatter with full span context — each JSON line includes the complete span chain for correlation
- Suppressed noisy Demux CDC logs: events with 0 subscribers now log at TRACE instead of INFO, eliminating ~22% of log volume

**Decisions:**

- Structured fields over string formatting: Makes logs queryable with `jq` and compatible with observability systems (SigNoz, etc.)
- `:json` as a destination suffix rather than global flag: Allows simultaneous human-readable stderr and machine-parseable JSON files
- Kept both `#[instrument]` naming and explicit structured fields: Spans provide correlation, fields provide detail

**Status:** Both task 4 and task 6 complete and compiling. Logs now emit structured spans with entity/provider context and can be output as JSON for robust script analysis.
