## Fixed Trailing-Whitespace Divergence in PBT (Bug C)

Debugged and fixed the last bug in the `sql_only` variant of the property-based tests, where org-mode content with trailing spaces on the title line diverged between the reference state and SQL storage.

**What happened:**

- Identified root cause: `Mutation::{Create,Update}::apply_to()` in `crates/holon-integration-tests/src/pbt/types.rs` used `content.trim_end()` on the entire string, but the org serializer (`serialize_block_recursive` in `org_utils.rs:144`) trims only the **first line** (title), and the parser reads it back normalized
- For multi-line content like `"nL q \npAWaeG..."`, ref_state kept the trailing space while SQL stored it normalized, causing divergence
- Added `normalize_content_for_org_roundtrip(content, content_type)` helper that splits on first `\n`, trims the title line, preserves body verbatim, and passes Source blocks unchanged
- Applied normalization in both `Mutation::Create` and `Mutation::Update` apply paths
- Verified fix: `general_e2e_pbt_sql_only` now passes 50/50 cases (was failing ~10/50 before)

**Challenges:**

- Initial diagnosis required tracing through both the serialization layer (org_utils) and deserialization (parser) to spot the asymmetry — the trim happened in two different places with different scope
- Had to distinguish between "trim the whole string" vs. "trim just the headline" semantics in org-mode

**Status:**

Bug C (sql_only variant) is fixed. Bug #2 (peer merge blocks not propagating to SQL with Loro enabled) remains pre-existing and scoped out; handoff doc updated with shrunk repros isolating it as the next layer.
