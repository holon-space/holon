## Code Review & Plan Refinement for Mobile Focus-Chain Bar

**Context:** A plan for rendering a mobile-optimized focus-chain navigation bar on Android was submitted. The initial design made several architectural assumptions that didn't align with the actual codebase — primitive limitations, missing DSL features, and incorrect layer assignments.

**What happened:**

- Performed deep code audit across `render_dsl.rs`, `widget_builder.rs`, `entity_profile.rs`, `reactive.rs`, and mobile integration points
- Identified 12 critical hypotheses about DSL composition, streaming collections, ops resolution, and IME integration — validated each against the actual codebase
- Found 4 **fatal** issues: `row()` cannot host streaming collections (panics on `CollectionData::Streaming`); `separator:` and `vline()` don't exist; Rhai is a parse-time layer only, not a runtime; per-arg data sources are unsupported
- Discovered ops are scheme-level (not entity-level) and repeat identically across the focus chain; variants don't control ops; `set_focus` deliberately avoids bumping `ui_generation`
- Confirmed Android IME integration is missing entirely — no soft-keyboard inset signals exist
- User rejected the "dedicated widget" shortcut and requested primitives-first approach instead

**Challenges:**

- **Streaming collections in `row`/`col`**: These call `into_static_items()` which panics on streaming data. Solution required: refactor the `ShadowBuilder` trait to support `ReactiveView`-driven children, or introduce a new `StreamingRowBuilder` variant.
- **Per-arg data sources**: The macro extracts `Collection` from `ba.ctx.data_source` (singular, inherited). Nested collections with distinct sources (outer `focus_chain()` + inner `ops_of(col)`) need a new interpreter primitive and data-source resolver.
- **Ops deduplication**: All `block:*` entities share the same five ops by scheme. The chain repeats them four times. Needs explicit deduplication logic or scope-labeling.
- **IME insets**: Required a new signal path: `ime_bottom_px: Mutable<f32>` threaded from Android JNI through `UiState`, not auto-derived from `ui_generation`.

**Decisions:**

- Commit to **four new DSL primitives** (streaming `row`/`col` with `collection:` + `item_template:`, `separator:`, `vline`/`hline`) rather than a bespoke widget — enables reuse and user customization
- Drive focus chain from a **standalone signal graph** (`focused_block → parent-walk → SignalVec<FocusedEntity>`) separate from `ui_generation`, to avoid multiplying cursors
- Phase the work: (1) per-arg data sources + interpreter, (2) streaming row/col builders, (3) separator/vline widgets, (4) IME integration, (5) ops resolution refactor

**Status:** Refined plan under construction; awaiting completion of rewritten plan document with phased implementation strategy and explicit validation tasks for each DSL primitive.
