## Drag-Drop Handoff Continuation: Verify, Revert, Diagnose #10

**Context:** Picked up `HANDOFF_DRAG_DROP.md` to close out remaining open points. The architecture/wiring was already done; the gate (`DRAG_DROP_ENABLED = true`), inv16-as-panic, and widget binding fix were in place from the previous session. Goals: verify open points #3 (PRQL templates), #5 (widget registry audit), and pin down whether #10 (post-move nested rendering) was a renderer bug or something deeper.

**What happened:**

- Open point #3 — REVERTED. Confirmed `orgmode_hierarchy.prql` and `todoist_hierarchy.prql` are not loaded at runtime: the only programmatic emitter is `TODOIST_HIERARCHY_CTE` in `crates/holon-todoist/src/queries.rs`, which builds its CTE from a Rust string constant; the `.prql` files are referenced only from `crates/holon/tests/json_aggregation_e2e_test.rs` via `include_str!`, and that test strips render directives before compiling. Reverted both files to their pre-drag-drop content. Added a doc comment to `assets/default/types/block_profile.yaml` calling out that it is the canonical per-block render path.
- Open point #5 — VERIFIED CLEAN. `shadow_builders/mod.rs` has only one direct `interp.register(...)` call (`column`), already in the extend list at mod.rs:42. The three `interp.register_value_fn(...)` calls (ops_of, focus_chain, chain_ops) are also in the extend list. No additional names need registering.
- Open point #10 — DIAGNOSED ROOT CAUSE. Reproduced the bug via:
  ```
  PBT_WEIGHT_SPLIT_BLOCK=0 PBT_WEIGHT_MOVE_UP=0 PBT_WEIGHT_MOVE_DOWN=0 \
  PBT_WEIGHT_INDENT=0 PBT_WEIGHT_OUTDENT=0 PBT_WEIGHT_DRAG_DROP_BLOCK=8 \
  PROPTEST_CASES=10 cargo test -p holon-integration-tests \
    --test general_e2e_pbt -- sql_only
  ```
  Right after a successful `DragDropBlock`, the inv10i diagnostic shows `IVM MATVIEW INCONSISTENCY DETECTED: Data rows (from matview): 1 IDs ... Reference model: 17 total blocks, 3 expected visible. Missing from matview: [target, source, third-sibling]`. The entire main-panel matview collapses — not just the moved block. The hypothesis updates from "renderer doesn't recurse" to "Turso IVM doesn't reconcile recursive `CHILD_OF*0..20` joins after `parent_id` mutation".
- Open point #1b — EXPANDED. Discovered that even with `DRAG_DROP_ENABLED=false`, sql_only fails reproducibly on `MoveUp`/`MoveDown`/`Indent`/`Outdent` with backend "no previous/next sibling" errors that disagree with the reference model's `previous_sibling`/`next_sibling`. Likely the same IVM root cause as #10 (parent_id/sequence mutations don't propagate through matviews). Documented in handoff with isolation env vars to keep drag-drop work testable around it.

**Challenges:**

- Initially confused by proptest shrinking output where the shrunk minimum looked drag-drop-free; turned out the underlying IVM bug is shared between drag-drop and Indent/Outdent/MoveUp/MoveDown, so disabling drag-drop just shifts the shrink target.
- Could not fix #10 in this session — the root cause is in matview/IVM territory, far from the drag-drop wiring layer this branch owns.

**Decisions:**

- Keep the drag-drop wiring as-is. The architecture is sound; the failures are in a layer below.
- Drop dead PRQL template edits rather than chase a phantom code path.
- Leave the IVM root cause for a separate spec (probably belongs alongside the existing `turso-chained-matview-hang` and `turso-ivm-context-param-preload` skills).

**Status:** Drag-drop wiring + handoff are complete. Remaining gates (#7 GPUI smoke test, #8 weight tuning, #4 UX) need a real GPUI window. #10 root cause documented; fix is out of scope for this branch.
