# Subtree Sharing — Phase 2 follow-ups after Task 3

## Where we are

Phase 1 (share/accept over iroh, `pull_from_peers`) landed earlier. **Phase 2 Task 3 (snapshot persistence + startup rehydration) has now landed too**, per `plans/b-please-write-this-wild-squid.md`. Coverage: 76/76 sync tests pass, including the extended `subtree_share_round_trip_pbt` action-stream PBT (Edit, SettleSaves, PullBtoA, RestartA/B, CorruptSharedOnA) with invariants P-KEY, P-MOUNT, P-REG, P-CONTENT, P-CONVERGE, P-DEGRADED-ON-CORRUPT, P-NO-SILENT-CORRUPT, P-NO-TMP-LEFTOVER. Two supplementary unit tests cover 150 ms debounce (`save_worker_coalesces_burst`) and read-only-dir error surfacing (`readonly_shares_dir_emits_degraded`).

**What works today**: share + accept persist `shares/<shared_tree_id>.loro` atomically; `device.key` is loaded/generated once per storage dir; a save worker debounces commits at 150 ms; startup walks global-tree mount nodes and rehydrates each share (register in `SharedTreeSyncManager`, re-advertise, spawn save worker); corrupt snapshots are quarantined as `<id>.loro.corrupt-<ts>` with `ShareDegraded::SnapshotLoadFailed` on the bus.

**What does not work yet**: a process restart drops all known-peer addrs, so cross-peer sync after restart needs manual re-handshake. A graceful shutdown does not flush pending debounced saves. There is no UI; ops go through the MCP `execute_operation` surface only.

## Follow-ups in priority order

### 1. Phase 2 Task 2 — known_peers persistence + auto-resync

**Why this is next:** Task 3 delivered "shares survive restart". Without Task 2, shares on A and B cannot reach each other after either side restarts — A's advertiser comes back up on a fresh iroh endpoint addr, B's cached addr for A is stale (and not even persisted today), so `pull_from_peers` returns zero synced peers.

**Scope:**
- Capture the dialer's `EndpointAddr` on inbound connections. In `crates/holon/src/sync/iroh_sync_adapter.rs`, `sync_doc_handle_connection` (around line 150) receives a `Connection` — extract `conn.remote_node_id()` / `conn.remote_address()` and thread it up to a callback on `IrohAdvertiser`.
- Wire a callback: `IrohAdvertiser` grows a `on_peer_connected: Arc<dyn Fn(shared_tree_id, EndpointAddr)>` or a broadcast channel. `LoroShareBackend` subscribes and calls `self.remember_peer(shared_tree_id, addr)`.
- Persist `known_peers` to disk per share. Options: extend `shares/<id>.loro` metadata (embedded in the shared doc itself under a `_meta` map) OR a sidecar `shares/<id>.peers.json`. **Recommended: sidecar JSON.** CRDT-syncing peer addrs between A and B is undesirable (A shouldn't learn B's private IPs via the tree); per-peer local state belongs outside the shared CRDT.
- On rehydration, load the sidecar and populate `known_peers`.
- Rename `pull_from_peers` → `sync_with_peers` (the current name suggests unidirectional pull, but the function actually initiates full bidirectional sync via `sync_doc_initiate`).
- **Auto-resync on local commit**: each shared doc already has a `subscribe_root`-backed `SaveWorker`. Attach a **second** subscriber that, on non-remote commits, triggers a sync round to known peers. Filter out remote-origin commits by comparing the change's peer ID to the local shared-doc peer ID via `share_peer_id::stable_peer_id(device_key, shared_tree_id)`.
- Debounce the auto-resync separately from the save debounce — suggested 500 ms for sync vs the 150 ms for save.

**Verification:**
- Extend `subtree_share_round_trip_pbt` with a `CrossPeerSyncAfterRestart` action that restarts A, then edits on B, and asserts A picks up the edit via auto-resync (no explicit `PullBtoA` action). Drop the "Cross-peer pull after restart is explicitly **not** asserted" note in the v4 plan's Verification section once this lands.
- Manual: Mac and iOS sim with holon-live MCP both running; share on Mac, accept on sim, kill Mac, restart, edit on sim, see edit on Mac without any user action.

**Files of interest:**
- `crates/holon/src/sync/iroh_sync_adapter.rs:150` — `sync_doc_handle_connection` (extract remote addr here).
- `crates/holon/src/sync/iroh_advertiser.rs` — plumb the addr callback.
- `crates/holon/src/sync/loro_share_backend.rs:238-245` — `remember_peer` (already exists; wire it up).
- `crates/holon/src/sync/loro_share_backend.rs::spawn_save_worker` — consider unifying the auto-resync worker here or creating a sibling `spawn_sync_worker`.

**Estimate:** medium. The addr capture is ~50 lines. The sidecar persistence is ~30 lines. Auto-resync worker is ~80 lines. PBT extension is ~60 lines. Allow 1–2 days including the cross-rig manual test.

### 2. Phase 2 Task 1 — share/accept UI

**Why:** ops currently only fire via `execute_operation` over MCP. End users cannot share a block without hand-writing JSON-RPC.

**Scope on GPUI (primary per `gpui_primary_frontend` memory):**
- Context menu on any block → "Share subtree". Calls `execute_operation("shared_tree", "share_subtree", {"id": "<stable_id>", "retention": "full"})`. Copy the returned ticket to clipboard, show a modal with:
  - The ticket (selectable, copy button)
  - A security warning referencing the bearer-capability threat model in `docs/SUBTREE_SHARING.md`
  - The quarantine path if a degraded event fires before dismissal
- Accept UX: a "Paste share ticket" entry in the main drawer + a parent-block picker. Calls `execute_operation("shared_tree", "accept_shared_subtree", {...})`.
- Subscribe to `DegradedSignalBus` via a bridge in `LoroShareBackend::degraded_bus()`. Render:
  - `SnapshotSaveFailed` → yellow toast with retry CTA
  - `SnapshotLoadFailed(quarantine_path)` → red modal: "Share `<id>` could not be restored. Your edits before the corruption are quarantined at `<path>`. Re-accept the ticket from the other peer to restore."
  - `RehydrationFailed` → yellow toast

**Scope on iOS/Flutter:** same two entry points. Check `frontends/flutter/` for how FRB exposes `execute_operation` today and add the two ops there.

**Blocker:** bridging `DegradedSignalBus::subscribe()` (a `tokio::broadcast::Receiver`) across FRB needs a small adapter. Check how other `Stream`-shaped things are exposed in `crates/holon/src/sync/ui_watcher.rs` (the `UiEvent` stream — `Model D — Tagged Enum Stream` memory).

**Estimate:** medium-large. 2–3 days of focused GPUI work; iOS depends on FRB bridge shape.

### 3. Block.external_id field + SHARE_ROLE property (scheduled deferral)

**Why:** the MVP chose "org properties later" for the mount-block projection. Every downstream query has to traverse Loro tree metadata instead of reading SQL. This is also the cleanup path for Todoist's Loro-meta `external_id` (see `loro_backend.rs:832-857`), which lives outside the `Block` struct entirely today.

**Scope:**
- Add `external_id: Option<String>` to the `Block` struct in `crates/holon-api/src/block.rs:257-301`. Entity macro will add a SQL column.
- Migrate Todoist's `external_id` storage: `LoroBackend::set_external_id`/`get_external_id` should go via the Block row instead of Loro tree meta. Two-phase migration: dual-write during rollout, then read-only from Block.
- When the Loro→Block materialization site materializes a mount node (currently skipped in `loro_backend.rs:1118-1127`), project `MOUNT_SHARED_TREE_ID` → `Block.external_id = "shared-tree://<id>"` and emit `SHARE_ROLE` as a property. This requires mount nodes to become visible as Block rows; see follow-up #4.
- Org parser/renderer already round-trip `Block.properties` through `:PROPERTIES:` drawers (no allowlist per `holon-org-format/src/models.rs:655-713`), so `SHARE_ROLE` comes along automatically.

**Verification:**
- Extend `general_e2e_pbt.rs` to seed a mount block, edit it, round-trip through org, reload — confirm `external_id` and `SHARE_ROLE` survive. (This is the one PBT in the project where additions are welcome per CLAUDE.md.)
- MCP: `execute_query("SELECT id, external_id, properties FROM block WHERE external_id LIKE 'shared-tree://%'")` returns every active share.

**Open design question:** Should `external_id` unify with the old Loro-meta entry or coexist? Unification is cleaner; coexistence is safer during rollout. Recommend unification with a one-shot migration on first startup that reads all Loro-meta `external_id`s and writes them to the new Block column.

**Estimate:** medium. The schema change + migration is the bulk; the projection is small. 1–2 days.

### 4. Mount node visibility as Block rows

**Why:** Follow-up #3 needs mount nodes to materialize as Block rows, which they currently don't — `loro_backend.rs:1118-1127` skips them and grafts their shared children up to the mount's parent. As long as mount nodes are invisible, the `external_id` projection has nowhere to land in SQL.

**Scope:**
- In `loro_backend.rs:1118-1127` (and the parallel traversal at `:1191-1199` for `list_children`), stop skipping mount nodes. Emit a `Block` row for the mount with `external_id` set and the shared tree's children nested as children of the mount block (not the mount's grandparent).
- Update any rendering that assumed children of a share's host parent include the shared subtree directly; children now go through the mount block.
- This may cascade into navigation, command resolution, etc. — the all-in-LoroTree architecture memory notes that "mount nodes are invisible to the Block layer" is a load-bearing assumption in the current code.

**Verification:**
- Re-run `general_e2e_pbt.rs` with scenarios that traverse share boundaries.
- Check all frontends still render shared subtrees correctly (GPUI primary, others as available).

**Estimate:** medium-large because of the cascade. Pair with #3.

### 5. Shutdown flush strategy (V4 from the plan)

**Why:** Today `flush_all()` exists on `LoroShareBackend` but is only called manually from the PBT's `RestartA` action. On graceful process shutdown, pending edits in the 150 ms debounce window can be lost.

**Scope:**
- DI container (fluxdi) has no drop hook today. Options:
  - Add a `Shutdown` marker resolved by application-level code (top-level signal handler) that calls `backend.flush_all().await` before exit.
  - Register a `ctrl_c` handler in the CLI / each frontend's main that flushes before exit.
- Signal handler is the simplest path: the frontend binaries already have tokio main functions; `tokio::signal::ctrl_c()` awaited there, followed by `backend.flush_all().await`.

**Verification:** kill -TERM the process mid-edit, confirm the edit survives on restart. The debounce window is small enough that testing this deterministically is fiddly — accept a weak assertion (edit survives most of the time) or force a large debounce in a test-only config.

**Estimate:** small. 2–3 hours.

### 6. Low-priority cleanups

- **Orphan snapshot GC.** `rehydrate_shared_trees` logs orphans at `info!` but never deletes them. Per the plan, this must be user-driven (requires UI confirmation so we don't delete snapshots that will soon have a restored mount node). Add a `shared_tree:gc_orphans` operation once UI exists.

### 7. Architectural opportunities (not scoped but worth noting)

- **`extract_for_share` + `commit_share_prune` as a general primitive.** The split (in `crates/holon/src/sync/shared_tree.rs`) is reusable for any "fork a subtree, do I/O, commit the mutation atomically on success, drop on failure" flow. Good candidate: backup/export operations, snapshot-before-destructive-refactor operations.
- **SaveWorker as a utility.** The `subscribe_root → Notify → debounce → save → AbortHandle cleanup` pattern in `loro_share_backend.rs::spawn_save_worker` overlaps with `loro_sync_controller.rs:179-184` (which is non-debounced). A unified `DebouncedCommitWorker` in `sync::` would let the sync controller share the implementation, and new subsystems (e.g., the global doc's own save cadence) can reuse it.
- **Degraded signal pattern.** `DegradedSignalBus` is share-specific today. Generalizing to a process-wide degradation bus (covering cache staleness, sync lag, disk-full, etc.) would let the UI centralize its degraded-mode banner logic. Keep the per-feature enum variants in each feature's module and re-export through a single `tokio::sync::broadcast`.
- **Selective sync.** Snapshot-file-per-share means we already have a discrete on-disk artifact per share. Primitives for unshare-but-keep-local (`snapshot_store.save` + `manager.remove` + mount node removal) are close to free if we want them.

## Ground rules for whoever picks this up

- Never swallow errors with `.ok()`/`filter_map+ok`/warn+continue. `/fix-defensive-code` skill audits the codebase.
- `tee` before filtering any test output: `cargo test -p holon --features iroh-sync --lib 2>&1 | tee /tmp/sync.log | tail -40`.
- Loro doc write-lock invariants in `LoroShareBackend` are load-bearing; the v4 `extract_for_share` / `commit_share_prune` split is specifically designed so durable I/O happens without holding Loro locks longer than needed. Don't fold them back into one function.
- New ops must land on entity `"shared_tree"`, not `"block"` (dispatcher silently picks the first provider on entity collision — see plan D5).
- The only existing PBT file that accepts additions is where it already lives per topic: `general_e2e_pbt.rs` for UI-visible behaviour, `sync_pbt.rs` for sync behaviour. User confirmed any `*_pbt.rs` is fine.
