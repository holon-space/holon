## Debugging `holon-live` displaying "[no result]" for Claude Code sessions

The `holon-live` instance showed `[no result]` placeholders instead of Claude Code sessions, conversations, and tasks, despite the backend database containing all the data (23 projects, 184 sessions, 878 tasks).

- Diagnosed the issue through parallel inspection of `/tmp/holon.log`, database state via `holon-direct` MCP, and GPUI shadow tree rendering pipeline
- Found that source blocks (holon_prql/holon_sql/holon_gql) rendered as `query_result()` which expected a `"result"` field missing from block rows, causing `[no result]` output
- Fixed by adding `is_holon_source` computed field to `block_profile.yaml` and a `holon_source` variant that renders `spacer(0)`, hiding query source blocks whose results are already displayed by parent `block_ref()` blocks
- Attempted to fix a secondary layout issue where heading blocks collapsed to zero height inside tree items by removing `h_full()` from ReactiveShell, but this broke the columns layout and required reverting

The root cause was architectural: `query_result` shadow builder didn't account for block row schema, and source blocks shouldn't render individually in trees when their parent heading already displays their results. The `holon_source` profile variant eliminates the `[no result]` text clutter while preserving the data display through `block_ref()`.

**Status**: Core `[no result]` issue fixed via profile change. Secondary layout collapse of heading content in tree items identified but not yet resolved — requires context-aware sizing in ReactiveShell to detect tree_item parent context.
