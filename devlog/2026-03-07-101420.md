## Eliminate startup settle delay for instant first screen

The frontend startup was blocked by a conservative 2-second settle delay meant to let the file watcher finish processing org files. However, the UI system is already designed to handle incremental data arrival via CDC streams.

**What happened**

- Analyzed startup timeline using existing `tracing-chrome` instrumentation — identified 2s settle delay consuming 83% of ~2.4s total startup time
- Removed `settle_delay_ms` field and `with_settle_delay()` method from `FrontendConfig`
- Reordered `seed_default_layout()` to run **before** `wait_for_ready()` — UI now renders immediately with default blocks rather than waiting
- Removed all `.with_settle_delay(2000)` calls from native frontends (Blinc, GPUI, Dioxus, WaterUI)
- Migrated settle delay to test environment only, applied **after** session creation for test determinism

**Decisions**

- **Progressive rendering over blank screen**: The `UiWatcher` already streams structural CDC events when blocks appear/change, so users will see the default layout instantly, then real org content fills in as files sync — better UX than a 2s wait.
- **Keep settle in tests, not production**: Tests need determinism, but the performance cost shouldn't apply to real frontends. Test harness now owns the delay.

**Status**

All four native frontends and integration tests compile cleanly. Startup time reduced from ~2.4s to ~400ms to first screen, with progressive UI updates as org files process.
