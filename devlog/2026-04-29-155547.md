## Comprehensive Code Review & Plan Refinement for Virtual-Child Streaming

A plan to implement virtual-child row streaming in the editor was reviewed against actual codebase behavior and found to contain several critical false assumptions that would cause runtime crashes and data corruption.

### What happened

- **Read and analyzed** `the-behaviour-seems-quite-rustling-sparrow.md` plan alongside relevant source files (`entity_profile.rs`, `view_event_handler.rs`, `sut.rs`, `tree.rs`)
- **Identified 8 major assumption failures**: tree-variant priority logic inverted, ID prediction mechanism broken, coexisting legacy/new virtual-child systems, synthetic ID parent crash risk, invariant definitions too coarse, product bugs papered over, and `focusable_rendered_block_ids` pollution
- **Performed cheap validations immediately** on variant priority (`entity_profile.rs:402,573`), ID flow through dispatch (`view_event_handler.rs:152`), virtual ID format parsing, registry wiring, and prelude defaults
- **Documented 5 open hypotheses** requiring follow-up work: CDC timing race conditions, correct defaults source (must read from loaded `EntityProfile.virtual_child.defaults`, not hardcode), sentinel parent invariant, static-path empty-tree suppression as potential product bug, and legacy path retention decision
- **Rewrote plan in place** with explicit invariant checks, split inv10j into 4 sub-validations (streaming/static/scheme/disappear), and introduced separate `editor_target_ids` collection to prevent `block_state.blocks.get(id)` crashes
- **Added Step 0 gate** that cancels work if core assumptions flip during initial validation tasks

### Challenges

- **Invisible coexistence**: `TrailingSlot` (new format `<scheme>:__virtual:<local>`) and `VirtualChildRowProvider` (legacy `virtual:<full-uri>`) both exist; `parse_virtual_id` only handles new format, causing silent failures in static paths
- **ID generation opacity**: Loro auto-generates UUIDs on `block.create` dispatch, but plan assumed synthetic IDs could be predicted â€” required mapping the existing `SplitBlock` syntheticâ†’UUID infrastructure

### Decisions

- Introduced `editor_target_ids` as separate collection instead of polluting `focusable_rendered_block_ids`, preventing cascading `block_state.blocks.get(id)` crashes downstream
- Created explicit Task queue for 5 expensive validations rather than blocking plan on uncertainty, allowing work to proceed once assumptions are gated

**Status**: Plan rewritten with holes patched, cheap validations passed, 5 follow-up validation tasks created as blockers before implementation begins.
