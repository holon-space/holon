## Reorganized UI PBT tests to eliminate circular dependencies

The `gpui_ui_pbt.rs` and `blinc_ui_pbt.rs` tests were living in `holon-integration-tests`, which created a problematic dependency structure: the integration test crate dev-depended on frontend crates, while those frontends would need to dev-depend on the integration tests to use the PBT infrastructure.

**What happened:**
- Moved `crates/holon-integration-tests/tests/gpui_ui_pbt.rs` → `frontends/gpui/tests/gpui_ui_pbt.rs`
- Moved `crates/holon-integration-tests/tests/blinc_ui_pbt.rs` → `frontends/blinc/tests/blinc_ui_pbt.rs`
- Added `holon-integration-tests` as `[dev-dependencies]` in both `frontends/gpui/Cargo.toml` and `frontends/blinc/Cargo.toml`
- Added `[[test]]` entries for the new test locations
- Removed all frontend-specific dev-deps from `holon-integration-tests/Cargo.toml` (holon-gpui, gpui, gpui-component, gpui-component-assets, holon-blinc, blinc_app, blinc_layout, blinc_theme)

**Decisions:**
- Kept the PBT infrastructure (`GeometryDriver`, `XcapBackend`, etc.) in `holon-integration-tests` rather than extracting a new crate — dev-dependency cycles are allowed in Cargo as long as they don't create actual circular compilation dependencies
- Each frontend now owns its UI PBT test binary, making the structure clearer and more maintainable

**Status:** All three crates compile successfully; dependency graph is now acyclic (frontends dev-depend on integration-tests, not vice versa).
