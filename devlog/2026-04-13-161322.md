## Setting up a Headless Devcontainer with VCS Isolation

**Context**
The project needed a reproducible development environment for headless Rust work (testing, building, running integration tests) that could be safely used from Claude Code worktrees without risking the main repository state.

**What happened**
- Built a three-file devcontainer setup (`.devcontainer/devcontainer.json`, `Dockerfile`, `post-create.sh`) targeting Debian bookworm with Rust nightly, `jj`, `cargo-nextest`, and `cargo-llvm-cov`
- Investigated GPU-accelerated UI rendering (GPUI) inside the container on macOS and determined it's not practical — the Linux VM in Docker Desktop lacks GPU passthrough, so rendering would fall back to llvmpipe over XQuartz/VNC
- Installed Claude Code inside the container (Node 20 via NodeSource, `@anthropic-ai/claude-code` global package) with a named volume at `~/.claude` for auth persistence
- **Tested jj workspaces with devcontainers** in `/tmp` and discovered that workspaces use relative pointers to the shared repo store — mounting only the workspace dir breaks jj inside the container; must either mount the parent or skip VCS entirely
- Implemented VCS isolation: masked `.jj/` and `.git/` with tmpfs mounts so they're unreachable from the container (safe by construction); parameterized volume names and `workspaceFolder` with `${localWorkspaceFolderBasename}` so the same config works for the main checkout and all worktrees under `.claude/worktrees/`
- Verified `devcontainer` CLI can build, start, and exec commands (interactive and non-interactive) from the command line

**Challenges**
- **jj workspace relative pointers**: Workspaces store `../../main/.jj/repo` as a relative path; masking `.jj` with tmpfs prevents access to the host's op-log, which is the desired outcome for isolation
- **Parameterizing volume names per-worktree**: Used `${localWorkspaceFolderBasename}` to derive per-worktree Cargo and target directories so concurrent containers don't collide

**Decisions**
- **VCS on the host only**: Agents inside the container can't run jj/git; all version control happens on the host where you retain full control. This trades convenience (in-container commits) for safety (no repo corruption risk)
- **Claude Code shares auth across worktrees**: Single named volume at `~/.claude` so you authenticate once and every container reuses the login

**Status**
Devcontainer is ready for use. On first boot in a worktree, run `jj st` to verify it errors ("no repo found"), confirming the VCS isolation is working. Rust toolchain, cargo-nextest, and Claude Code are all functional; use `devcontainer exec` to run commands from CI/scripts.
