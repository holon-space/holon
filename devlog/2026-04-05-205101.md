## Researching Zed Plugin Architecture as a Frontend Option

We investigated whether a Zed plugin could serve as an alternative frontend to the GPUI-based interface for Holon.

**What we found:**

- Zed extensions are Rust compiled to WASM (wasm32-wasip2) running in a Wasmtime sandbox, with APIs defined via WIT and exposed through `zed_extension_api`
- Extensions can provide language servers, tree-sitter grammars, themes, slash commands, debugger adapters, and HTTP capabilities
- **Critical limitation:** Extensions have no custom UI/panel API, no webview support, and no access to GPUI's closure-heavy GPU-backed rendering system
- The GPUI frontend uses `Application::with_platform().run()`, creates native windows, spawns tokio tasks for reactive streams, and uses `Entity<T>`, `Render`, and `IntoElement` APIs — none available in the WASM sandbox
- Additional constraints: no direct filesystem writes (only worktree reads), no TCP sockets (HTTP only), no persistent background event loops

**Key findings:**

- The Zed community has requested custom UI support since late 2023 (issues #6679, #37270, #21208), but the team has stated it's "a ways off" with no committed timeline
- An MCP server integration is viable — Holon could expose queries as Zed slash commands for the AI assistant, but this provides no custom UI rendering
- Forking Zed to embed custom views would require massive maintenance burden
- The standalone GPUI app remains the architecturally correct choice for now

**Status:** Standalone GPUI frontend is confirmed as the right approach; MCP server integration could be explored later as a complementary feature for Zed users.
