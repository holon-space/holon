## Cross-Frontend Geometry: Research Findings

### Context
Understanding how to query element positions and sizes after layout is critical for the `AccessKitDriver` — a test harness needs to know where to click, and ideally in a framework-agnostic way.

### What happened

- **Researched geometry APIs** across all 6 frontends (Flutter, Blinc, GPUI, Dioxus, Ply, WaterUI)
- **Mapped coordinate systems** (logical pixels vs screen pixels vs CSS pixels)
- **Identified which frontends already expose bounds** queries by element ID
- **Assessed feasibility** of adding geometry querying to GPUI and WaterUI

### Findings

**Good news (4/6 frontends ready):**
- **Blinc**: `query("id").bounds()` — best API, returns `ElementBounds { x, y, w, h }` as f32
- **Ply**: `ui.bounding_box(id)` — immediate-mode, straightforward
- **Dioxus**: `get_client_rect()` (async) or DOM eval — backed by real browser APIs
- **Flutter**: Via Semantics → NSAccessibility → `AXUIElement` — already working

**Fixable (1 frontend):**
- **GPUI**: Computes bounds in `prepaint()` but doesn't persist them. Could add a bounds-stashing layer: wrap `Div` to store `HashMap<ElementId, Bounds<Pixels>>` during prepaint, then expose a query API

**Not worth pursuing yet (1 frontend):**
- **WaterUI**: No public geometry API (internal to Hydrolysis renderer), lowest maturity (15/23 builders)

### Decisions

- **Annotator hook is now strongly justified**: The shared `RenderInterpreter::dispatch()` can tag every widget with its entity ID (via post-dispatch annotator), then each frontend queries bounds using its native API at test time
- **GPUI bounds-stashing is low-effort**: Worth adding as a small wrapper if we're serious about GPUI PBT coverage

### Status

Geometry querying is **viable on 4/6 frontends today**, and **achievable on 5/6 with modest effort**. The annotator + geometry query pattern can form the foundation of the `AccessKitDriver`. Ready to move forward with Phase 1 (shared Rust PBT) + geometry implementation per frontend.
