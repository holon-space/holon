## Shared ReactiveEngine Between GPUI and MCP via Dependency Injection

The MCP `describe_ui` tool was returning empty data because it couldn't access the `ReactiveEngine` that GPUI had. The goal was to create the engine early in GPUI's main and share it with both GPUI and the MCP server via dependency injection.

### What happened

- **MCP server (`server.rs`)**: Added `builder_services: Option<Arc<dyn BuilderServices>>` field to `HolonMcpServer` and updated constructor
- **DI threading (`di.rs`)**: Updated `run_http_server()` signature to accept `builder_services` parameter, threaded it through `McpServerHandle`
- **describe_ui rewrite (`tools.rs`)**: When `builder_services` is available, now calls `svc.get_block_data()` to get real render expressions and data rows from the shared ReactiveEngine instead of using headless interpretation
- **GPUI early creation (`main.rs`)**: Created `ReactiveEngine` before MCP server starts, passed it to both `run_http_server()` and `launch_holon_window_with_engine()`
- **GPUI window refactor (`lib.rs`)**: Added `launch_holon_window_with_engine()` accepting pre-created engine, made `make_interpret_fn` public, extracted shared `launch_holon_window_impl()`
- **Other frontends**: Updated all callers (`blinc`, `ply`, `flutter`, `mcp`) to pass `None` for `builder_services` — no behavior change

### Challenges

- **GPUI main.rs doesn't use FluxDI**: The implementation assumes FluxDI is used, but GPUI's main creates services manually without the DI container. The `Arc<OnceLock<BuilderServices>>` pattern still works for sharing, but may need adjustment if DI integration is desired later.

**Status**: All target packages build successfully (`holon-gpui`, `holon-mcp`, core libs). MCP's `describe_ui` now has access to real block data when called from GPUI; ready for testing.
