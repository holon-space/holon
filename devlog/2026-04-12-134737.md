## Debugging the Claude Code Tree View Display

**Context**: The holon-live instance was displaying Claude Code sessions/conversations/tasks but showing only `[no result]` placeholders, despite data being present in the database (23 projects, 184 sessions, 878 tasks).

**What happened**

- Diagnosed via `holon-direct` MCP that backend data and queries were working correctly, but the GPUI frontend rendered nothing
- Traced the issue to two problems: (1) source blocks with holon queries rendered as `[no result]` because the `query_result` shadow builder expected a `"result"` field absent from block rows, and (2) heading blocks' `block_ref()` content had zero height inside tree items due to `size_full()` collapse
- Root cause: The 5 heading blocks (cc-projects, cc-sessions, etc.) correctly resolved to `block_ref()` which contained query results, but the ReactiveShell wrapper used `.size_full()` (forcing `height: 100%`) which collapsed to zero inside tree_items lacking definite parent height
- Fixed the `[no result]` symptom by adding an `is_holon_source` computed field to `block_profile.yaml` and a `holon_source` variant that renders `spacer(0)`, hiding query/render source blocks whose results are already shown by parent heading's `block_ref()`
- Attempted to fix the layout by removing `h_full()` from ReactiveShell, but this broke the sidebar/columns which needed `flex_1()` to grow properly
- Reverted the layout change; the `holon_source` profile change alone eliminated the `[no result]` noise

**Challenges**

- Initial layout fix collapsed all content to zero height, requiring revert
- Identified the true issue as context-dependent sizing: `size_full()` works in columns (which use absolute positioning) but fails in tree_items (which have no definite height)
- Distinguishing between pre-existing layout bugs vs. regressions from profile changes

**Decisions**

- Opted for a profile-only fix (`holon_source` variant) to ship quickly rather than attempt complex layout surgery
- Will address the remaining heading block visibility issue (H3 from diagnostic session) with context-aware ReactiveShell sizing in follow-up work

**Status**: The `[no result]` items are eliminated; heading blocks in trees remain invisible (pre-existing issue awaiting layout fix via context-aware ReactiveShell height handling).
