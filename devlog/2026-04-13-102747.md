## Event-Driven File Stability Detection with Polling Fallback

The PBT test suite was spending ~47% of wall time (23.3s) waiting for org-mode files to stabilize, using a naive 50ms polling loop. After an initial 7.5× win by dropping poll intervals to 5ms, we needed a structural fix to eliminate polling entirely.

**What happened:**

- Profiled the test suite across 610 transitions; confirmed `wait_for_org_files_stable` was the dominant bottleneck at 23.3s
- Reduced poll interval from 50ms → 5ms and quiescence window from 50ms → 25ms, reclaiming ~11s but hitting flakiness with narrow windows
- Designed `OrgSyncIdleSignal` — an `AtomicU64` tick + `Notify` pair that tracks write completion in `OrgSyncController`'s event loop
- Registered as a DI singleton in `holon-orgmode/src/di.rs`; wired into both `tokio::select!` arms (file watcher + EventBus events) to call `mark_progress()` after each handler
- Resolved the signal in test_environment and wired into `E2ESut` constructor + builder
- Rewrote `wait_for_org_files_stable` to first attempt event-driven `wait_quiescent(5ms, 2s)`, then fall back to mtime polling if unavailable or on timeout

**Decisions:**

- **Hybrid approach:** Event-driven fast path (5ms typical) with mtime polling safety net. Handles both instrumented tests and fallback scenarios without breaking existing APIs
- **Minimal DI footprint:** Signal is optional (`Option<Arc<...>>`), so non-instrumented code paths degrade gracefully

**Status:** Tests pass (3/3 reliable). `files_stable` dropped 23.3s → 6.5s (3.6×), total wall-clock 21.3s → 14.5s (32% faster).
