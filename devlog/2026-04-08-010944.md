## Refactored GPUI frontend styling + LayoutStyle struct + theme hot-reload fix

### Context
The GPUI frontend needed a LogSeq-inspired visual refresh, with all hardcoded layout constants extracted into a configurable struct. Additionally, theme changes in settings were not applying at runtime due to holon's custom YAML colors never being wired into gpui_component's global theme system.

### What happened
- **LogSeq redesign** applied across 6 builder files: removed pill-style badges (now plain teal text), enlarged section titles (16→28px, SEMIBOLD→BOLD), increased line heights, expanded spacing (indent 24→28px, row height 28→32px, content padding 32→40px), added CANCELLED state with `✗` icon
- **LayoutStyle struct** created in `frontends/gpui/src/render/builders/style.rs` with ~20 configurable fields (indent, bullet size, row height, sidebar width, padding values, font sizes, etc.), all defaults preserving current values
- **GpuiRenderContext** extended with `layout_style: Mutable<LayoutStyle>` and methods for reactive access (`style()`, `with_layout_style()`, `layout_style_signal()`)
- **11 builder files** refactored to use `ctx.style()` instead of hardcoded constants: `tree_item.rs`, `columns.rs`, `text.rs`, `section.rs`, `badge.rs`, `state_toggle.rs`, `checkbox.rs`, `icon.rs`, `card.rs`
- **Theme hot-reload fixed**: New `apply_holon_theme()` function in `lib.rs` maps holon's YAML `ThemeColors` (primary, text, background, sidebar, border, success, error, warning, etc.) onto gpui_component's global `ThemeColor`. Called at window launch instead of generic `Theme::change(mode, None)`, so theme palette changes now apply immediately

### Challenges
- **Worktree git state confusion**: jj worktrees share the working directory but have separate git state. Pre-existing changes in the repo made it initially unclear whether the agent's new files (`style.rs`, `mod.rs` changes) had been created. Resolved by directly checking file contents and understanding jj's file-sharing model.
- **Theme application architecture**: holon's YAML theme config (`load_theme_def()`) was loaded but never connected to gpui_component's global `ThemeColor` struct. Solution required mapping all 10+ color fields and ensuring the mapping happens at initialization.

### Decisions
- **Struct mutability via `Mutable<>`** — LayoutStyle lives in `Mutable` so it can be swapped/updated reactively without requiring rebuild. This enables future settings UI to change layout dynamically.
- **Theme application at window creation** — rather than patching theme changes reactively, centralized all color mapping into `apply_holon_theme()` called once at startup. This keeps theme logic in one place and avoids scattered `tc()` call-site changes.

### Status
All changes compile cleanly. App launched and tested — LogSeq-inspired UI applied, all constants extracted into `LayoutStyle`, and theme colors now correctly apply from holon.yaml at startup. Ready for user feedback on visual design and potential layout/theme customization in future iterations.
