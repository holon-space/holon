## Debugging GPUI Reactive Bridge and Fixing describe_ui MCP

The GPUI frontend wasn't rendering any content after introducing futures-signals between Turso and the frontends. The backend was processing correctly, but the UI stayed blank. Additionally, the `describe_ui` MCP endpoint was useless—returning hardcoded empty data instead of actual block content.

### What happened

- **Identified channel bug**: `try_send` on a bounded(1) channel was dropping ViewModels. Initial empty ViewModel filled the channel, then real data arrived but `try_send` failed because GPUI hadn't consumed yet. Signal wouldn't re-emit (no changes), leaving the screen blank.
- **Fixed reactive bridge**: Replaced `try_send` with `send().await` in the signal-to-GPUI flow so the tokio task blocks until the channel is consumed.
- **Found runtime panic**: `BlockRefView::new` called `tokio::runtime::Handle::current()` from GPUI context (outside tokio runtime). Same `try_send` issue existed here.
- **Threaded rt_handle**: Passed `Arc<tokio::runtime::Handle>` through `EntityRegistry` to `BlockRefView` and `LiveQueryView`, eliminating `Handle::current()` calls.
- **Discovered MCP architectural flaw**: `describe_ui` MCP used `HeadlessBuilderServices` with hardcoded empty data (`let data_rows = vec![]`), completely decoupled from the reactive engine powering GPUI.
- **Redesigned MCP bootstrap**: Planned to create `ReactiveEngine` once in `main.rs`, pass it as `Arc<dyn BuilderServices>` to both MCP server and GPUI, so both share real live data.

### Challenges

- **Channel-GPUI timing mismatch**: Initial empty ViewModel consumed the bounded channel's capacity before real data arrived, causing silent drops. Solution: blocking `send().await` ensures consumption before next emission.
- **Runtime context violation**: Can't call `Handle::current()` outside tokio runtime. Solution: explicit handle passing through DI instead of implicit lookup.
- **MCP data isolation**: `HeadlessBuilderServices` was a stub returning empty data. The real fix requires architectural change: single `ReactiveEngine` instance shared between all consumers.

### Decisions

- **Blocking sends over try_send**: Guarantees ordering and prevents data loss when channel bounded capacity is tight. Trade-off: task blocks briefly, but this is acceptable for UI reactivity.
- **Explicit runtime handle in DI**: Cleaner than relying on `Handle::current()` and enables testing in non-tokio contexts.
- **Unified ReactiveEngine**: Single source of truth for all data access (MCP, GPUI, tests). Requires refactoring bootstrap to construct it once and share via `Arc`.

**Status**: Build succeeds with no panics; blocks render and logs show correct data flow. UI still blank—likely signal not re-firing after initial ViewModel. MCP redesign planned to create single shared `ReactiveEngine` in main.
