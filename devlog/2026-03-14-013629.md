## Setting Up MCP Proxy for Auto-Reconnect After App Restarts

**Context**: Claude Code required manual MCP client reconnection every time holon-live (the embedded HTTP MCP server) restarted, which was error-prone. A transparent proxy was needed to survive backend restarts.

**What happened**:

- Researched existing MCP proxies (`mcp-proxy`, `mcp-rust-proxy`) and found they didn't support proxying to already-running external HTTP servers
- Forked `mcp-rust-proxy` and enhanced it with external server support: made `command` optional, added `streamableHttp` transport type to speak holon-live's HTTP protocol
- Created `/Users/martin/Workspaces/pkm/holon/mcp-proxy.yaml` with config pointing to `http://127.0.0.1:8520/mcp` over streamable HTTP with `restartOnFailure: true`
- Verified proxy successfully connected and exposed all 24 holon-live tools via stdio
- Updated `~/.claude.json` to connect to holon-live via proxy (stdio mode) instead of direct HTTP
- Ran `/mcp` and successfully reconnected to proxied holon-live

**Challenges**:

- **Tool name prefix collision**: Initial proxy output was `mcp__proxy__holon_live__*` (double-prefixed). Fixed by deploying `toolNamesPrefix` config option; accepted `mcp__holon-live__holon_live__*` pattern and re-granted permissions
- **Stale session on backend restart**: After restarting holon, all tool calls returned HTTP 404. The proxy survived the restart but never re-initialized its streamable HTTP session. Detected transport errors don't currently trigger reconnection

**Status**: Proxy is live and stdio connection is stable, but reconnect-on-error logic still needs implementation — currently requires manual `/mcp` after backend restarts, though less frequently than before.
