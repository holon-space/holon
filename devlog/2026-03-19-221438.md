## Evaluating Hooks vs LSP Diagnostics for Rust Development

The question arose whether git hooks could improve the holon development workflow similarly to how they work in a 668K-line TypeScript codebase, and whether constant compile-error feedback during multi-file changes would be helpful or distracting.

- Analyzed DevMoses's Reddit post on Claude Code maturity levels, which advocated for post-edit hooks running per-file typechecks in TypeScript — catching errors immediately rather than 10 edits later
- Identified that TypeScript's per-file `tsc` with full project context resolves quickly enough (~milliseconds) to run after every edit, whereas Rust's `cargo check` always resolves the full dependency graph and takes seconds
- Recognized that DevMoses's use case was autonomous fleet agents in worktrees (Level 5), where hooks act as guardrails for unsupervised code generation — different from interactive human-guided refactoring
- Concluded that for interactive multi-file Rust refactoring, compile errors after every edit would be noise: the developer already knows intermediate edits are broken and is following a mental plan across files

**Challenges**: Reddit's aggressive programmatic access blocking (CAPTCHA, API blocks) required using Playwright to access the original post and comments.

**Decisions**: 
- **Skip real-time hooks for interactive work** — LSP diagnostics are already the right granularity, queryable on demand rather than after every intermediate edit
- **Pre-commit `cargo check` hook is valuable** — acts as a structural gate at the moment of "I'm done," catching errors with low noise
- **Hooks make sense for autonomous agents** but not for human-in-the-loop interactive sessions, since humans can maintain mental models that agents cannot

**Status**: Concluded that holon's current workflow (LSP + E2E PBT + interactive escalation) is well-suited to Rust's compilation model; pre-commit hook recommended, but real-time post-edit hooks would be counterproductive.
