## Batch Org File Sync Operations

Org file parsing was submitting block creates/updates/deletes one-by-one to the database, causing 100s of individual transactions instead of one per file.

- Identified that `OrgSyncController.on_file_changed()` looped over parsed blocks and called `create_block()` sequentially, each triggering an individual auto-committed INSERT + event publish
- For a 100-block file: 200 separate SQL statements hitting the database
- Added `execute_batch()` method to `OperationProvider` trait with sequential default implementation
- Implemented transactional override in `SqlOperationProvider`: collects all SQL statements and events, executes via single `db_handle.transaction()`, publishes via `event_bus.publish_batch()`
- Refactored `OrgSyncController.on_file_changed()` to collect all create/update/delete operations into a single batch call instead of looping
- Removed now-unused `create_block()`, `update_block()`, `delete_block()` methods

**Challenges**: Turso has a known IVM bug with explicit transactions on materialized views with recursive CTEs, but the team is working on a fix upstream. Once fixed, batching will reduce CDC/IVM recalculation from N passes to 1 per file.

**Decision**: Batching at `OrgSyncController` level rather than in the generic `OperationProvider` trait, since only the org sync controller knows it's doing bulk file import. The trait gets a default sequential impl for other use cases.

**Status**: Refactoring complete, all crates compile clean. For 100-block files: 1 transaction + 1 batch event publish instead of 200 individual writes.
