## Fixes landed this session

### 1. `gpui-component` context-menu extension hook

**Problem:** Right-clicking a block showed `gpui_component::Input`'s built-in
Cut/Copy/Paste menu. Our `.context_menu(...)` + `.on_mouse_down` + `.id(...)`
attempts on the outer `div` lost the race against `Input`'s own right-click
listener.

**Fix:** Added `InputState::context_menu_extender(Fn(PopupMenu, &mut Window,
&mut Context<PopupMenu>) -> PopupMenu)` in the fork at
`/Users/martin/Workspaces/rust/gpui-component` branch `holon` (commit
`b8e62667`). Callback runs inside `handle_right_click_menu` after the
built-in items are appended; the returned `PopupMenu` replaces the previous
value so embedders append via `menu.separator().item(...)`.

Holon `Cargo.toml` has a `[patch."https://github.com/holon-space/gpui-component"]`
pointing at the **local path** — **must be pushed + switched to git+branch**
before another checkout can build.

`editor_view.rs` uses the hook to append "Share subtree…" to the built-in
menu (no more competing menus).

### 2. `loro.enabled` default is `false` — surface-level footgun

**Desktop symptom:** `LoroModule::configure` never ran → `Arc<LoroShareBackend>`
never registered → dispatcher reported "No provider registered for entity:
Arc<…LoroShareBackend>". Hidden by `.ok()` in `main.rs`.

**Fix (desktop):** User added `[loro] enabled = true` to
`~/.config/holon/holon.toml`. LoroModule is gated on `loro_enabled()` at
`crates/holon-frontend/src/frontend_module.rs:128`.

**iOS symptom:** same error, but `~/.config/holon/holon.toml` doesn't apply —
iOS builds `HolonConfig` in-code at `frontends/gpui/src/mobile.rs:14`,
`..Default::default()` → `loro.enabled` = `None` → `false`.

**Fix (iOS):** `frontends/gpui/src/mobile.rs` now unconditionally sets
`holon_config.loro.enabled = Some(true)` after the default construction.
Mobile is a first-class sharing target; this is not opt-in material.

**Follow-up worth considering:** flip `loro.enabled`'s default to `true`
globally (or tie it to `orgmode_root.is_some()`) so neither config files
nor frontend-specific overrides are needed.

### 3. `Arc<Arc<LoroShareBackend>>` + `.ok()` swallow in GPUI main

`main.rs:92` did `injector.try_resolve::<Arc<LoroShareBackend>>().ok().map(|a|
(*a).clone())`. Returns `Arc<Arc<LoroShareBackend>>` on success (fluxdi wraps
the provider's `Shared` in another `Arc`). The `.ok()` swallowed fluxdi's
`ServiceNotProvided` error, so the earlier "ShareTrigger global missing"
warning had no root-cause info.

**Fix:** replaced the `.ok()` with an explicit `match` that logs the real
error, and kept the `(*a).clone()` flatten.

**Remaining DI cleanup (task #24):** `DebugServices` in `frontends/mcp/src/server.rs`
has a pure-defensive `OnceLock<Arc<RwLock<LoroDocumentStore>>>` populated in
`frontends/mcp/src/main.rs:420-425` via the same
`try_resolve::<LoroBlockOperations>().ok()` pattern. This is not DI — drop
the `OnceLock`, inject the store as a real constructor dependency, and fail
loud when Loro isn't configured. Also asymmetric: GPUI never populates this
field at all.

### 4. Entity-name normalization bug: `shared_tree` vs `shared-tree` vs `tree`

**Symptom:** dispatcher routed the op to `LoroShareBackend`, but
`execute_operation` rejected it with "expects entity 'shared_tree', got
'shared-tree'".

**Root cause:** `EntityName::new(s)` normalizes `_` → `-` (URI-scheme-safe
form). `SHARED_TREE_ENTITY` const was the raw `"shared_tree"`. Comparison
`entity_name != SHARED_TREE_ENTITY` was always true.

**Fix:** renamed const to `TREE_ENTITY: &str = "tree"` — single bare word,
no `_`, normalization is a no-op. UI callers (`share_ui.rs`) use
`EntityName::new("tree")` which hits the same value unchanged.

**Smell to clean up later:** the `_` → `-` normalization inside
`EntityName::new` is a hidden contract that bit us twice in two days (the
entity-name check here, and likely the source of other surprises). Worth a
dedicated pass to either:
- Require callers to pass the canonical hyphenated form explicitly, OR
- Document every comparison site that must go through normalization.

### 5. `seed_loro_from_persistent_store` ordering

**Root cause** (took three iterations to pinpoint):

`LoroSyncControllerHandle`'s factory calls `seed_loro_from_persistent_store`,
which runs `SELECT … FROM block ORDER BY created_at ASC` and mirrors rows
into Loro. Originally this resolved inside `FrontendSession` factory
**before** `seed_default_layout` AND **before** OrgMode finished loading
user content. Block table had 1 row → seed got 1 row → user's blocks never
made it to Loro → `share_subtree` lookup failed.

**Fix:** moved `try_resolve_async::<LoroSyncControllerHandle>` to **after**
`seed_default_layout()` AND after `signal.wait_ready().await` (the
orgmode-ready signal). Seed now sees 313 rows (full table) and seeds the
~3 blocks not already in Loro (most are already there from other paths).

**Diagnostic scaffolding left in place** — `eprintln!` checkpoints
(`STAGE 1–3f`, `SEED-STAGE 1–3`) and `info!` mirror lines in
`loro_module.rs`, plus `[FrontendSession] factory: …` lines in
`frontend_module.rs`. **These must be removed** before shipping — they
bypass tracing and clutter stderr.

**Mystery still unresolved:** fluxdi's `try_resolve_async::<LoroSyncControllerHandle>()`
returned `Ok` even when the factory body never ran (before the reorder).
The factory body executed AFTER the reorder. The only explanation is that
something cached a placeholder instance, but grep turned up exactly one
`provide::<LoroSyncControllerHandle>` registration and no `store_instance`
calls for this type. Didn't chase to root cause — the reorder made it
moot. **Worth a second look** if you want bulletproof DI semantics.

### 6. `find_tree_id_by_stable_id` prefix mismatch

**Root cause:**

- Loro stores `STABLE_ID` metadata as the bare UUID (`"01806047-…"`), no
  scheme.
- `LoroBackend::resolve_to_tree_id("block:01806047-…")` (used by
  `apply_seed_row`) strips the prefix via `EntityUri::from_raw(id).id()`
  → looks up `"01806047-…"` → finds the node → returns `Some`. Seed skips
  "already in Loro".
- `loro_share_backend::find_tree_id_by_stable_id(doc, "block:01806047-…")`
  (used by `share_subtree`) compared the raw string against `STABLE_ID` →
  never matched → `None` → "block not found in Loro tree" error.

**Fix:** `loro_share_backend::find_tree_id_by_stable_id` now strips `block:`
prefix before the comparison. Both code paths now use the same normalization.

**Better long-term fix:** consolidate these two `find_tree_id_by_stable_id`
implementations into one. Currently both `sync/loro_share_backend.rs:486`
and `api/loro_backend.rs:774` exist — the latter has a cache, the former
doesn't.

## The remaining blocker: #4 (mount-node visibility)

Post-accept evidence from the iOS simulator:

```
Documents/holon-pkm/.loro/
├── device.key                              32 B
├── holon_tree.loro                       4 KB   ← global tree with mount node
├── holon_tree.loro.sync                  12 B
└── shares/
    ├── 86d2c04e-…-b67c8678a3fe.loro      201 KB ← shared subtree content
    └── 86d2c04e-…-b67c8678a3fe.peers.json 251 B ← cached peer addr
```

iOS SQL `block` table: **13 rows** — bootstrap layout only. No shared
content, no mount row. `LIKE '%LogSeq%'` returns zero.

**Why the UI sees nothing:**

`loro_backend.rs:1094 get_all_blocks` does grafting of mount-node children
into the parent hierarchy (`loro_backend.rs:1118-1132` skips mount nodes,
`collect_shared_tree_blocks` grafts their children). **But grep confirms
this function is only called from PBT tests and `memory_backend.rs` — never
from the runtime UI path.** The UI reads from SQL matviews. Mount nodes
never enter SQL. Content is invisible.

**Plan to fix (next session):**

- When `accept_shared_subtree` creates a mount node in Loro, also insert a
  corresponding row into the SQL `block` table:
  - `id` = the mount block's stable id
  - `parent_id` = the parent the user pasted into
  - `content` = name/title of the shared tree (or empty — see #3 for
    `SHARE_ROLE` property)
  - `external_id` = `shared-tree://<shared_tree_id>` (requires #3 —
    `Block.external_id` column)
- When `rehydrate_shared_trees` runs at startup, do the same for every
  persisted mount.
- When the shared doc receives updates over the network, project new/changed
  blocks from the shared doc into SQL with `parent_id` rooted at the mount
  row. Bidirectional: local edits on shared blocks must flow back into the
  shared Loro doc (not the global one).
- Removing a share (the converse op, not implemented yet) must delete the
  mount row + descendants from SQL.

Cascade risks per the original devlog: "may cascade into navigation, command
resolution, etc. — the all-in-LoroTree architecture memory notes that
'mount nodes are invisible to the Block layer' is a load-bearing assumption
in the current code." Pair #3 and #4; budget medium-large.
