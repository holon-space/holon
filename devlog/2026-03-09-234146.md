## PageStack Cursor Panic During IVM Join Operator Commit

Investigated a critical panic in Turso's incremental view maintenance (IVM) system where a btree PageStack cursor enters a cleared state (`current_page=-1`) before attempting an insert operation during materialized view delta processing.

**Context:** The bug manifests when processing a self-join materialized view (`block AS _v0 JOIN block AS _v2`) with high btree page pressure (15+ tables with FK constraints). The panic occurs in `PageStack::current()` when `insert_into_page()` expects a valid page on the stack.

**What happened:**

- Analyzed stack trace spanning `join_operator::commit()` → `persistence::write_row()` → `BtreeCursor::insert()` → `PageStack::current()` panic
- Confirmed reproducer scenario: ~20 tables (with FK constraints) + ~9 materialized views + ~370 row inserts triggers crash during final INSERT when IVM applies view deltas
- Identified that removing graph schema tables collectively (but not individually) prevents crash, indicating tight coupling of btree page cache state
- Reproduced that CDC callback fires ~1100 events before crash, and crash occurs in `vdbe::commit_txn()` → `apply_view_deltas()`, not during INSERT itself
- Mapped the root cause hypothesis: either `move_to_root()` skips `_push()` on error/early return, or cursor is reused across operations without repositioning after `clear()`

**Challenges:**

- **Page cache pressure dependency**: Bug only manifests with high table count; minimal schemas don't trigger it, making local reproduction difficult without full replay file
- **Cursor lifecycle ambiguity**: `clear()` sets `current_page=-1`, but only `_push()` restores it—three possible code paths (line 1161 `move_to_root()`, error handling, cursor reuse) could skip restoration
- **FK constraint interaction**: Multiple index btrees created by 15+ FK constraints create page eviction pressure that corrupts cursor state in ways unrelated to the matview itself

**Decisions:**

- Prioritized reproducing with the minimized 432-statement replay file (reduced from 1189) rather than building from scratch—time-to-bug-trigger justifies this approach
- Focused investigation on `join_operator.rs:757`, `persistence.rs:279`, and btree cursor lifecycle in `btree.rs` lines 1161–1167, 6766–6773 rather than broad code survey

**Status:** Root cause hypothesis refined; next step is to instrument `move_to_root()` and `_push()` call sites to confirm whether cursor repositioning is being skipped, then implement targeted fix in cursor lifecycle management.
