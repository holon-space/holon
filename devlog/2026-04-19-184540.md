## Critique of Android Mobile Bar Plan & Refinement Through Peaceful-Strolling-Muffin Dependency

**Context:**
A code review of the Android mobile operation bar plan revealed 22 implicit hypotheses and a fundamental misalignment with the broader architecture direction (peaceful-strolling-muffin). The plan required substantial rework.

**What happened:**

- Performed senior-dev critique of the mobile-bar plan, validating 22 explicit hypotheses against in-tree code (IME FFI, block op dispatch, dedupe semantics, signal types, manifest behavior).
- Discovered critical blockers: 60% of block operations can't dispatch via `for_row()` (missing params), `ui_generation` bumps on IME cause cursor multiplication (anti-pattern), `keep-innermost` dedupe renders empty outer levels.
- Cross-referenced peaceful-strolling-muffin.md and found **contradictory design directions**: the mobile-bar plan pursued a specialized mega-widget; peaceful-strolling-muffin explicitly rejects that in favor of composable building blocks.
- Resolved the conflict by **making mobile-bar depend on peaceful-strolling-muffin**, shrinking the mobile-bar scope from ~1000 LOC (new ReactiveEngine API, shadow builder, DSL widget) to ~300 LOC (two primitives + YAML composition).
- Validated five orthogonal hypotheses against current code: parent-mutation re-emission (safe), `ensure_watching` idempotency (verified), transient missing-parent handling (debounce spec added), test seeding patterns (existing), GPUI a11y surface (minimal but present).

**Challenges:**

- **Plan contradiction detection**: Peaceful-strolling-muffin v3.1 lists `focus_chain_bar` as an explicitly rejected alternative, but the mobile-bar plan was built on it. Resolved by deferring to the newer, explicitly-intentional plan.
- **Hypothesis validation at architectural boundaries**: Some concerns (parent-mutation invalidation, ancestor-watching coverage) live at the intersection of reactive core and the forthcoming value-fn layer. Pinned concerns as cross-references between plans to prevent loss at handoff.

**Decisions:**

- **Depend on peaceful-strolling-muffin first**: Rather than ship an interim specialized widget, wait for the value-fn / streaming-collection primitives and compose the bar in YAML (`chain_ops(0)` → `op_button`). Aligns with stated intent and avoids churn.
- **Minimize the immediate PR**: Add `bottom_dock(main, dock)` container + `op_button(op_name)` tappable primitive + manifest flip. No new ReactiveEngine API.
- **Interim dedupe via index**: Use `chain_ops(0)` (innermost level only) until URI scheme differentiation ships; cross-chain grouping is follow-up #2.

**Status:**
Mobile-bar plan rewritten as an Option-1 composition layer atop peaceful-strolling-muffin (not yet shipped). Five orthogonal hypotheses validated against current code. Ready to implement once peaceful-strolling-muffin §9 lands.
