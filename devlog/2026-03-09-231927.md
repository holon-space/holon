## Jujutsu Rebase Skill Documentation Review

**Context**: Reviewed and refined the comprehensive jj-rebase skill documentation covering isolated branch rebasing, conflict resolution, and result transfer for a multi-author jujutsu repository.

**What happened**

- Reviewed the complete `SKILL.md` specification documenting the full jj-rebase workflow (12 major steps plus edge cases)
- Verified critical revset patterns for finding user branches (`roots(author(...) ~ ::main)`) and handling divergent commits
- Confirmed bundled script responsibilities: clone isolation, auto-resolve via mergiraf, divergent cleanup, incremental rebase (`jj-imerge2.sh`), and finish-phase transfer
- Validated the workflow's handling of pre-existing conflicts (step 3b) with explicit `jj restore` strategy to avoid cascading re-conflicts
- Reviewed decision points for rebase strategy: bulk rebase for <20 merge commits behind, incremental for >80 (based on first-parent count, not total DAG size)
- Examined edge case handling: immutable divergent copies, mega-merge topology preservation, empty-description commits blocking push, generated file restoration
- Confirmed validation step (9) uses pre/post commit IDs to detect code loss via cross-diff comparison

**Challenges**

- **Divergent commit identification complexity**: The `/0` vs `/1` offset is unstable — only `immutable` property and `parents()` provide reliable discrimination. Solution: explicit log template checking parent change_id against most-recent main.
- **Conflict cascading vs. pre-existing**: Rebase-induced conflicts cascade (resolve root = descendants clean), but pre-existing conflicts are snapshot per-commit. Solution: explicit `jj restore` from resolved ancestor when descendant doesn't modify the file, else manual re-resolution.
- **Clone working copy pitfall**: `@` inherits user's author and appears as spurious root. Solution: abandon in step 2 (twice, since jj creates replacement).
- **Bulk vs. incremental rebase decision**: Total DAG commit count is misleading (370 commits might be only 78 merge commits). Solution: use first-parent count only.

**Decisions**

- **Prefer `jj-imerge2.sh` over legacy `jj-imerge.sh`**: Uses `-s` (safer) vs `-b`, macOS bash 3.2+ compatible, auto-resolve via mergiraf with `--no-auto-resolve` flag to allow manual intervention
- **Always use `-s` (source), never `-b` (branch)**: `-b` is dangerous; `-s` rebases all descendants correctly with merge topology preserved
- **Use commit ID directly for immutable divergent copies**: Cannot be abandoned; exclude from `-s` list and pass the mutable copy's commit ID to avoid collision

**Status**: Skill documentation is comprehensive and ready for implementation. All 12 workflow steps, 4 bundled scripts, critical revsets, and 9 edge cases are fully specified. No script implementation work done in this session — documentation review only.
