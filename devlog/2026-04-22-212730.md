## Fixed Widget Registration Panic in PBT via DI-Driven Interpreter Injection

The property-based test was panicking during strategy generation because `parse_render_dsl()` required widget names from a global `OnceLock` that hadn't been initialized yet.

**What happened:**

- Identified the root cause: `parse_render_dsl()` at `state_machine.rs:1773` in the reference model's `StartApp` transition was hitting the panic at `render_dsl.rs:57` (`registered_widget_names() must be called before any render DSL parsing`). This happened during proptest generation, before `start_app()` had a chance to register names.
- Refactored `render_dsl.rs` to separate concerns: extracted `parse_render_dsl_with_engine()` (core parsing logic), added `parse_render_dsl_with_names()` (explicit names variant), and exposed `parse_dsl()` on `RenderInterpreter` itself.
- Added `Debug` impl to `RenderInterpreter` and a `dsl_names()` method that returns builder + value_fn keys.
- Replaced the `REFERENCE_INTERPRETER` static in `reference_state.rs` with an `interpreter: Arc<ShadowInterpreter>` field on `ReferenceState`, injected via constructor.
- Updated `state_machine.rs::init_state()` to resolve the interpreter from a FluxDI `Injector` and pass it to `ReferenceState::new()`. The `StartApp` transition now calls `state.interpreter.parse_dsl()` instead of the global function.

**Decisions:**

- **DI-driven approach**: Rather than lazily initializing a global, the interpreter is now resolved through the same FluxDI pipeline as production code, making the flow explicit and testable.
- **Single source of truth**: Widget names come from the `RenderInterpreter`'s builders/value_fns, not a duplicate registration list.

**Status:** Widget registration panic eliminated. PBT now progresses past initialization; remaining failures are pre-existing (`inv16 CDC quiescence` issue).
