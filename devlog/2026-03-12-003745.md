## Refactoring Dataflow Architecture: Eliminating RenderContext Spaghetti

**Context:** The codebase has accumulated parallel layering (ResolvedRow, RenderContext, BlockRenderCache) that duplicates the responsibility of the cleaner LiveData → ProfileResolver → ViewModel pipeline. This session aimed to consolidate to a single clean dataflow path.

**What happened:**

- Analyzed current architecture: ResolvedRow/RenderContext/BlockRenderCache act as intermediate carriers and god-objects; they should be replaced by LiveData-based per-node streams
- Identified split of RenderContext: infrastructure (session, ext) captured once; traversal state (depth, query_depth) stays internal; everything else moves to ViewModel or eliminated
- Planned 4-step migration to execute in single session:
  1. **Eliminate ResolvedRow** — inline profile resolution into ViewModel construction; removes intermediate carrier
  2. **Extract InterpretationState from RenderContext** — split god-object; keep traversal state internal, expose only session/infrastructure
  3. **Replace BlockRenderCache with per-node LiveData** — BlockRef nodes own their own `LiveData<ViewModel>`; CDC application becomes declarative
  4. **Push LiveData to frontends** — GPUI/Blinc/Ply subscribe to ViewModel streams instead of receiving fully-built trees
- Evaluated FRP libraries: `futures-signals` with `MapDiff` supports key-based CDC (matching Turso's structure); `tokio` channels acceptable as fallback
- Mapped critical touchpoints: InterpretationPipeline → builders (shadow, row, block_ref) → ViewModel serialization → frontend dispatch via builder_registry
- Designed incremental validation: each step produces compilable state; frontends unchanged until step 4

**Challenges:**

- **BlockRef recursion**: query_depth prevents infinite chains — must stay as internal counter in InterpretationState, enforced when spawning child LiveData
- **Widget state threading**: collapsed/expanded, column widths currently on RenderContext — converted to input signal; changes trigger re-interpretation of affected subtree
- **CDC at ViewModel level**: need incremental re-interpretation (only recompute when structure changes) not full tree rebuild — deferred to step 3

**Decisions:**

- Use `futures-signals` MapDiff for key-based CDC rather than Vec-based patching — aligns with Turso's structure
- Depth computation stored on ViewModel rather than computed at render-time — enables early filtering/truncation
- Keep InterpretationState as internal-only struct — never exposed to frontends, preventing future RenderContext re-accumulation

**Status:** Plan validated and ready for execution. ResolvedRow elimination (step 1) can begin immediately; subsequent steps unblock in sequence.
