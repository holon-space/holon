## Comprehensive Code Review & Plan Refinement for Responsive Layout Feature

### Context
A senior dev reviewed the responsive-layout (container-query) plan and identified ~10 critical gaps: streaming column count unknowability, physical DPI assumptions, profile resolution machinery, render context leakage, resize-rebuild storms, and untested core assumptions. The plan required substantial rearchitecture, not just bug fixes.

### What happened

- **Read and critiqued the original plan** against actual codebase (verified `reactive_view.rs:34`, `entity_profile.rs:982`, `row_signal_vec()`, profile priority sorting at `:362`). Identified 10 classes of issues ranging from architectural (H1–H3 hypotheses) to footguns (combined predicates, headless test invisibility).

- **Validated quick claims via grep/source inspection** (8 confirmations, 2 falsifications). Most critical: `row_render_context` does drop parent context on purpose; profile resolution uses URI scheme, not block-id or properties.

- **Reframed the streaming-columns count problem** from "unsolvable" to "partition at `VecDiff::Replace` time, force subtree re-interpret on incremental diffs." Hypothesis H1 (initial render gets Replace; steady-state gets incremental) falsified by existing `row_signal_vec()` docs, but Replace-time partitioning works for v1.

- **Escalated physical DPI investigation** from "quick lookup" to "real work": T1 now means patching GPUI in-tree if `scale_factor()` isn't exposed. Dropped the 38 px/cm fiction and pivoted to cm as the primary API unit.

- **Rearchitected around reactive `available_space`** (user feedback on point 6): instead of gating variant selection on breakpoint boundaries, make `available_space` a `Mutable<Option<AvailableSpace>>` signal per `ReactiveView`. Eliminates rebuild storms (R8), count staleness (R2), and simplifies to "signal graph does the work." Estimated ~590 LoC net (new `reactive_signal_full`, side-table management); kills 4 major risks.

- **Fixed root-layout profile resolution** (user feedback on point 4): use a data-condition `id == "<root_layout_uri>"` on the generic block profile, no new URI scheme or per-block machinery.

- **Demoted phone variant hardcoding** (user feedback on point 5): variants are now *different `RenderExpr`s*, filtering the same `focus_roots` query via `if_col` list-match. Removed `block_ref("main")`.

### Challenges

- **H1 streaming partition hypothesis falsified**: Incremental `InsertAt`/`RemoveAt` diffs occur at steady-state, not just Replace. **Solution**: Partition only at Replace time; on incremental diffs, flag the entire collection-local subtree dirty and let the existing re-interpret path handle it (cheaper than full tree rebuild).

- **Profile resolution has no per-block-id override**: grep confirmed no existing mechanism. **Solution**: Use a data-condition `id == "<literal>"` keyed to the root layout block's URI, matching at profile lookup time in `entity_profile.rs:982`.
