## Making GPUI Panels Scrollable with Per-Block Scroll State

The GPUI columns/panels were clipping content with `.overflow_hidden()` instead of allowing scrolling. The fix required threading block IDs through the layout partition so each scrollable panel could maintain independent scroll state.

**What happened:**

- Identified that GPUI's `.overflow_y_scroll()` requires `.id()` for scroll state persistence, but builders must return `Div` not `Stateful<Div>`
- Solution: wrap panels in scroll containers at assembly time (in `build_screen_layout`), not in builders, using `.child(div.id().overflow_y_scroll())`
- Created `MainRegion<W>` struct in `ScreenLayoutPartition` to carry block IDs for main content (sidebars already had `CollapsibleRegion` with block_id)
- Updated `partition_screen_columns()` to preserve block IDs for all regions, not just sidebars
- Modified GPUI columns builder to use per-region `ElementId`s from block IDs for scroll state tracking
- Fixed 5 other frontends (blinc, ply, waterui, dioxus, holon-frontend) consuming `partition.main` to access `.widget` on the new `MainRegion` wrapper

**Challenges:**

- `.id()` changes `Div` → `Stateful<Div>`, breaking return types — solved by wrapping at the child-addition point, not in builder functions
- Main content regions were losing block identity during partition — solved by making `main` carry `MainRegion` structs like sidebars already did
- Multiple frontends depended on `partition.main` being a simple `Vec<W>` — had to update all consumers, not just GPUI

**Status:** All panels now scroll independently with persisted scroll state per block. GPUI and all downstream frontends compile and tests pass.
