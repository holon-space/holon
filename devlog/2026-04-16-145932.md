## Code Review & Plan Refinement: Two-Phase Layout for Positional Children

Conducted a senior-dev code review of the "enchanted-bouncing-panda" two-phase layout plan, treating it as hostile critique to surface blind spots.

**What happened:**
- Read the plan and codebase to validate assumptions about `StaticCollection`, `LayoutHint`, reactive space propagation, viewport resize behavior, and drawer state management
- Discovered 6 significant issues and 2 critical gaps: (1) GPUI columns renderer never reads `layout_hint` to set flex properties, undoing the shadow layer's partition; (2) viewport resize freezes positional children because `StaticCollection` has no `space_mutable()`, making unequal partitions break visibly on window resize
- Validated that drawer toggle works (shadow re-interprets on toggle) and that `widget_state.width` fallback is dead code today (`Option::None`)
- Tested spacer macro: generates `..Default::default()` which gives `Flex { weight: 1.0 }` (wrong), requiring conversion to raw fn instead of `.with_layout_hint()` chaining
- Honestly compared Two-Phase vs. Mutable Per-Child approaches: Mutable wins on every product quality axis (resize correctness, performance, simplicity) but requires new `PartitionedStatic` collection variant (~300-400 LOC, not trivial but doable)

**Challenges:**
- **GPUI-shadow layer coupling**: Shadow partition is invisible to GPUI renderer. Solution: add Step 7 to set `flex: 0 0 Xpx` on Fixed children, `flex: N` on Flex children in the GPUI columns layout function
- **Resize limitation**: Acknowledged as pre-existing bug, documented in Known Limitations (L1). Mutable approach is proper fix; two-phase is stepping stone only
- **Drawer state in shadow layer**: Redesigned to avoid coupling shadow (pure) to `widget_state` (mutable). Choose Option A: drawer builder always outputs Fixed `layout_hint`, animation handled in GPUI renderer

**Decisions:**
- Kept two-phase as valid stepping stone because `LayoutHint` foundation is reusable, but marked the phase 1/2 logic in columns.rs as throwaway
- Documented viewport resize as Known Limitation L1 (accept or migrate to Mutable)
- Added 4 manual validation checkpoints (phase 1/2 correctness, nested columns, drawer toggle, all-Fixed case)

**Status:** Plan updated with all findings, concrete code for spacer conversion, honest Mutable trade-off analysis, and 5 Known Limitations explicitly documented. Ready for implementation with eyes wide open on both the approach's benefits and its size constraints.
