# Subtree Sharing — mount-row SQL projection landed

Continuation of `devlog/2026-04-20-103900.md`. That handoff's #1 priority —
"Implement #4 + #3 together (mount visibility + external_id)" — is now

which is what the UI reads. Descendants of the shared tree still don't

## What landed this session

### 1. `SHARE_ROLE` + `SHARED_TREE_ID` as Block properties (V1 storage)

New constants in `crates/holon/src/sync/shared_tree.rs`:

- `SHARE_ROLE_PROPERTY = "share-role"` — key in `block.properties`
- `SHARE_ROLE_MOUNT = "mount"` — value for mount rows
- `SHARED_TREE_ID_PROPERTY = "shared-tree-id"` — mirror of
  `MOUNT_SHARED_TREE_ID` in Loro metadata

**Why properties instead of a new column (as the original devlog asked):**

Adding `Block.external_id: Option<String>` to the struct was the original
plan but touches ~12 struct-literal `Block { .. }` initialisers across the
workspace AND requires a SQL schema change + migration. V1 uses
`properties` (which round-trip through SQL JSON + org `:PROPERTIES:`
drawers for free per the recent CacheEventSubscriber fix) to keep the
blast radius small. Queries that need to find share rows:

```sql
SELECT id, properties FROM block
  WHERE json_extract(properties, '$."share-role"') = 'mount';
```

Follow-up #7 tracks the column migration + Todoist `external_id`
unification from `LoroBackend::set_external_id` / `get_external_id`.

### 2. Mount row projection — accept path

`LoroShareBackend` now has an optional `sql_ops: Option<Arc<SqlOperationProvider>>`
field. DI wires it up in `register_subtree_share` using the same
`DbHandle` + `TursoEventBus` that `LoroSyncControllerHandle` uses.

New helper `project_mount_to_sql(mount_uri, parent_uri, shared_tree_id,
title)` builds a `StorageEntity`, calls `sql_ops.execute_operation("block",
"create", params)`. The create op:

- INSERT OR IGNOREs into `block`
- Publishes `EventKind::Created` — `CacheEventSubscriber` picks it up,
  `QueryableCache<Block>` refreshes, UI sees the row

Called from `accept_shared_subtree` AFTER `save_all()` flushes the global
Loro doc — so Loro durability is strictly before SQL visibility. If SQL
insert fails, the mount exists in Loro; the next
`rehydrate_shared_trees` repair path catches it.

For backwards compatibility, `LoroShareBackend::new(...)` stays as-is
(no `sql_ops` parameter) — tests that don't need UI visibility pass
`None` via the delegation to `new_with_sql(...)`. DI-wired production
code uses `new_with_sql(..., Some(sql_ops))`.

### 3. Mount row re-projection — rehydrate path

`rehydrate_shared_trees` now collects `MountRehydrationRecord` instead of
bare `MountInfo` — each record additionally carries
`mount_stable_id` and `parent_stable_id` read from Loro tree meta during
enumeration. Per-record loop calls `project_mount_to_sql` with the same
`INSERT OR IGNORE` semantics, so rehydrate is idempotent across
restarts:

- Normal case: row already exists, INSERT OR IGNORE is a no-op
- Repair case: SQL was wiped but Loro survived (or vice versa) →
  rehydrate recreates the row

Missing `mount_stable_id` or `parent_stable_id` logs a `warn!` and
skips; a mount node without either is already broken, projecting it
would add no information.

### 4. STABLE_ID prefix bug fix

`loro_share_backend.rs::set_stable_id` used to do
`meta.insert(STABLE_ID, stable_id)` verbatim — but every caller passed
`format!("block:{}", Uuid::new_v4())`, so the metadata stored the
prefixed form while the rest of the stack (`find_tree_id_by_stable_id`,
`resolve_to_tree_id`, `set_external_id`) all store/read the bare UUID.

Before mount-row projection nothing triggered the mismatch — mount
nodes weren't looked up by stable_id after creation. The new flow
broke the invariant: the SQL `Created` event reaches
`LoroSyncController::apply_create`, which resolves `block:<uuid>` to
bare UUID, looks up Loro, would have found no match (prefixed STABLE_ID
vs bare lookup), and tried to CREATE A DUPLICATE Loro node.

Fix: `set_stable_id` strips `block:` before insert. Both share_subtree
and accept_shared_subtree paths now store bare UUIDs, matching every
reader. All 13 `loro_share_backend::tests` still pass; all 3
`api::sync_pbt::tests` (including `subtree_share_round_trip_pbt` and
`test_multi_peer_sync_iroh`) still pass.

### 5. Notes.org asset file

The uncommitted changes had `include_str!("../../../assets/default/notes.org")`
in `frontend_module.rs` but the file didn't exist — workspace builds were
broken. Created `assets/default/notes.org` with `* Notes\n` so both

## Not done — explicit handoff

### Follow-up #5 (high priority): descendant projection

The mount row alone tells the user "a shared tree is here" — it doesn't
show the shared content. Remaining sub-items:

1. **Static projection on accept**: after the mount row is written,
   walk `shared_arc.get_tree(TREE_NAME)` and for each node:
   - Read `STABLE_ID`, `CONTENT_RAW`, `CONTENT_TYPE`, `SOURCE_LANGUAGE`
     from meta
   - Build a Block URI and parent URI (mount for root, parent block
     for others)
   - Call `sql_ops.execute_operation("block", "create", ...)`
2. **Static projection on rehydrate**: same walk, same inserts (with
   `INSERT OR IGNORE` idempotency)
3. **Live updates from network**: subscribe to `shared_arc.subscribe_root`,
   translate diffs into SQL UPDATE/INSERT/DELETE ops
4. **Bidirectional**: SQL writes to blocks whose `shared-tree-id`
   property is set must flow into the corresponding `shared_arc`, not
   the global doc. Currently all block ops go through
   `SqlOperationProvider` → global doc via `LoroSyncController`. Needs
   a routing layer keyed on `shared-tree-id` property.

Budget: medium-large. (3) and (4) together probably need a new
"shared tree sync controller" that mirrors `LoroSyncController` but
scoped per shared doc.

### Follow-up #7 (low priority): `Block.external_id` column

Original devlog's "pair #3 with #4" request. V1 storage used
`properties`. Migrating to a column needs:

- Add `external_id: Option<String>` field to `Block`, `#[serde(default)]`
- Add `external_id TEXT` column + index to `crates/holon/sql/schema/blocks.sql`
- Add `"external_id"` to `BLOCKS_KNOWN_COLUMNS` in
  `sql_operation_provider.rs`
- Update ~12 struct-literal `Block { .. }` initialisers across the
  workspace
- One-shot migration: scan Loro tree meta for `external_id`
  entries (Todoist etc.), copy into Block column
- Deprecate `LoroBackend::set_external_id` / `get_external_id` once
  reads switch to the column

Budget: medium. Mostly mechanical.
