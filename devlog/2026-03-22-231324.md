## Widget Gallery UI Component Showcase

Building a development-focused UI component gallery to quickly visualize all available widgets in different states and configurations, following the existing Settings overlay pattern.

**What happened**

- Created `crates/holon-frontend/src/widget_gallery.rs` with 6 showcase sections: Text & Labels, Inputs, Layout, Collections, Data Display, and Feedback
- Exposed `widget_gallery_render_data()` method on `FrontendSession` following the same `RenderExpr` builder pattern as preferences
- Extracted shared `modal_overlay()` and `interpret_and_render()` helpers in GPUI to eliminate duplication between settings and gallery overlays
- Refactored existing settings overlay to use the new helper
- Added `show_widget_gallery` flag and 🎨 gallery button in the GPUI title bar (alongside the settings gear)
- Progressively expanded widget coverage: started with 4 basic widgets, then added **columns** (multi-column layout), **list** (generic lists with templates), **tree** & **outline** (hierarchical data with parent_id tracking), **error** (error display), and **pref_field** (preference fields in choice/toggle variants)

**Decisions**

- Used the existing `ViewModel`/`RenderExpr` architecture rather than creating a separate gallery system, ensuring all 3 frontends (Flutter, GPUI, TUI) can theoretically display it
- Extracted the modal overlay pattern to reduce code duplication and provide a reusable template for future modal dialogs
- Included only static, self-contained widgets in the gallery—excluded live-data widgets (`block_ref`, `live_query`) and context-dependent widgets (`pie_menu`, `block_operations`) since they require external state

**Status**

Widget gallery fully implemented with 6 sections showcasing 18+ widgets across Text, Inputs, Layout, Collections, Data Display, and Feedback categories. All tests pass; holon-frontend and holon-gpui compile without errors. Graceful fallback behavior confirmed: unsupported widgets render as `[unknown: widget_name]` rather than panicking.
