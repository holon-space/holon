## IVM Rollback State Recovery

Recursive CTE matviews were retaining in-memory DBSP operator state after transaction rollback, causing subsequent transactions to produce wrong deltas. The pager rolled back btree pages correctly, but the `RecursiveOperator` state (`seen_rows`, `next_rowid`, etc.) was never reset.

**What happened:**

- Identified that `RecursiveOperator` holds mutable state across transactions that must be cleared on rollback
- Added `reset_for_rollback()` method to `DbspCircuit` (calls `reset_for_new_transaction()` on each recursive operator)
- Wired reset into ROLLBACK paths: `AutoCommit::rollback_tx()` in `execute.rs` and `rollback_current_txn()` in `mod.rs`
- Also fixed `rollback_current_txn()` to explicitly clear `view_transaction_states` (previously only done via `apply_view_deltas(rollback=true)`)
- Created new `.sqltest` file with recursive CTE matview rollback scenarios (counter-style recursion, UNION ALL mode, multi-iteration state)
- Verified fix: existing 9 rollback tests pass, new recursive CTE rollback tests pass

**Decisions:**

- Used `reset_for_new_transaction()` approach over snapshot/restore — simpler, reuses existing code, next `execute()` rebuild state from btree (which is now rolled-back)
- Added the reset call after `pager.rollback_tx()` so btree pages are reverted before operator state is cleared, enabling correct rebuild

**Status:** Fix implemented and tested. All rollback tests (including new recursive CTE matview cases) pass. Ready for integration.
