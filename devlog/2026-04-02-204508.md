## Fixed Merge Conflicts in Multi-Branch Integration

Multiple feature branches were merged into a jj revision, creating dependency conflicts and unresolved pattern matches across the codebase.

### What happened

- **pyo3 version mismatch**: `workspace-hack/Cargo.toml` pinned pyo3 0.27 while `bindings/python` upgraded to 0.28.2. Updated all pyo3/pyo3-ffi/pyo3-build-config references in workspace-hack to 0.28.x.

- **Missing `Table::RecursiveCte` pattern coverage**: One branch added the `RecursiveCte` variant to the `Table` enum; match expressions in `core/incremental/view.rs:257` and `core/schema.rs:2132` didn't cover it. Added arms returning `None` (like `FromClauseSubquery`) and `Identity` layout respectively.

- **`Drop` struct shadows Rust trait**: A branch introduced `sql_generation::model::query::Drop`, shadowing the built-in `Drop` trait. Renamed throughout to `DropTable` (11 files: drop.rs, mod.rs, query.rs, simulator model/generation files, profiles).

- **Cascading missing imports**: Simulator crate files (property.rs, plan.rs, metrics.rs, interactions.rs, etc.) lost imports after module restructuring. Added explicit imports for `Query`, `QueryCapabilities`, `ResultSet`, `SimulatorEnv`, query type variants, and runtime types across 7+ files.

- **Field rename not fully propagated**: One branch renamed `experimental_mvcc` → `mvcc` in `Profile`. Updated remaining references in runner/env.rs, runner/execution.rs, and profiles/mod.rs.

- **Stale mergiraf conflict**: Reran `jj resolve --tool mergiraf` on plan.rs, which cleaned up residual merge markers.

### Decisions

- Renamed `Drop` → `DropTable` rather than re-exporting to avoid trait shadowing entirely — cleaner than scoping workarounds and catches future similar issues at the import site.
- Added fine-grained imports instead of glob imports to maintain clarity and catch missing types at compile time.

**Status**: Full workspace builds successfully (zero errors, 8 warnings from unrelated dead code).
