## Fixed tokio runtime mismatch in GPUI frontend

The production logs revealed a panic in the GPUI frontend that was initially suspected to be a Turso issue, but turned out to be a runtime incompatibility.

**What happened**

- Inspected `/tmp/holon-gpui-turso-bug.log` and extracted the backtrace from interleaved SQL trace output
- Located panic at `frontends/gpui/src/lib.rs:175`: "there is no reactor running, must be called from the context of a Tokio 1.x runtime"
- Root cause: code was calling `tokio::time::sleep()` inside a GPUI async task spawned with `cx.spawn()`
- GPUI uses `smol`-based async executor, not tokio, so tokio-specific APIs fail in that context
- Verified no other `tokio::time` usages elsewhere in the GPUI frontend
- Added `smol = "2"` as explicit dependency (was only transitively available through `gpui_linux`)
- Replaced `tokio::time::sleep(Duration::from_secs(2)).await` with `smol::Timer::after(Duration::from_secs(2)).await`
- Verified clean compilation

**Decisions**

- Used `smol::Timer` instead of `futures_timer::Delay` since smol is already in the dependency tree and is the executor backing GPUI
- Added smol as explicit dependency rather than relying on transitive dependency through linux-only features

**Status**: Fixed and verified to compile cleanly. Ready for testing against production logs.
