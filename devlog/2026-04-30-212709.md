## Turso Recursive-CTE UPDATE CDC Fix Verified

Upstream Turso shipped Option B (DELETE-then-INSERT ordering) for recursive matview UPDATEs. The holon mitigation and end-to-end flow now work seamlessly.

**Context**

The split-block staleness issue traced back to Turso's `WITH RECURSIVE` matviews silently dropping UPDATE events. A pure-Turso reproducer was created to track the fix, and a holon-side content-aware coalesce mitigation was landed as a forward-compatible safety net.

**What happened**

- Rebuilt with latest Turso; ran the standalone reproducer (`turso_ivm_recursive_update_cdc_pair`)
- Turso now emits `Delete(old)` then `Insert(new)` for recursive matview UPDATEs — Debezium-style ordering
- Holon's existing `coalesce_row_changes` DELETE-then-INSERT branch already handles this, folding to `Updated`
- All 6 `turso_ivm_split_block_cdc_drop_repro` tests pass (was 5/6 before mitigation)
- All 76 storage::turso tests pass; all 16 `cdc_coalescer_tests` pass including the 6 new content-aware collapse tests
- The new `collapse_insert_delete_pairs` mitigation is dormant on the Turso fix (correct behavior flows through existing paths) but remains load-bearing should Turso order change or for robustness against future matview CDCs

**Decisions**

- Kept the holon-side content-aware mitigation live rather than removing it. It's forward-compatible: in (A) single-Update emission or (B) DELETE-then-INSERT order, the new branch becomes dead code but safe. Cleanup can be deferred until the upstream fix has aged in production.

**Status**

TUI split-block staleness is now unblocked; matview UPDATE events flow end-to-end correctly. The upstream fix is verified and the holon mitigation is in place. Ready to move to collaborative-editing integration into TUI edit-mode.
