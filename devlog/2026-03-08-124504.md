## Extracting Display Tree from Rendering Pipeline

Separating extraction from assertions and handling collection-valued elements with a proper data structure.

**Context**: The current `UiElement` approach only stores a single `HashMap` per element, can't represent nested structures, and conflates data extraction with assertion logic. We need a reverse-rendered display tree that mirrors the actual render tree structure.

**What happened:**

- Analyzed `RenderInterpreter<W, Ext>` — it's already generic over widget type `W` (Blinc/GPUI use `Div`, we can use `DisplayNode`)
- Discovered we can create a shadow interpreter that uses identical builder dispatch but produces a diffable data structure instead of pixels
- Designed `DisplayNode` enum to represent the reverse-engineered render tree: `Collection` (iterated data rows), `Layout` (structural containers), `Element` (one row + sub-widgets), and `Leaf` (resolved values)
- Proposed `RenderInterpreter<DisplayNode>` paired with a new `DisplayBuilder` trait — same signature as `WidgetBuilder<W>` but returns `DisplayNode`
- Each frontend builder (list, tree, table, columns, etc.) gets a corresponding `DisplayBuilder::list()`, `DisplayBuilder::tree()` implementation that just re-wraps its children in `Collection { items, .. }`
- Wiring: in test builds, `RenderContext.ext` carries a `DisplayBuilder` instance; builders call both the pixel-builder and display-builder in parallel during rendering
- `UiSnapshot` becomes a simple container: `root: DisplayNode`, methods like `.flatten()` (depth-first traversal), `.at_path()` (navigate by breadcrumbs), `.data_at_depth()` (extract all `Element` nodes at a specific nesting level)
- Assertions become external: tree-diff assertions, ordered-subset checks, field comparisons — all written as functions that operate on `DisplayNode`, independent of extraction

**Decisions:**

- Keep extraction (rendering → `DisplayNode`) and assertion (comparing `DisplayNode`s) completely separate — `DisplayNode` is just a data structure with no assertion methods
- Reuse `RenderInterpreter` dispatch for display building — no duplicate builder registration logic, same `RenderExpr` flows through both paths
- `Element.data` stores the full row `HashMap` — for collection-valued fields, child `DisplayNode`s are in `children`, data row is preserved for field comparison

**Status**: Design complete; ready to implement shadow `RenderInterpreter<DisplayNode>`, `DisplayBuilder` trait registration in Blinc, and `UiSnapshot` container with tree-diff tooling.
