## Optimized `cargo watch` startup and fixed premature app restarts

The `just watch gpui` recipe had two issues: 1) a 60+ second startup delay before compilation began, and 2) the app was killed on every file change regardless of build success.

**What happened**

- Diagnosed the startup delay by running `cargo watch` with debug output (`WATCHEXEC_LOG=debug`) and found it was scanning the entire repo root, including `vendor/zed/` (a full Zed repo checkout with 3.6k files)
- Root cause: cargo-watch defaults to watching `.` which requires traversing the file tree to discover `.gitignore` files — a slow operation in a monorepo with large vendored directories
- Fixed by adding explicit watch directories with `-w crates -w frontends/gpui`, reducing startup from 60+ seconds to instant
- Discovered the app-restart issue: `cargo watch -s` kills the *entire process group* on file change, so the "run detached" trick didn't work
- Refactored the recipe to separate concerns: cargo-watch only builds, an outer shell script manages the app lifecycle via signals (`USR1`)
- On successful build, cargo-watch sends `USR1` to the outer script, which kills the old app and starts the new one
- On build failure, no signal is sent, old app continues running uninterrupted

**Challenges**

- **Process group inheritance** — subshells still inherit the parent's process group, so cargo-watch could still kill the child. Solved by moving app management outside cargo-watch entirely.
- **Signal coordination** — needed a way for cargo-watch (which doesn't know about app PIDs) to tell the outer script "restart now". Used SIGUSR1 as a clean, reserved signal.

**Status**

`just watch gpui` now starts immediately and only restarts the app after successful compilation. Failed builds leave the running app untouched.
