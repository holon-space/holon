## Navigation Click Not Updating Main Panel View

The left sidebar doc-block click wasn't triggering a main panel content update, despite backend processing the navigation change correctly.

- Analyzed logs showing `focus_roots` CDC firing with 2 changes after click, but downstream `watch_view_*` matview showed no CDC events
- Verified Turso IVM chained matview CDC bug was actually fixed (unit tests pass), ruling out that hypothesis
- Traced through production database state: matview had 272 stale rows vs 6 fresh rows from current focus, confirming staleness
- Wrote minimal Turso reproducer matching production schema (recursive CTE matview, exact SQL) — passed in isolation
- Scaled test to production numbers (131 rows/doc, 10 matviews) — still passed, eliminating Turso as culprit
- Checked CDC pipeline: broadcast channel not dropping messages, callbacks properly registered
- **Found root cause**: When app restarts, Turso auto-loads DBSP graph for persisted matviews. Then `NavigationSchemaModule` runs `DROP VIEW IF EXISTS current_focus` and `CREATE MATERIALIZED VIEW current_focus`. This DROP + CREATE destroys the DBSP dependency chain for auto-loaded downstream matviews like `watch_view`. The upstream matview (current_focus/focus_roots) gets re-registered in DBSP, but downstream loses its connection.

**Challenges**
- Initial misdiagnosis as Turso IVM bug (found outdated internal documentation claiming it unsolved)
- Test isolation masked the bug — manually re-creating DDL in session 2 reconstructed the graph
- Reproducer needed to NOT re-send DDL on reopen to expose the actual issue

**Decisions**
- Turso bug confirmed and isolated in minimal reproducer (no Holon APIs)
- Fix belongs in Turso: either preserve auto-loaded DBSP entries through DDL operations, or require explicit re-registration

**Status**: Root cause identified and reproduced in minimal Turso test case; ready to report upstream.
