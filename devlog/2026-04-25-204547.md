## Wiring inv10h_live to Main Panel and Fixing ToggleState Generator

The property-based test framework for detecting live-tree divergence was ready to run, but the integration harness wasn't properly wired and the state machine generator was creating no-op transitions that prevented CDC propagation.

**What happened**

- Re-pointed `inv10h_live` in `crates/holon-integration-tests/src/pbt/sut.rs` from the root layout block to `block:default-main-panel`, since the root's `data_rows` are always empty and never trigger `HeadlessLiveTree` initialization
- Updated the framework to call `reactive.ensure_watching()` on the main panel, drain its watch stream until rows arrive, then initialize the live tree with actual panel data and extract `item_template` from the `view_mode_switcher` render expression
- Fixed `ToggleState` generator in `crates/holon-integration-tests/src/pbt/state_machine.rs` to filter out the block's current task state, preventing no-op `set_field` calls that don't emit CDC events
- Verified the framework compiles cleanly and now outputs expected initialization messages (`[inv10h_live] HeadlessLiveTree initialized on main panel (N items)`) with matching live vs. fresh rows on subsequent invariants

**Challenges**

- Root layout block has render but no data query, so its `data_rows` remain empty—required tracing through reactive initialization to find the actual data source
- Generator was creating state transitions to the current state, which the SQL layer treats as no-ops and never emit `UpdateAt` events—live tree's `set_data` path never exercised

**Status**

Framework is correctly wired and running. Pre-existing failures (keychord flakes, `STARTED → DONE` set_field divergence) trip before `inv10h_live` completes. Unit test `set_data_propagates_to_state_toggle_child` still fails, confirming the bug exists; integration test will catch it once blockers are cleared.
