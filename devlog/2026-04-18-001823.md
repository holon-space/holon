## Subtree Sharing: Phase 1 Complete → Phase 2 Planning

**Context**

Loro subtree sharing backend is complete and tested. Two `LoroShareBackend` instances can share a subtree over real iroh, accept on the other side, and converge after edits. Phase 1 passed: 22 sync tests including a 16-case PBT exercising the full share → accept → bidirectional sync loop.

**What happened**

- Fixed pre-existing `mount_traversal` test failures (5 tests) — root cause was `uri_to_tree_id` returning `None` for UUID-based URIs. Switched to real lookup path `backend.find_tree_id_by_stable_id(id.id())`.
- Repaired `STABLE_ID` key mismatch in `loro_share_backend.rs` — was using `"stable_id"` but `LoroBackend` stores under `"id"`.
- Un-ignored `advertiser_serves_initiator` test — now runs by default and passes.
- Implemented bidirectional sync: `LoroShareBackend.known_peers` tracks endpoint addresses; `pull_from_peers(shared_tree_id)` initiates sync against all known peers.
- Added PBT coverage (`subtree_share_round_trip_pbt`): seeds tree on A, shares subtree, accepts on B via ticket, applies random edits on both, pulls from A → B, verifies convergence.

**Challenges**

- Two test failures remain, both pre-existing and unrelated to sync: `test_execute_operation` (DI dispatcher/provider wiring) and `test_loro_backend_state_machine` (PBT content-matching flake when siblings have identical content).

**Status**

Phase 1 backend + testing complete. Ready to hand off Phase 2: (1) persistence of shares across restart, (2) auto-resync on local edits, (3) frontend UI (GPUI + iOS), (4) optional: fix pre-existing test flakes. See `docs/SUBTREE_SHARING.md` for threat model and `crates/holon/src/sync/*` for integration points.
