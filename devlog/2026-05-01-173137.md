## Rigorous Code Review and Plan Refinement for ORDER BY + LIMIT on Materialized Views

**Context**: A plan existed to add ORDER BY and LIMIT support to materialized views, but it glossed over critical implementation details. A senior-dev review was needed to surface hidden assumptions, edge cases, and validate them against the actual codebase.

**What happened:**

- Audited the plan against five core modules (`view.rs`, `compiler.rs`, `cursor.rs`, `persistence.rs`, `recursive_operator.rs`) and three support files (`schema.rs`, `util.rs`, `types.rs`), with specific line numbers verified
- Identified 12 high-impact implicit assumptions (H1–H12), ranging from aggregate rowid stability to matview-on-matview cascading semantics to VDBE seekRowid behavior
- Performed 10 quick validations directly against code (e.g., confirmed `WriteRowView` panics on non-`TableRowId` keys at compiler.rs:148–155; verified recursive CTE delta keying at recursive_operator.rs:346–358)
- Discovered the plan would recreate a bug identical to the "IVM JoinOperator TryAdvance Seek Bug" fixed 2026-03-11 — `WriteRowView` only handles `Found/NotFound`, not the `TryAdvance` state that index-btree seeks return
- Created 8 follow-up validation tasks for work requiring code-path execution (VDBE rowid audit, aggregate operator group rowid stability, matview-on-matview downstream semantics, schema-reload persistence)

**Challenges:**

- **Composite-key assumptions were invisible**: the plan assumed "DBSP circuit unchanged," but aggregates with `ORDER BY count` and recursive CTEs with content-hash-keyed rowids impose hidden requirements on stable rowid assignment and delta ordering that were never named
- **Storage unboundedness was glossed over**: LIMIT applied only at the cursor means the btree stores the entire input set, silently violating user expectations; COUNT(*) and CDC consumers would see different cardinalities
- **Schema reload and persistence had no answer**: the plan suggested storing "is index btree" but didn't specify where or how re-opening would re-derive identical storage shape from the AST without diverging

**Decisions:**

- Reject matview-on-matview with upstream LIMIT at DDL time in v1 to avoid undefined cascading semantics
- Make "LIMIT stores full set" an explicit design constraint with documentation; defer true top-K cardinality bounds to later
- Create a parallel `WriteRowViewIndex` state machine (not a parameterized variant) to handle `TryAdvance` and composite-key extraction separately, learning from the 2026-03-11 join-operator fix

**Status**: Plan rewritten with all 12 hypotheses incorporated, 10 quick validations confirmed with code citations, 8 follow-up tasks created with clear acceptance criteria, and three architectural decisions locked in. Ready for implementation phase with risk surface reduced.
