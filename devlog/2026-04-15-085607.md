## Refactoring Layout PBT Harness and Preparing for Multi-Frontend Support

The shared `holon-layout-testing` crate had stale trait-based design from an earlier plan iteration; the actual implementation used a closure-driven `run_scenario` approach but left VMS button-clicking unwired, invariant 10 unimplemented, and the architecture partially annotated.

- Inspected post-refactor state: `GpuiScenarioSession` owns a per-instance `BlockTreeRegistry`, `run_scenario` takes `FnMut(StepInput) -> BoundsSnapshot`, `UserDriver::click_element(&str)` exists but VMS builder doesn't resolve button IDs
- Fixed build errors: `GpuiScenarioSession::open` was shadowing `services` binding; consolidated `with_registry(quiescent: bool)` to provide both shared registry and quiescent runtime
- Baseline proptest passes cleanly
- Started A1 (wire real VMS clicks): began resolving `vms_button_id_for(block_id, mode)` by walking rendered tree to find the switcher button and extract its `data-holon-id`
- Discovered the VMS builder emits `data-holon-widget-type="view-mode-switcher"` and `data-block-id="{block_id}"` for each mode button; button element ID is available via bounds registry lookup

**Challenges:**
- The VMS builder's button ID encoding (`vms_button_{block_id}_{mode_index}`) required tracing through `ViewModeSwitcher` render path to confirm it actually gets attached to the DOM element as `data-holon-id`
- `UserDriver::click_element` needs the bounds registry to resolve ID → rect for synthetic event injection in GPUI; current code assumes the lookup is always successful but should handle missing elements gracefully

**Decisions:**
- Kept `with_registry(quiescent)` as a single constructor parameter rather than splitting into separate methods, to avoid accidental mismatches between registry and runtime flavor
- Deferred full VMS click implementation and invariant 10 (display-node decompilation) to the next session once A1 landing is verified

**Status:** Build is clean, baseline proptest passes, A1 (real VMS clicks) partially implemented — need to land the button ID resolution and test it before moving to invariant 10 or Dioxus follow-up plan.
