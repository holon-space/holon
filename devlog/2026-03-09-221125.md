## Jujutsu Rebase Skill Documentation Review

**Context**

The jj-rebase skill provides a comprehensive workflow for safely rebasing user-authored branches onto upstream main with automatic conflict resolution. This session involved reviewing and documenting the complete skill specification, including bundled scripts, critical revset patterns, and edge case handling.

**What happened**

- Documented all five bundled scripts with their purposes: `jj-rebase-clone.sh` (isolated clone creation), `jj-auto-resolve.sh` (mergiraf automation), `jj-cleanup-divergent.sh` (divergent commit removal), `jj-imerge2.sh` (preferred incremental rebase), and `jj-rebase-finish.sh` (result transfer)
- Established critical revset patterns: correct root identification using `roots(author(...) ~ ::main)` instead of the broken `roots(main..) & author(...)` pattern; divergent commit identification by checking `immutable` property and `parents()` rather than trusting `/N` offset numbering
- Structured the workflow into 12 steps covering clone creation, divergent cleanup, conflict resolution strategy, rebase method selection (bulk vs. incremental), auto-resolution with mergiraf, manual conflict handling, validation, and result transfer
- Documented the bulk vs. incremental rebase decision tree based on first-parent count (< 20 merges = bulk; > 80 = incremental)
- Captured pre-existing conflict handling strategy using `jj restore` to propagate resolutions without destroying descendant changes
- Included comprehensive edge case coverage: divergent merges (keep copy with most parents), immutable divergent copies (exclude from `-s`, use commit ID), mega-merges (rebase parents first, then handle merge's own conflicts), generated files (restore from main, regenerate), cascading vs. pre-existing conflicts, and `jj squash -u` re-conflicting parents

**Decisions**

- **Incremental rebase as default**: `jj-imerge2.sh` preferred over `jj-imerge.sh` (legacy) because it uses `-s` (safe) instead of `-b` (dangerous) and auto-runs mergiraf, keeping conflicts small and traceable to specific upstream merges
- **Divergent cleanup before rebase**: Pre-existing divergent copies must be abandoned in step 3 before any rebase to prevent cascade through both chains (doubling conflicts)
- **`jj restore` over manual editing**: For propagating pre-existing conflict resolutions to descendants, use `jj restore` only on files not modified by the descendant to avoid silent destruction of descendant changes

**Status**

Skill documentation complete with 12-step workflow, 5 bundled scripts, critical revset patterns, decision trees for rebase strategy selection, and comprehensive edge case handling. Ready for use; edge cases and script behavior thoroughly specified.
