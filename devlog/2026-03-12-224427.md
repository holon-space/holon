## Fix: editable_text nodes missing entity/operations from render_block

**Context**: The `EditViaDisplayTree` transition was added to catch when `editable_text` nodes lack entity/operations metadata, but the test revealed the bug still exists — the nodes are being created without the required context.

**What happened**:
- Test hit the new `EditViaDisplayTree` transition on block `block:-p7-29ei--q12-1`
- Walked the display tree looking for an `EditableText` node with `entity["id"]` matching that block
- Found **no such node** — all nodes in the tree were `table_row` elements
- The block's content ("MATCH (root:Block)...") exists in the reference data but wasn't rendered as an editable text node

**Root cause identified**: The block `block:-p7-29ei--q12-1` is a **query block** (has `::src::0` suffix indicating it's a source code block), not a plain text block. The `render_block()` function likely renders query/source blocks differently — possibly as `table_row` within a table context, not as standalone `EditableText`.

**What was fixed**:
- Modified `crates/holon-frontend/src/shadow_builders/render_block.rs` to pass `ba.ctx.row().clone()` and `ba.ctx.operations.clone()` to `ViewModel` for the `TextContent` arm
- This only applies to the `TextContent` variant — query blocks (`RenderBlockResult::QueryResult` or similar) were never going through the `TextContent` path, so the fix doesn't apply to them

**Decisions**:
- The test is correctly exercising the bug — it picked a **query block** at random, not a text block
- The transition generation strategy should be updated to **only target plain text blocks** (those with `content_type == "text"`), not query blocks
- We need to filter `text_block_ids` to exclude blocks that are rendered as tables/queries

**Status**: The `render_block` fix is correct for text blocks, but the `EditViaDisplayTree` transition strategy needs refinement to skip query blocks. The PBT generation should only select blocks where the display tree will actually render an `EditableText` node.
