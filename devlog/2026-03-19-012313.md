## Fixed MCP Server Type Mismatches from Incomplete DI Migration

The MCP frontend had compilation errors from an incomplete refactoring that changed `HolonMcpServer::new` to accept `FrontendSession` instead of `BackendEngine`, but didn't thread this change through the dependency injection layer or implement the required methods.

**What happened:**
- Identified the root cause: a prior session migrated from `ferrous_di` to `fluxdi` and updated `server.rs` to take `FrontendSession`, but left `di.rs`, `run_http_server`, and `tools.rs` in an inconsistent state
- Found three distinct compilation errors: wrong type passed to `HolonMcpServer::new()` in `di.rs:176`, missing `interpret_to_view_model()` method in `tools.rs:1243`, and incompatible signature for `run_http_server`
- Added `interpret_to_view_model()` method to `FrontendSession` that creates a headless `RenderContext` and uses `create_shadow_interpreter()` to interpret render expressions
- Updated `di.rs` to resolve `FrontendSession` from the DI container instead of `BackendEngine`
- Fixed `run_http_server` and `McpServerHandle` to accept `Arc<FrontendSession>` throughout the call chain

**Challenges:**
- The prior refactoring session (ID `11ff5894-a293-405e-be9a-b40f9f0a7e79`) couldn't be reopened, requiring reconstruction from git history and code analysis
- `interpret_to_view_model()` wasn't defined anywhere despite being called in the FFI bridge, indicating incomplete migration — resolved by examining how `RenderContext::headless()` and shadow interpreters work in other frontends

**Status:** All three compilation errors fixed; MCP frontend should now compile and properly use `FrontendSession` for interpretation.
