## Fixed vertical scrolling in GPUI main panel with relative/absolute layout pattern

The main panel had stopped scrolling even though it had `overflow_y_scroll()`. GPUI's Taffy layout engine requires a **definite bounded height** for scroll to activate, but percentage-height constraints don't resolve through flex-computed parent heights.

### What happened

- **Diagnosed the root cause**: Taffy doesn't resolve `height: 100%` through flex-determined dimensions, matching CSS spec behavior
- **Initial attempt**: Added `h_full()` + `min_h_0()` to the flex chain — didn't work
- **Confirmed scroll works with hardcoded height**: Added `max_h(px(800.0))` as a test — confirmed the issue was the height constraint, not scroll event handling
- **Initial dynamic fix**: Computed window height in `HolonApp::render`, stored in `BoundsRegistry`, used as explicit `max_h` constraint — worked for top-level but broken for nested columns
- **Final robust fix**: Replaced window-height plumbing with a standard CSS pattern: **relative-positioned flex child containing an absolute-positioned scrollable inner** that fills via `inset_0()`

### Challenges

- The flex chain approach (`h_full()` + `min_h_0()`) failed because GPUI's Taffy doesn't treat flex-computed heights as explicit constraints for percentage-height resolution
- The hardcoded-height workaround revealed the real problem but was fragile for window resizing and impossible to handle correctly for nested columns

### Decisions

- Chose the relative/absolute pattern over recursive height tracking because it works at **any nesting depth** without plumbing constraints through the render context
- This pattern is standard CSS for scrollable flex children and doesn't require window-size computation

**Status**: Vertical scrolling works correctly in the main panel and any nested column layouts; no window size tracking needed.
