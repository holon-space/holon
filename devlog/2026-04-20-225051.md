## Implement Stability Invariant for Backend CDC Quiescence

The PBT wasn't detecting spurious CDC event churn happening in the GPUI frontend signal layer, even though the backend CDC streams were clean.

**What happened**

- Added `assert_cdc_quiescent()` method to `TestEnvironment` that polls all CDC streams (`all_blocks_watch`, `region_streams`) after a brief settlement period and asserts zero additional events
- Called this method from `apply_transition_async()` after the initial `drain_cdc_events()` to catch any spurious backend emissions before invariant checks run
- Ran the full PBT suite and confirmed **inv16 passes** — the backend CDC is genuinely quiescent
- This ruled out the backend as the churn source and isolated the problem to the **frontend's signal-to-VecDiff conversion layer**

**Challenges**

- The pre-existing `inv1` failures (backend divergence) masked the new invariant's behavior during test runs
- Had to trace through the PBT's `apply_transition_async` → `check_invariants_async` ordering to confirm inv16 runs before invariant panics
- Discovered that even though inv16 runs silently (no spurious events), it doesn't fully explain the frontend churn because the problem lives in GPUI's `ReactiveShell` entity reconciliation, not the CDC stream itself

**Decisions**

- Placed the stability check in `apply_transition_async` (not `check_invariants_async`) since it requires `&mut self` for stream polling
- Guarded the check to only run after app startup (when watches exist) to avoid empty loop overhead

**Status**

Invariant inv16 is implemented and passing. Root cause confirmed: backend CDC is clean, but frontend's flat driver (`reactive_view.rs:594-630`) converts every fine-grained CDC update into a full `VecDiff::Replace`, triggering unnecessary GPUI re-renders. Next step: write a unit test in `reactive_view.rs` that reproduces the churn at the signal level before attempting a fix.
