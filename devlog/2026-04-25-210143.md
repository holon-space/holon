## Critiqued and refined the "apply parent_id to Loro" plan

The original plan proposed a mechanical fix for parent_id changes not reaching Loro's tree structure, but skipped reproduction-first discipline and deferred a load-bearing safety guard to a "follow-up."

**What happened:**

- Ran a senior-dev code review against the plan, examining 8 distinct failure modes: missing reproduction test, unnecessary verification steps, ignoring simpler primitives, deferring the actual concurrency guard, bespoke test patterns that rot, unhandled sentinel/unseeded-parent crashes, atomicity gaps, and implicit trade-offs
- Verified each criticism against the actual codebase (loro_backend.rs, loro_sync_controller.rs, sql_operation_provider.rs, traits.rs)
- Confirmed the core bug (parent_id never reaches `tree.parent` due to falling into `props` map instead of triggering `tree.mov`, L744-L758)
- Confirmed the concurrency risk: `_expected_parent_id` guard is missing in `loro_sync_controller.rs:473-489` — same class as Bug #1 (`_expected_content` for content column)
- Confirmed crash risk: `resolve_parent_tree_id` errors on unseeded-parent URIs (L278-280)

**Decisions:**

- **Reproduction first**: two failing tests using existing `loro_sync` stub harness, not a bespoke two-LoroDoc setup
- **Promote `_expected_parent_id`**: move from "follow-up" to core scope; it prevents the same stomping window that Bug #1 fixed for content
- **Use `update_parent_id` primitive** (loro_backend.rs:615) instead of `move_block` — narrower, honest about intent, no `after` parameter to defend
- **Replicate `create_placeholder_root` dance** for unseeded-parent URIs, same pattern already in `apply_create`

**Status**: Plan rewritten with 6 tasks; reproduction tests defined; concurrency guard promoted to core scope; crash risks mitigated. Ready for implementation.
