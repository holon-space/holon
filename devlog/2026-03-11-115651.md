## GPUI Blank Screen: Streaming Works, Rendering Stalls

The GPUI was displaying only a gray bar despite data being in the database. Logs appeared to show no events arriving, masking the real issue.

**What happened**

- Added `eprintln!` diagnostics to the CDC loop to bypass log filtering and confirm event flow
- Confirmed `watch_ui` delivers `Structure(gen=1, rows=3)` events successfully — streaming pipeline works
- Root render produces `columns` with 3 `block_ref` nodes (Left Sidebar, Main Panel, Right Sidebar)
- Added ViewModel tree printer to track what each block_ref renders to across frames
- Observed 4 render passes over ~1 minute: Frame 1 (empty table), Frames 2–3 (all block_refs as `spacer` placeholders), Frame 4 (left sidebar resolves to `list` with 5 rows, main panel to `tree`, right sidebar still `spacer`)
- No additional renders after Frame 4, despite the 10-iteration 2-second periodic timer that should trigger ~10 more re-renders

**Root cause identified**

The `BlockRenderCache` spawns background watchers for sub-blocks that populate real widget data (table/list/tree), but has **no mechanism to signal GPUI for re-renders** when that data arrives. The code relies on a periodic `smol::Timer` loop that calls `cx_handle.update()`, but the timer stops firing after the initial burst. The `cx_handle.update()` call from an async context may not properly schedule re-renders in GPUI's main loop.

**Status**

Streaming and event delivery confirmed working. Block caching partially works (left sidebar renders on Frame 4). The periodic refresh timer stalls, leaving right sidebar and subsequent updates as `spacer` placeholders. Next session: investigate why `cx_handle.update()` fails to re-schedule renders from the async timer task.
