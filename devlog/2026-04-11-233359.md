## Fixed stale matview showing unnamed documents in sidebar

The left sidebar was displaying 50 items instead of 4, with 46 showing empty names. This was caused by a stale materialized view created before a Turso IVM NULL-comparison fix was deployed.

**What happened**

- Identified the sidebar query in `/Users/martin/Workspaces/pkm/holon-pkm/index.org` (line 23-24): `from block | filter name != ""` was missing explicit NULL filtering
- Verified Turso IVM bug: matview with `WHERE name != ''` returned 50 rows (wrong), but regular SQL returned 4 rows (correct)
- Confirmed Turso fix at commit `c4dc3fc68` ("fix: IVM filter treats NULL comparisons as true instead of NULL/false") was present in the compiled binary
- Ran existing reproducer `turso_ivm_null_where_neq_repro.rs` and confirmed the fix works for fresh matviews
- Enhanced reproducer with 5 test scenarios: fresh inserts, NULL↔value UPDATEs, DB reopen, and production-like schema — all pass
- Updated org file sidebar PRQL to add explicit `filter name != null` and exclude `"index"` and `"__default__"` entries

**Challenges**

- The production matview had stale data populated *before* the fix was compiled in. The fix works correctly for newly-created matviews, but stale views persist across restarts. Root cause identified: running binary didn't recreate the matview after fix deployment.

**Decisions**

- Added `filter name != null` to org file despite Turso fix being present — explicit NULL filtering is good defensive programming and doesn't rely on SQL three-valued logic semantics

**Status**: Sidebar now shows only 3 named documents; reproducer enhanced and all NULL-comparison scenarios validated.
