## Macro-Based Widget Builder System — Plan Refinement

**Context**

The macro-based widget builder system requires aligning with existing type systems (particularly `OperationParam` / `TypeHint`) and clarifying how frontend support is enforced. Previous session identified a design contradiction: exhaustive match enforcement is incompatible with frontend-specific widget support.

**What happened**

- Explored `crates/holon/src/entity_profile.rs` and discovered `UiInfo` + `ProfileResolver::filter_profile()` already handles frontend-specific widget filtering — unsupported widgets never reach a frontend's render pipeline
- Reviewed `crates/holon/src/operation.rs` and found `OperationParam` with `name`, `type_hint: TypeHint`, `description` — nearly identical to the proposed `ParamMeta`
- Confirmed `DisplayNode` represents the *fully expanded* render tree (e.g., a `list` with 1000 rows contains 1000 materialized `DisplayNode` entries), not a template
- Identified three overlapping type systems: `TypeHint` (operations), `ParamMeta` (proposed), and template args — unified into single hierarchy

**Decisions**

- **Frontend rendering**: Switch from exhaustive `match node.kind` to non-exhaustive match with `_ => render_fallback()`. Compiler-enforced exhaustion is unnecessary; `UiInfo` filtering prevents incompatible widgets at the profile resolution level, before frontend rendering begins.
- **Type unification**: `WidgetMeta.params` becomes `&'static [OperationParam]` directly (no new `ParamMeta` type). Extend `TypeHint` enum with `Template` and `Children` variants to cover all builder parameter types.
- **Parameter defaults**: Add optional `#[default = value]` attribute syntax. When omitted, use type-appropriate fallbacks (`""` for `String`, `false` for `bool`, `None` for `Option<T>`).

**Status**

Plan refined with type system unification and frontend rendering model clarified. Ready to proceed with implementation phase.
