## Unified Multi-Variant Frontend Rendering with UI State

We needed a way to support frontend-local switching between render modes (edit vs. display, tree vs. table layouts) without backend round-trips, while keeping the rendering model declarative and avoiding data duplication.

### What happened

- **Analyzed the architecture**: `ReactiveQueryResults` holds canonical data; `RenderExpr` is just a tiny interpretation layer. Switching layouts doesn't duplicate data — only the ViewModel tree is re-interpreted.
- **Identified the pattern**: `FilterExpr` already exists in the codebase and is almost exactly what we need for representing UI state conditions.
- **Designed a minimal `UiCondition` type**: A small, JSON-serializable expression enum (Not, And, Eq, Var) that can be evaluated frontend-side without needing a Rhai runtime — critical for Flutter/WASM portability.
- **Unified two resolution paths**: Both collection-level (`tree(...)` / `table(...)`) and per-row block rendering (`source_editor` vs `query_results`) now use the same `Vec<VariantCandidate>` type. Frontend evaluates UI conditions locally to pick the active variant.
- **Redesigned profile authoring**: Move away from standalone `#+BEGIN_SRC render` blocks. Entity profiles in YAML now define all variants (collection + per-row) with conditions. Single-variant profiles remain simple; multi-variant profiles declare both data conditions (backend-evaluated) and UI conditions (frontend-evaluated).

### Decisions

- **Separate evaluation contexts**: `FilterExpr` operates on row data; `UiCondition` operates on UI state. Kept distinct for now, can unify later if the pattern matures.
- **Top-level conjunctions only**: Condition splitting extracts `&&` conjuncts to move data predicates to backend. Nested boolean logic stays in Rhai for now.
- **Unified types, phased authoring**: Update the type system to support collection + row variants with the same mechanism; introduce YAML multi-variant syntax; keep standalone render blocks working as a bridge.

### Status

Plan written. Ready to implement: start with `UiCondition` type and condition-splitting logic (backend-only, backward-compatible), then integrate into profile resolution, then update frontends to materialize multiple candidates and evaluate UI state locally.
