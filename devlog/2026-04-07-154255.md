## Memory Budget Enforcement for PBT Transitions

**Context:** Holon's memory footprint varies wildly (250MB–4GB+), and there was no way to catch memory bloat during property-based testing. We needed a lightweight invariant to enforce per-transition and cumulative memory budgets, with on-breach diagnosis.

**What happened:**

- Added `MemoryMetrics` struct to track RSS before/after each transition, plus cumulative growth across the full test run
- Implemented per-transition RSS delta limits (200MB for `StartApp`, 30MB for most others, scaled for heavy ops) in `max_rss_delta_bytes()`
- Added cumulative RSS cap (800MB across entire PBT) to catch slow leaks
- Extended `check_budget()` to validate both per-transition deltas and cumulative growth
- Wired RSS tracking into `sut.rs`: capture baseline on first transition, build `MemoryMetrics` in inv13, trigger diagnosis on breach
- Added `diagnose_memory()` function to dump system memory stats (`footprint` on macOS, `/proc/self/status` + `smaps_rollup` on Linux) when a budget violation fires

**Challenges:**

- **`allocative` ruled out:** Third-party types (`LoroDocument`, Turso client internals) can't derive `Allocative` without forking their repos. Instead, deferred struct-level diagnosis to dhat follow-up profiling runs.
- **Avoiding allocator conflicts:** Didn't add `#[global_allocator]` wrapper (like `cap`) to avoid conflicts with the existing dhat profiling feature.
- **Calibration run incomplete:** StartApp spiked to +1.5GB (575K OTel spans), far exceeding the 200MB budget guess. Pre-existing Loro assertion failures in the test harness prevented a full calibration run. Limits need real data from a stable test run.

**Decisions:**

- Two-tier diagnosis: RSS gate in tests (instant feedback), dhat for follow-up (struct-level detail on demand)
- System tools over custom allocators: `footprint`/`/proc` are zero-cost and available everywhere
- Per-transition + cumulative budgets: catches both sudden spikes and slow leaks

**Status:** Memory budget enforcement implemented and compiling; calibration of per-transition limits pending a stable test run (current test harness has pre-existing failures blocking data collection).
