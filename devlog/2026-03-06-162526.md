## Memory Leak Detection Infrastructure

**Context:** The application experienced a catastrophic 185GB memory leak overnight. Rather than searching blindly through code, we built two complementary profiling layers to catch future leaks in minutes.

**What happened:**
- Added `memory_monitor` module to `holon-frontend` that logs RSS every 30 seconds via `sysinfo`, with threshold alerts at >100MB/500MB growth deltas
- Integrated `dhat` heap profiler behind a `heap-profile` feature flag for allocation-site tracking—writes `dhat-heap.json` viewable at the DHAT web viewer
- Wired memory monitor into all `FrontendSession` constructors (all frontends automatically get it)
- Added Ctrl+C handler to `dioxus` frontend so graceful shutdown triggers dhat output write
- Tested both layers on the Dioxus frontend—verified memory monitor logging and dhat profiler writing output
- Added missing `error` builder to Dioxus renderer (`frontends/dioxus/src/render/builders.rs:536`) to handle error widget specs from Rust
- Documented the full profiling workflow in `docs/MEMORY_PROFILING.md`

**Challenges:**
- **dhat output not writing on app close:** Dioxus's webview calls `std::process::exit()` which bypasses Rust drop semantics. Solved by installing a `ctrlc` handler that calls `std::process::exit(0)`, triggering `atexit` handlers.
- **Profiler initialization timing:** dhat init needed to happen in `main()` on the stack, not in a `FrontendSession` static, to ensure drop on exit.

**Decisions:**
- Two-layer approach: always-on RSS monitor catches growth immediately; optional heap profiler finds the exact allocation site. One layer alone wouldn't be sufficient for a 185GB leak.
- Used `dhat` (pure Rust) over `valgrind`/`perf` for cross-platform simplicity—works on macOS ARM without external tools.

**Status:** Memory profiling infrastructure complete and tested. Next leak can be pinpointed in minutes using the documented workflow.
