## Fixed PBT Flakiness and Source-Aware Cache Trimming

Property-based tests were failing intermittently due to multiple race conditions and normalization mismatches between the reference model and SQL backend when Loro synchronization was enabled.

**What happened**

- Fixed `inv10e` false-positive by gating behind `root_render_expr().is_some()` check
- Added missing widget names (`bottom_dock`, `op_button`) to DSL parser
- Fixed MCP sync engine `usize` underflow using `saturating_sub`
- **Rewrote `trimmed_content` logic** to match reference model's source-vs-text normalization, ensuring SQL cache reflects correct content after external changes
- Diagnosed and fixed **Bug #2**: external Updates weren't reaching SQL when Loro was enabled — added 100ms `poll_external_changes` fallback to OrgSync select loop to catch FSEvents misses
- Updated all `trim_content` call sites to pass `is_source` flag for context-aware normalization

**Challenges**

- Flakiness masked underlying bugs: fixing Bug #2 revealed a separate property update regression in SQL that only surfaces with Loro enabled
- Race condition between `on_file_changed` and `on_block_changed` could overwrite disk with stale cache; prevented by polling backstop
- Content trimming differences between reference model and SQL required tracing through both Loro outbound updates and CacheEventSubscriber paths

**Decisions**

- Added periodic polling fallback instead of relying solely on FSEvents, trading minor CPU overhead for reliability
- Made trim logic source-aware rather than trying to normalize after-the-fact, reducing state confusion

**Status**

`cross_executor` and `sql_only` variants pass at 50 cases; full variant still flaky due to unrelated property regression in SQL. Handoff document updated with investigation pointers for remaining bug.
