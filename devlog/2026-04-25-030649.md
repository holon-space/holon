## Fixed Concurrent Text CRDT Merge in Peer Sync Reference Model

The property-based test `general_e2e_pbt.rs` was failing when primary and peer concurrently modified the same block's text. The reference model's naive "peer wins" strategy didn't match Loro's RGA text CRDT behavior.

**What happened:**

- Identified root cause: `merge_peer_blocks_into_primary` treated peer modifications as simple replacement, ignoring concurrent local edits. Loro's RGA-style CRDT concatenates both insertions (e.g., `"hahzzvglMHj tVTqCaJYj"`).
- Tracked AddPeer-time baseline content per peer in `PeerRefState` to enable proper CRDT merging
- Applied existing `loro_merge_text` helper (three-way merge with baseline) to peer block merging logic
- Updated `merge_peer_blocks_into_primary` to accept baseline and use CRDT merge when both primary and peer diverged from baseline
- Updated both `MergeFromPeer` and `SyncWithPeer` apply paths with baseline tracking and refresh after sync
- Added `refresh_peer_baseline` helper with correct semantics: `MergeFromPeer` (one-way, peer doc unchanged) vs `SyncWithPeer` (bidirectional, both converge)

**Challenges:**

- Distinguishing concurrent mutations (both sides edited same block) from sequential overwrites required tracking the AddPeer-time baseline separately
- `SyncWithPeer` required bidirectional baseline refresh (peer baseline after applying primary→peer changes), while `MergeFromPeer` only refreshes primary baseline

**Decisions:**

- Reused `loro_merge_text` pattern rather than duplicating merge logic — keeps CRDT semantics consistent across the codebase
- Stored baseline in `PeerRefState` at AddPeer time rather than reconstructing it — cleaner state tracking

**Status:** CRDT merge fix complete and compiling. Test now surfaces a separate pre-existing `ClickBlock` precondition timeout unrelated to this change.
