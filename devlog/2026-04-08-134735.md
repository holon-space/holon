## GraphQL Schema Registry Case-Sensitivity Bug in Block Resolution

The Holon GPUI launcher was failing to render the root-layout block with "Failed to create materialized view" errors. The root cause was a case-sensitivity mismatch in GraphQL table resolution that caused the query compiler to fall back to the generic EAV `nodes`/`edges` schema instead of using the registered `block` table.

**What happened:**

- Traced "Failed to create materialized view" error through render pipeline to GQL query compilation
- Discovered compiled SQL was referencing `nodes` and `edges` tables instead of the `block` table
- Found Block entity registered in GraphSchemaRegistry with `graph_label = "block"` (lowercase)
- Located GQL query in org file using `(root:Block)` with uppercase B
- Confirmed the issue: `node_resolver()` performs case-sensitive HashMap lookup (`self.nodes.get("Block")`) which fails to find the lowercase `"block"` key, causing fallback to EAV resolver
- Issue is in `gql-to-sql` crate at line 176 where the lookup happens

**Challenges:**

- Generic error message ("Failed to create materialized view") didn't directly point to schema resolution — required tracing through entire GQL compilation pipeline
- Case sensitivity was subtle; graph_label convention uses lowercase but GQL query convention uses PascalCase

**Decisions:**

- Two viable fixes identified: (1) make `node_resolver()` case-insensitive, or (2) change GQL query to lowercase `block`
- Option 1 preferred as more robust — aligns with GQL naming conventions where types are PascalCase

**Status:** Root cause identified and fix strategy determined; awaiting decision on implementation approach before patching `gql-to-sql`.
