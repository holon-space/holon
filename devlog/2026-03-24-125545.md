## Entity Profile Priority System Redesign

Variants now carry explicit priorities instead of profiles, eliminating the need for a separate "default" designation.

**What happened**

- Moved `priority: i32` from profile level to individual variant level in `EntityProfile` and `RawEntityProfile`
- Removed the `default: String` field entirely — the highest-priority variant is chosen implicitly
- Updated `build_cache_from_source()` to sort variants by priority descending (highest wins) after merging profiles
- Set `priority: -1` on all variants in `assets/default/index.org`, ensuring seeded variants always lose to user variants (implicit priority `0`)
- Updated variant selection logic in `render_eval.rs` to pick the first variant after priority sort

**Decisions**

- **Priority as variant property** — Each rendering mode (default, editing, task, etc.) can have its own priority, enabling fine-grained control over which variant appears in which context
- **No explicit default field** — The highest-priority variant is the implicit default. Simpler model, fewer special cases
- **Negative priority for seeded content** — `-1` for bundled profiles guarantees they lose to user variants (default `0`), with room for explicit user overrides at `1+`

**Status**

Profile priority system now deterministic — highest-priority variant always wins the merge, with seeded defaults at `-1` and user profiles at `0` by default. `doc:__default__` heuristics eliminated in favor of explicit priorities.
