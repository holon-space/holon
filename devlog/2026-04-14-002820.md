## Implement fail-loud geometry in user drivers and add `wait_for_element_bounds` helper

**Context:** The previous session identified that user drivers had silent fallbacks when element bounds weren't available, masking timing issues in tests. This session implemented the critique's recommendations: making geometry failures explicit and providing a quiescence helper.

**What happened:**

- Rewrote `GpuiUserDriver` methods (`click_entity`, `type_text`, `send_key_chord`, `scroll_entity`) to return `Result` instead of silently synthesizing when element bounds are unavailable; added `require_element_center(verb)` helper with context-rich errors
- Refactored `EnigoUserDriver` to fail loud on `Enigo::new()` failures (previously traced a warning and fell back to synthetic dispatch) and rewrote all `try_*` helpers to `Result`-returning versions
- Built `wait_for_element_bounds(entity_id)` polling helper in `holon-integration-tests` that yields `ElementInfo` or a timeout error; tests can now explicitly synchronize before interacting
- Removed stale deprecation language from trait docs (the parallel F10 refactor had already renamed `apply_ui_mutation` → `synthetic_dispatch`, making the old `#[allow(deprecated)]` blocks moot)
- Fixed a collapsed-newline bug in `phased.rs:246` introduced by the parallel refactor

**Challenges:**

- **Parallel refactor collision:** F10 refactor had already renamed the deprecated method to `synthetic_dispatch` and added new quiescence primitives. Adapted the plan to work on top of the new baseline instead of fighting it.
- **BoundsRegistry semantics:** Confirmed that `staged` → `committed` promotion requires a render-frame boundary; tests creating and immediately interacting with blocks need the new `wait_for_element_bounds` helper (not optional).
- **Entity prefix hardcoding:** `GpuiUserDriver::element_center` only resolves `render-block-{id}` entities; added a `debug_assert` documenting the constraint and deferred fixing non-block support.

**Status:** All drivers compile; `general_e2e_pbt_sql_only` passes with zero bounds-missing errors or driver panics. A pre-existing custom-property round-trip backend bug is unrelated to these changes.
