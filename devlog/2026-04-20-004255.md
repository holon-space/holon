## Fixed block profile registration and renamed col → column for consistency

The Holon app was rendering a blank window because the `block` entity profile from `block_profile.yaml` never reached the `ProfileResolver` cache. Investigation revealed the YAML used `col(columns(...), columns(...))` as a vertical stack widget, but `col` was reserved exclusively for `ColumnRef` construction in the render DSL, causing the profile parse to fail silently.

### What happened

- Used `describe_ui`, `describe_navigation`, and `execute_query` with holon-direct to trace the blank window to `_profile.name == "fallback"`
- Removed the fallback profile entirely and added panic-on-resolution-failure to expose the real bug: `"No profile registered for entity 'block'"` with only `["collection", "person"]` in cache
- Added diagnostic eprintln output to identify where block's profile variants were being dropped — found parse error: *"Function not found: col (map, map)"*
- Fixed root cause by renaming the vertical stack widget from `col(...)` to `column(...)` in both DSL and all internal dispatch points
- Updated `block_profile.yaml` to use `column(columns(...), columns(...))`
- Renamed all enum variants and files for full internal consistency: `ViewKind::Col` → `ViewKind::Column`, `ReactiveViewKind::Col` → `ReactiveViewKind::Column`, and `col.rs` → `column.rs`
- Fixed collateral damage from overzealous find-replace (`Columnumns` → `Columns`, etc.)

### Challenges

- DSL parse errors were silently converted to `None` instead of panicking, making diagnosis difficult until removing the fallback forced loud failures
- `col` was ambiguous: both a function name for column references and a desired widget name, resolved by reserving `col("name")` for `ColumnRef` and using `column(...)` for the widget

### Status

All crates compile; ready to rebuild and restart. The three-panel layout should now render correctly instead of showing a blank window.
