## Fix: Org File Round-Trip After UI Mutations

After UI content mutations (EditViaDisplayTree/EditViaViewModel), invariant checks were finding stale org file content. The root cause was a test synchronization issue, not a production bug.

### What happened

- **Identified root cause**: `OrgSyncController` re-renders org files in a background tokio task after mutations are committed to the database. The test was calling `check_invariants()` immediately after the mutation returned, before the background task had been scheduled.
- **Reviewed existing pattern**: `apply_external_mutation()` already calls `wait_for_org_files_stable(50, 5000ms)` to synchronize async event delivery. UI mutation transitions were missing this.
- **Applied fix**: Added `wait_for_org_files_stable(50, Duration::from_millis(5000))` in `check_invariants_async()` (line 1307 in `crates/holon-integration-tests/src/pbt/sut.rs`) before the org file parse assertion, but only when the transition wasn't nav-only.
- **Verified**: Compilation succeeded with no new warnings; integration test suite compiles cleanly.

### Challenges

The `QueryableCache` is not an in-memory cache—it reads from the same Turso DB, so the SQL write *is* committed immediately. The issue was purely about the async event-driven re-render needing time to execute, which is the same async behavior users experience in production but don't race against.

### Decisions

- Placed the wait in `check_invariants_async()` rather than in each mutation transition, centralizing the synchronization concern where all org file assertions happen.

**Status**: Fix applied and compiled. Ready for test verification.
