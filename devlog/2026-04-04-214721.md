## Fixed panic in editor cursor watcher by bypassing enrichment pipeline

The frontend was crashing when syncing editor cursor position due to a mismatch between query shape and enrichment assumptions.

**What happened:**

- `watch_editor_cursor()` queries the `current_editor_focus` materialized view, which only has columns `{region, block_id, cursor_offset, updated_at}` — no `id` column
- This CDC stream was routed through `session.query_and_watch()`, which wraps all results in `enrich_stream()`
- The enrichment pipeline unconditionally calls `row_id(row).expect("No id found")` to resolve entity profiles
- Rows without `id` caused a panic: `No id found in {"cursor_offset": Integer(17), "block_id": String("block:f9a2e27c-...")}`

**Root cause:** The enrichment pipeline assumes every row has entity identity, but cursor position updates are state rows, not entity rows. They don't need profile resolution.

**Solution:**

- Rewrote `watch_editor_cursor()` in `crates/holon-api/src/lib.rs` to call `engine.query_and_watch()` directly, bypassing enrichment
- The raw `RowChangeStream` is consumed directly and updates a `Mutable<Option<EditorCursorState>>` signal with `(block_id, cursor_offset)` from the latest row
- Deleted the now-unused `watch_raw_query()` method

**Status:** Fixed and verified — `holon-api` and `holon-frontend` both compile cleanly. The API surface for consumers of `watch_editor_cursor` is unchanged.
