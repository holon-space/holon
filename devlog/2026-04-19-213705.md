## Fixed Bug #1: UI-Loro race causing content regression

Loro snapshot-based updates were overwriting SQL rows with stale content when a concurrent UI write had already advanced the value. The UI spot-check at `sut.rs:4241` was catching this mismatch reliably in `cross_executor` after the vfn11 false-positive was cleared.

**What happened:**
- Identified that `on_loro_changed` emits full-row updates via `block_to_params` using a Loro snapshot that can lag direct SQL writes, creating a window where a stale update regresses the target row
- Added `_expected_content` field to diff-emitted update ops when content is among the changed fields (refined to avoid spuriously gating property-only updates)
- Modified `prepare_update` in `sql_operation_provider.rs` to read `_expected_content` and append `AND content = '<escaped>'` to the UPDATE's WHERE clause, matching the write-side `trimmed_content` normalization
- Verified fix across 50 cases of `cross_executor` — Bug #1's previously-reliable panic site is now silent
- Also fixed vfn11 invariant false-positive by correcting ref-state `NavigateHome` to clear `focused_block` globally, mirroring production's `go_home` behavior

**Challenges:**
- vfn11 was masking Bug #1 in test output; fixing vfn11 revealed the underlying race condition
- Had to carefully scope the `_expected_content` guard to content-only changes to avoid blocking unrelated property updates

**Status:** Bug #1 fixed and verified. With it gone, cross_executor now reliably reproduces Bug #2 (peer merge blocks disappearing), and the handoff doc has been updated to reflect all three variants' current behavior.
