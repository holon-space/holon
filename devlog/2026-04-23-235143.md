## Fixed Loro content overwrite race condition in cross-executor variant

The PBT test `general_e2e_pbt.rs` was failing in the cross_executor variant when content updates from `ApplyMutation::Update` were being overwritten by stale Loro sync operations.

### What happened

- Ran the PBT test suite and identified `cross_executor` variant failing with "Blocks differ between actual and expected"
- Traced the failure to block `block:x-t-s---95-zz7u` where content was `"Kg4 GN"` in the backend but expected `"GMkjxI1C"` in the reference model
- Analyzed the transition logs and discovered a race condition: `ApplyMutation::Update` correctly wrote the new content to SQL, but a concurrent Loro `BulkExternalAdd` event triggered a full-row UPDATE that overwrote it with stale data
- Root cause: when Loro content didn't change between snapshots, the `_expected_content` guard was intentionally skipped (lines 473-476), but `block_to_params` still included the old content value in the SET clause without a guard
- The diff guard at lines 521-528 saw SQL already had the newer value (`"GMkjxI1C"` ≠ `"Kg4 GN"`) and incorrectly allowed the stale overwrite through

### Challenges

- **Race condition timing**: The Loro sync ran from stale snapshot data, creating a window where SQL updates could be clobbered
- **Guard logic inversion**: The guard was missing precisely when needed — when Loro content was unchanged but other fields (properties, parent) were modified

### Decisions

- Instead of including all fields in UPDATE params regardless of changes, created `block_diff_params()` function that only includes fields that actually differ between old and new Loro snapshots
- This prevents unchanged content from entering the SET clause during properties-only or parent-only updates, eliminating the race condition entirely

**Status**: Cross_executor variant now passes; running full test suite to validate no regressions (sql_only and full variants pending verification).
