## Fixed Custom Property Sync Barrier in Property-Only Mutations

The property-based test for the e2e integration suite was failing when mutations changed only custom properties like `effort`, without altering block content or count. The root cause was a missing synchronization barrier in the External mutation handler.

**What happened**

- Reproduced the failing test and added detailed tracing to track SQL execution paths
- Identified that `E2ESut::apply_mutation` waited for block `content` convergence but never validated custom `properties`
- Property-only mutations (like `effort` changes) triggered invariant checks before the file-watcherâ†’OrgSyncController path committed the UPDATE
- Extended the External mutation spot-check in `crates/holon-integration-tests/src/pbt/sut.rs` to poll the `properties` column until all mutation fields landed in the database
- Added two helper functions: `mutation_expected_properties` and `row_properties_to_map`
- Removed debug traces and verified both `cross_executor` and `sql_only` variants now pass

**Root cause**

The sync barrier only waited on structural changes (block count, content). An `effort`-only mutation changed neither, allowing `check_invariants` to read the database before the asynchronous UPDATE committed via the file-watcher path.

**Status**

The `effort` property mutation bug is fixed. Two pre-existing flakiness issues remain in the Full variant: a UI spot-check content race at `sut.rs:4241` and a peer-merge rows race in `assertions.rs:55`. Both documented in `HANDOFF_PBT_REMAINING_FLAKINESS.md` with repro commands and ranked hypotheses.
