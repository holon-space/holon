## Optimizing MCP Sync to Avoid Redundant Upserts

**Context**: Claude Code sessions were triggering full table rewrites on every step, with each sync clearing the cache and re-inserting all records via `INSERT OR REPLACE`, even when only a single message was appended.

**What happened**:
- Traced the sync flow in `McpSyncEngine::sync_entity()` to find that `ResourceSync` (used by Claude Code) has no cursor, triggering `cache.clear()` + full re-insert
- Implemented ID-set diffing: compare existing cache IDs against fetched MCP records to identify only genuinely new/deleted/changed records
- Added field-level comparison via `fetched_matches_cached()` to detect when records were actually updated, avoiding unnecessary writes
- Fixed `PartialEq` issues: `_change_origin` column and `type_name` mismatches made every record appear changed; switched to comparing only data fields present in the MCP response
- Added `QueryableCache::get_all_ids()` for lightweight ID-only queries (`SELECT {id_col}`) instead of deserializing full rows
- Implemented two-tier diffing: ID-set comparison first; only deserialize full rows if IDs overlap (potential updates)

**Challenges**:
- **Cache column mismatch**: Cache returned `_change_origin` from `SELECT *`, but MCP-built entities didn't have it, making `PartialEq` always fail. Solved by comparing only fields present in the fetched entity.
- **Type name drift**: Cached entities used prefixed names (`cc_task`), MCP entities used unprefixed (`task`). Solved by using fetched entity's field set as canonical.
- **`EntityUri` hashing**: Initially returned `Vec<String>` from `get_all_ids()`, but upgraded to `Vec<EntityUri>` for type safety and consistent ID handling across the diffing logic.

**Status**: Full optimization realized. For append-only syncs (common case), queries avoid deserialization and execute only `INSERT`; update detection works via content comparison without unnecessary writes.
