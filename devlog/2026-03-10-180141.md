## Diagnosed and Fixed Turso Btree Panic; IVM Data Corruption Remains

**Context**: The GPUI showed only a gray bar with a black void, caused by crashes during Holon's database initialization. Root cause investigation required extracting SQL traces and replaying them against Turso to identify the panic.

**What happened:**
- Extracted 1443 SQL statements from `/tmp/holon-gpui.log` using `@scripts/extract-sql-trace.py`
- Replayed via `turso_sql_replay.rs` and reproduced the panic at statement 1173: `[PageStack::current] current_page=-1`
- Minimized the reproducer from 1191 to **56 statements** — the panic triggered by inserting ~54 rows into a table with a **recursive CTE materialized view**
- Performed `cargo update` on the Turso fork (691e2ec3 → 2dc3885f) — the updated version fixed the btree panic
- Full 1443-statement replay now completes without crashing ✅

**Challenges:**
- The btree panic was a cursor state corruption bug in Turso's IVM path for recursive CTEs — after `clear()` without `push()`, accessing the cursor panicked
- After the panic fix, discovered a secondary **IVM data mismatch**: matview contains 280 rows vs. fresh re-evaluation gets 274 — 6 phantom stale rows with paths that shouldn't exist given the current table state
- The generic diff mechanism required refactoring from hardcoded columns to `EXCEPT`-based comparison to avoid being specific to the `block_with_path` view

**Decisions:**
- Made consistency checking fully generic using `SELECT * FROM a EXCEPT SELECT * FROM b` instead of hardcoded column lists, so future bugs can be caught without modifying `turso_sql_replay.rs`
- Used `--check-after-each` to binary-search for the exact statement triggering the inconsistency (statement 870 in the full trace)

**Status**: Btree panic resolved by Turso fork update. IVM data corruption persists — 6 stale rows in the recursive CTE matview that shouldn't be there based on fresh evaluation; investigating whether this is an incremental maintenance cascade-delete bug or a fresh matview creation issue.
