## Fixing GPUI Tree Rendering to Use Entity Profiles

The main panel in the holon GPUI frontend was rendering as a flat, non-editable list instead of a tree with editable blocks styled by entity profiles. The issue traced back to profile render expressions being lost during the data flow from the backend.

**What happened:**

- Diagnosed that `describe_ui(__root__)` returned empty while the backend correctly rendered `block:root-layout` with proper 3-column layout
- Traced the GPUI `tree` builder and found it produced flat flex columns with no indentation or scrolling
- Discovered the core issue in `shared_render_entity_build` (line 384-411): it hardcodes rendering logic and never resolves entity profiles, falling back to plain `text` nodes for all content
- Found that `BlockRenderCache::get_or_watch()` strips `ResolvedRow.profile` when converting to render context data, losing profile-specific render expressions entirely
- The entity profiles ARE resolved on the backend, but the profile's `RenderExpr` never reaches `render_entity` in the GPUI frontend

**Decisions:**

- Chose a pragmatic approach: inject the profile's render expression into row data as a special `_profile_render` key rather than refactoring the `RenderContext` API
- Will implement a helper `resolved_rows_to_data()` to preserve profile information during data conversion
- Need to update `shared_render_entity_build` to check for and interpret `_profile_render` when available

**Status:** Root cause identified and fix strategy designed; implementation pending in `BlockRenderCache` and the render block builder functions.
