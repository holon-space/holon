## Rigorous Code Review of the Majestic Sauteeing Lollipop Plan

**Context:** A senior developer reviewed an ambitious plan to port Holon's backend engine to WebAssembly and run it in a browser worker, surfacing reactive view updates via a subscription protocol. The review uncovered critical architectural errors, unvalidated assumptions, and underestimated risks.

**What happened:**

- Identified that B3's planned `RenderInterpreter<ViewModel>` doesn't exist — the snapshot path via `ReactiveViewModel::snapshot()` already exists in the codebase, making the 2-day estimate misaligned
- Discovered H3: `EngineState` Mutex held across `block_on()` calls will deadlock once subscription tasks spawn on the current-thread runtime — lock discipline was missing from the plan
- Confirmed H4: `memory_stats` crate reads `/proc/self/status` (not WASI-compatible) and calls `libc::raise()` which will fail to link; B1's native-only list omits it
- Found that `FrontendSession::save_runtime()` has no strategy for WASI environments without writable filesystem paths, violating the project's "no silent degradation" principle
- Validated that `AvailableSpace` will always be `None` in the worker, silently disabling responsive layout variants (H6)
- Noted that A1 introduces `watch_ui → UiEvent` streaming while B3 obsoletes it with `watch_view → ViewModelPatch`, wasting A4's TypeScript bridge work
- Flagged undiagnosed OPFS regression in A2 ("wrote != expected") with unknown root cause — estimate could balloon from 1 day to 1–3 weeks depending on whether it's a WAL race or OPFS truncation quirk
- Identified that multi-tab OPFS hand-off in A5 requires graceful `FileSystemSyncAccessHandle` close handshake, not just one reload

**Challenges:**

- **H13 (signal wakers under current-thread runtime):** Whether `futures-signals` Waker delivery will work between `block_on` calls is unvalidated; needs a proof-of-concept before B2
- **H12 (contenteditable cursor loss):** Dioxus re-rendering on ViewModel updates will clobber cursor position in focused editors; the plan claims "on-diff" patching avoids this but doesn't describe the diffing hook mechanism; C3's 2-day estimate is optimistic (4–5 days realistic)
- **H14 (MessagePort negotiation):** How `watch_view(blockId)` returns a new `MessagePort` to the main thread via napi's `ThreadsafeFunction` is undesigned — protocol needs explicit definition before implementation

**Decisions:**

- B3 should use existing `ReactiveViewModel::snapshot()` path instead of inventing a new `RenderInterpreter<ViewModel>` type
- Lock discipline: extract engine reference from Mutex *before* entering `block_on`, not inside
- For v1, always send full `ViewModel` snapshots on events; defer tree-diffing to v2 after throughput measurement

**Status:** Plan requires significant refinement before starting; critical architectural errors corrected, estimate revised upward from ~27 to ~31 days, and measurement/validation tasks added for H8, H9, H10, H11.
