## Adding Round-Trip Testing Capabilities to MCP Server

The MCP server could screenshot the frontend but couldn't interact with it, limiting testing to read-only inspection. The goal was to make it a fully capable testing mechanism with input simulation.

**What happened:**

- Debugged the `capture_image()` "Failed to copy data" error on macOS — root cause was a window state issue, fixed by restarting the MCP server
- Added diagnostic logging to `capture_image()` to track window properties and dimensions for future troubleshooting
- Explored GPUI's interaction model: discovered that `WindowHandle::dispatch_event()` exists but requires `Send + Sync` compatibility with the MCP server's `DebugServices`
- Investigated storage options: `AsyncApp`, `ForegroundExecutor`, and `WindowHandle` are all `!Send` due to `Rc`/`PhantomData<Rc<>>` constraints
- Designed a channel-based architecture: store `std::sync::mpsc::Sender<InteractionCommand>` in `DebugServices` (which IS `Send + Sync`), with GPUI's foreground task reading from the receiver and dispatching events

**Decisions:**

- Kept diagnostic code in place rather than removing it, for future debugging visibility
- Chose channel-based async communication over attempting to store window handles or executors, since the latter violate `Send + Sync` bounds required by `DebugServices`

**Status:** Architecture designed and ready to implement; interaction dispatch will be built as a separate feature branch using the channel pattern to bridge the MCP server and GPUI's foreground event loop.
