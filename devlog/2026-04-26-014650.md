## Drag-drop handoff audit and IVM root cause diagnosis

Completed a deep audit of the drag-drop wiring branch, closing three open points and diagnosing the root cause of post-move rendering failures.

- **Reverted PRQL hierarchy templates** (`orgmode_hierarchy.prql`, `todoist_hierarchy.prql`). Confirmed these files are test-only fixtures, never loaded at runtime; the canonical per-block render path is `block_profile.yaml`. Added doc comment clarifying this.
- **Verified widget registration is clean** (`shadow_builders/mod.rs:42`). Only one direct `interp.register()` call (column) plus three `register_value_fn()` calls in the extend list; no stray registrations outside this list.
- **Diagnosed root cause of post-move nested rendering bug (#10)**: Not a renderer issue. After `DragDropBlock`, the main-panel matview reports `IVM MATVIEW INCONSISTENCY` and *all three top-level blocks vanish* from the GQL query result—not just the moved block. Root cause is Turso's incremental view maintenance: recursive `CHILD_OF*0..20` joins don't reconcile after `parent_id` mutations. This same issue explains pre-existing `Indent`/`Outdent`/`MoveUp`/`MoveDown` precondition failures (#1b).

**Challenges:**
- Distinguishing drag-drop bugs from pre-existing failures required running the property-based test suite both with and without drag-drop enabled, then checking git history.
- The matview inconsistency was subtle: post-move GQL queries succeeded syntactically but returned incomplete result sets, pointing to a data layer issue rather than a query or renderer problem.

**Decisions:**
- Scoped IVM matview reconciliation bug out of the drag-drop wiring branch—it requires a Turso-level fix and affects multiple block operations beyond drag-drop.

**Status:** Handoff document updated; 4 open points remain (#4 drop UX, #6 extend `build_drop_intent`, #7 GPUI smoke test, #8 weight tuning). IVM matview bug (#10/#1b) escalated as a separate spec.
