## Fixed stale materialized view detection on database open

When a materialized view persisted in the database referenced a dropped column or table, `Database::open` would fail with a misleading "circular dependency" error instead of gracefully handling the stale view.

**What happened**

- **Identified root cause** in `core/schema.rs:1382`: the resolution loop caught all "not found in schema" errors uniformly, deferring both legitimate dependency-ordering issues (view A depends on view B not yet loaded) and permanent breakage (view references nonexistent column). Both look identical, so broken views got deferred forever, exhausting the loop and triggering a false circular dependency error.

- **Implemented classification logic** (lines 1391–1427): After the multi-pass resolution stalls, the code now distinguishes stale views from circular ones. A view is stale if its SQL doesn't reference any other pending view name—it can never succeed regardless of load order. These get logged as warnings and cleaned up; only genuinely circular views (where all pending views depend on each other) trigger an error.

- **Added regression tests** in `test_ivm_stale_matview_reopen.rs`:
  - Stale view with dropped column: table recreated without a referenced column → DB opens, view dropped
  - Stale view with dropped table: source table dropped entirely → DB opens, view dropped  
  - Valid chained matviews: confirms legitimate inter-view dependencies still load correctly

**Challenges**

- Borrow checker: collecting pending view names required a separate HashMap iteration to avoid holding a mutable reference into `populate_materialized_views`'s state while iterating pending views.

**Status**

All tests passing (5 total: 2 new stale view tests, 1 regression test, 2 existing chained matview tests). Database now opens with stale views gracefully cleaned up instead of failing with a misleading error.
