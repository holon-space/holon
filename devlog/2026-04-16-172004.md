## Implementing `PartitionedStatic` reactive variant and progressive sidebar collapse

We needed to fix two related issues: viewport resizes weren't propagating to positional column children (drawers, main panel), and both sidebars would disappear simultaneously at the wrong breakpoint instead of collapsing progressively.

**What happened:**

- Read the existing two-phase column interpretation (Branch A in `columns.rs`) which discovers layout hints but doesn't provide reactive space updates to children
- Added new `ReactiveViewInner::PartitionedStatic` variant in `reactive_view.rs` (~300 LOC) with per-child `(RenderExpr, LayoutHint)` config and a reactive `parent_space` Mutable signal
- Implemented `create_partitioned_driver` that watches `parent_space`, recomputes the Fixed/Flex partition, and re-interprets each child expression with correct slot width
- Updated all pattern-match sites: `space_mutable()`, `layout()`, `stable_cache_key()`, `start()`, `create_driver()`, `snapshot()`
- Wired `columns()` Branch A to create `PartitionedStatic` instead of `StaticCollection` when `available_space` is known
- Fixed `block_profile.yaml` breakpoints: phone (<600px overlay left), tablet (600–1000px overlay right), desktop (≥1000px both shrink) — previously both sidebars squeezed main panel to 49px at 601px

**Challenges:**

- Pattern matching on `&&LayoutHint` in filter closures with float comparisons — resolved by using direct comparisons instead of `matches!` macro with guards
- Scope issues with `LayoutHint` import in driver closure — added module-level import

**Status:** All 147 frontend+layout tests pass. `PartitionedStatic` now handles viewport resizes reactively; sidebars collapse progressively at correct breakpoints without full root rebuilds.
