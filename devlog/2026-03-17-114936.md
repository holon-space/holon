## Removed redundant `_backend_keepalive` from `BackendEngine`

The `_backend_keepalive` field was meant to keep the database actor alive when the DI container is dropped, but `DbHandle` already holds an `mpsc::Sender` that serves this purpose. The field was redundant.

- Removed `_backend_keepalive: Arc<RwLock<TursoBackend>>` field from `BackendEngine`
- Removed `backend` parameter from `BackendEngine::new()`
- Removed `TursoBackend` import from `backend_engine.rs`
- Updated `registration.rs` DI call site to not pass `backend.clone()`
- Removed related doc comments about keepalive semantics

**Challenges:**
- The `DbHandle` already kept the actor alive via its `mpsc::Sender` clone—the extra keepalive was defensive but unnecessary
- Only one `BackendEngine::new()` call site existed, so the refactor was contained

**Status:** Compiles cleanly. Simplified `BackendEngine` by removing the redundant keepalive mechanism.

## Added CDC broadcast to `DbHandle` and refactored `QueryableCache`

The CDC broadcast channel is intrinsically tied to the database connection and was being passed separately throughout the system. Moving it into `DbHandle` provides a unified interface and eliminates the need to thread the broadcast sender separately. `QueryableCache` was also directly accessing `TursoBackend` through an `RwLock`, duplicating work the actor already does for serialization.

- Added `cdc_broadcast: broadcast::Sender<BatchWithMetadata<RowChange>>` field to `DbHandle`
- Added `cdc_broadcast()` accessor method to `DbHandle`
- Removed `cdc_broadcast` field from `BackendEngine`; now delegates to `db_handle.cdc_broadcast()`
- Updated `MatviewManager::new()` to accept `DbHandle` and extract CDC from it
- Replaced `QueryableCache.backend: Arc<RwLock<TursoBackend>>` with `db_handle: DbHandle`
- Converted all `TursoBackend` method calls in `QueryableCache` to equivalent `DbHandle` methods (`execute_ddl`, `query`, `execute`, `transaction`)
- Updated static helper methods in `QueryableCache` to take `db_handle: &DbHandle` instead of `backend: &Arc<RwLock<TursoBackend>>`

**Challenges:**
- The `RwLock` in `QueryableCache` was purely defensive—actual serialization happens at the actor's `mpsc` channel, making the lock wasted contention
- All `TursoBackend` method calls were thin wrappers around `DbHandle` methods, so the refactor was mechanical but required updating ~15+ call sites across multiple methods and static helpers

**Status:** All changes applied. Code compiles and the refactor consolidates database access patterns around `DbHandle`.
