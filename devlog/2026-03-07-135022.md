## Auto-generating Widget Builder Discovery with Proc Macros

The manual list of 28 widget builders in `@frontends/gpui/src/render/builders/mod.rs` was redundant—this information could be derived from the directory structure itself.

**What happened**

- Evaluated options: `automod`, `build.rs`, `inventory`, `linkme`, and custom proc macros
- Chose proc macro approach for best IDE support (rust-analyzer evaluates proc macros, preserving autocomplete/go-to-definition)
- Created `builder_registry!` macro in `holon-macros` that scans directories and emits both `mod` declarations and registration code
- Applied to all five frontends: GPUI, Blinc, Dioxus, WaterUI, and Ply

**Challenges**

- **IDE degradation with `build.rs`**: rust-analyzer doesn't evaluate build scripts, so generated `mod` declarations caused builder files to grey out and lose IDE support
- **Macro flexibility**: Different frontends use different patterns—`RenderInterpreter` registration (GPUI, Dioxus, WaterUI), match-based dispatch (Blinc, Ply), and inline closures. Extended macro to support both `register: Type` and `dispatch: ReturnType` modes
- **Non-builder files**: `prelude.rs` and `operation_helpers.rs` needed `pub(crate)` visibility and have no `build` function, so added `skip: [...]` parameter to exclude them

**Decisions**

- **Proc macro over `automod`**: Though `automod` handles `mod` declarations, only a custom macro could also auto-generate registration/dispatch code
- **Two-mode macro**: `register:` for interpreter-style (emits `register_all` function), `dispatch:` for match-based (emits match arms in a dispatch function)
- **Manual inline closures stay manual**: GPUI's `col`, WaterUI's `render_entity`, etc. are hardcoded because they don't follow the `pub fn build` convention

**Status**

All five frontends now use `builder_registry!` macro. Adding a new builder requires only creating the `.rs` file with `pub fn build(...)`—mod declarations and registrations are auto-discovered. Full IDE support maintained across all builder files.
