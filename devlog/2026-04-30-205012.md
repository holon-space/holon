## Replacing watermark polling with CDC and futures-signals

The `watermark()` and `consumer_position()` functions were driving 50-83x N+1 query patterns due to a 1ms polling loop over the events table. We can eliminate this entirely by subscribing to CDC on a dedicated materialized view and backing the state with `futures-signals` for reactive reads.

**What happened**

- Identified that `mv_events_watermark` (projecting only `id, created_at, processed_by_loro, processed_by_org, processed_by_cache`) is the right CDC target — matviews are guaranteed to fire CDC callbacks, unlike base tables
- Replaced `DashMap<String, AtomicI64>` + `AtomicI64` with `MutableBTreeMap<String, i64>` + `Mutable<i64>` from `futures-signals` for consistency with existing patterns in the codebase
- Structured the CDC consumer as a background task: subscribe first (to not miss events), then bootstrap with one `SELECT MAX(...)` query, then forever update state via monotonic-max on each row change
- Exposed both sync accessors (`watermark()`, `consumer_position()` returning current values) and signal accessors (`watermark_signal()`, `consumer_position_signal()`) so callers can wait reactively instead of polling

**Challenges**

- **Bootstrap race**: must subscribe before running the bootstrap query, not after, to avoid missing inserts in the gap — same principle as PG LISTEN/NOTIFY
- **Turso UPDATE-CDC behavior**: need to verify that UPDATE-only writes through a simple projection matview (no recursive CTE) actually fire CDC; the `tui_split_block_cdc_drop` memory suggests UPDATE drops in recursive contexts, but this view should be safe — test via `mark_processed` then await the consumer signal

**Decisions**

- Use matview + tautology WHERE (`id = id`) rather than base-table CDC, since only matviews are guaranteed to emit reliably
- Defer the `signal_entry` / `signal_map_cloned()` shape until we know which callers need single-key vs. whole-map signals — expose `lock_ref()` on the map plus the signal and let the caller filter

**Status**

Design is solid and low-risk (no schema breakage, no API change). Ready for implementation; one unit test needed to confirm UPDATE→CDC on the matview before merging.
