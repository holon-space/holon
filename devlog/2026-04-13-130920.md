## Code Review: Hardening the Reactive Shadow Index Plan

**Context:** A plan existed to refactor the mutation driver and port it to headless/Flutter, but it contained load-bearing assumptions that would silently fail in production. A senior review identified type mismatches, broken APIs, missing unit test coverage, and architectural gaps.

**What happened:**

- Traced the double-shift bug diagnosis in `shadow_index.rs:119–190` and confirmed it: `patch_block` inserts entries with absolute positions, then the shift loop re-shifts any entry `>= old_range.end`, hitting stale indices when `delta > 0` and new local indices `>= old_len`.
- Discovered **H1 (type mismatch):** `patch_block` takes `&ViewModel`, but `ReactiveEngine::watch` emits `ReactiveViewModel` (wraps collections in `Mutable<Arc<…>>`). Production converts via `ReactiveShell::resolve_snapshot(&App)` — GPUI-specific, not available headless. Headless path needs either a closure-based snapshot resolution or a different subscription model entirely.
- Exposed **H2 (Flutter broken):** Plan assumed `WidgetTester.sendKeyEvent`, which only exists in `flutter_test` harnesses, not in live app runtime. `FlutterMutationDriver` runs via FFI in the live app; needs `ServicesBinding.keyEventManager.handleKeyData` instead or a different approach.
- Found **H3 (no guaranteed reproduction):** Existing PBT seeds don't necessarily hit the crash shape (`delta > 0 && local_idx >= old_len`). The three existing `patch_block` unit tests at `shadow_index.rs:460+` don't cover it either.
- Decoupled the production bug fix (one line, ~15 minutes) from the weeks-long driver refactor by splitting into Phase A and Phase B.
- Flagged async drain-task races (Step 2's watch stream consumer may lag test-thread invariant checks), enigo parallelism hazards, and premature renaming as gotchas.
- Created five validation tasks (V1–V5) for research-level concerns that need prototype work.

**Decisions:**

- **Ship the bug fix independently:** Add a unit test reproducing the double-shift crash, apply the reorder fix to `shadow_index.rs`, and ship Phase A before starting the driver refactor. Decouples production stability from refactor risk.
- **Promote the type-mismatch resolution to a blocking task:** Three candidate approaches (closure-based snapshot, full recursive reconcile replication, or subscription topology redesign) need prototyping before headless Step 3 can land.
- **Validate Flutter at prototype depth:** Test `ServicesBinding.keyEventManager.handleKeyData` against the Flutter side's keybinding model rather than betting on an API that doesn't exist at runtime.

**Status:** Plan rewritten with Phase A (fix + test, ships alone), Phase B (refactor behind validation gates), explicit hypothesis list, and five scoped prototype tasks. Ready for implementation with reduced risk of silent failures.
