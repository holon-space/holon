## Researching and Deploying mcp-rust-proxy for holon-live Resilience

Claude Code's MCP client doesn't auto-reconnect when the backend restarts, requiring manual `/mcp` restarts. The goal was to add a proxy layer that survives backend restarts and reconnects automatically.

- Evaluated `mcp-proxy` (Go) and `mcp-rust-proxy` — mcp-proxy lacks reconnection logic entirely; mcp-rust-proxy designed for spawning servers, not proxying external ones
- Forked and enhanced `mcp-rust-proxy` with support for external MCP servers via new `streamableHttp` transport type and optional `command` field
- Configured proxy to forward to holon-live's streamable HTTP endpoint at `http://127.0.0.1:8520/mcp` with health checks and auto-reconnect on failure
- Updated `~/.claude.json` to connect Claude Code to proxy via stdio instead of direct HTTP, keeping proxy alive across holon-live restarts
- Tested successful tool proxying: all 24 holon-live tools forward through proxy with naming adjustments

**Challenges**

- **Tool name prefixing**: Initial proxy output used `mcp__proxy__server__tool` format; configured per-server `toolNamesPrefix: ""` to reduce duplication, requiring permission rule updates
- **Stale session after backend restart**: Proxy survives restart but holds stale HTTP session — subsequent tool calls return 404. Proxy needs "reconnect-on-error" logic to detect failed requests, tear down stale connection, re-initialize, and retry

**Status**: Proxy deployed and routing traffic successfully, but reconnect-on-error behavior not yet implemented — manual `/mcp` still needed after holon restarts until proxy detects and recovers from stale sessions.
