## Fixing cargo watch infinite rebuild loop with chrome-trace

The `chrome-trace` feature was writing `trace-*.json` files to the working directory, causing `cargo watch` to detect changes and rebuild continuously.

**What happened:**

- Identified root cause: each run generated new `trace-{timestamp}.json` files in the project root
- These files triggered `cargo watch` to restart, creating more trace files in an infinite loop
- Checked `.gitignore` and found `*.log` was already excluded, but trace files weren't
- Added `trace-*.json` and `dhat-heap.json` patterns to `.gitignore`
- Verified SQLite WAL files (`2`, `2-wal`) don't cause issues since `cargo watch` only monitors while the app isn't running
- Added a `watch` recipe to `justfile` supporting any UI frontend with configurable flags

**Decisions:**

- Used `.gitignore` solution instead of `-i` flags on `cargo watch` — simpler and follows principle of least surprise
- Made the recipe flexible to work with all frontends (`holon-{{ui}}` pattern), even though only `gpui`, `blinc`, and `tauri` support `chrome-trace`
- Kept `orgmode-root` as a configurable argument rather than hardcoding paths

**Status:** `just watch` command now works without infinite rebuild loops; trace files properly gitignored and justfile recipe supports multiple frontends.
