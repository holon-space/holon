## Fixed `describe_ui` to Return Initial Data from CDC Stream

The `describe_ui` tool was returning an empty dataset to the view model, resulting in UI descriptions with no data. The first batch from the change-data-capture stream contains all currently existing rows that should be passed to the interpreter.

### What happened

- Identified the bug in `frontends/mcp/src/tools.rs` lines 1215-1234: `describe_ui` was ignoring the `_stream` parameter and passing `vec![]` to `interpret_to_view_model`
- Renamed `_stream` to `stream` and added logic to consume the first batch from the stream
- Extracted `Change::Created` items from the batch to get the initial `StorageEntity` (HashMap<String, Value>) data rows
- Passed extracted rows to `interpret_to_view_model` instead of empty vector
- Verified type compatibility: `StorageEntity` and `DataRow` are both `HashMap<String, Value>`
- Tested the fix by restarting the app and MCP, confirmed `describe_ui` now returns actual widget trees with real entity IDs instead of empty structures

### Root cause

The tool was written with a placeholder `_stream` parameter that was never consumed. The CDC stream's first batch delivers all current rows as `Change::Created` items—this data was being discarded instead of being used to populate the UI description.

### Status

Fixed and tested live — `describe_ui` now correctly returns populated widget hierarchies with actual data from the block being described.
