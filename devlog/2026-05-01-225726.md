## Architectural Review & Plan Refinement for Ordered Materialized Views

**Context:** A previous plan for implementing ORDER BY in materialized views had critical correctness bugs that would cause silent data corruption, in-transaction read anomalies, and undefined behavior in refresh operations. This session performed a senior code review to validate assumptions, uncover gaps, and rewrite the plan into a safer, phased approach.

**What happened:**

- **Validated three major correctness bugs** in the original plan (v4):
  - Storage layout mismatch: `build_composite_keys` stores `[sort_cols..., rowid, data..., weight]` but cursor reads as if logical order — requires explicit storage→logical value permutation
  - Wrong `has_rowid` flag: `IndexInfo.has_rowid: true` + leaf-index layout makes `BTreeCursor::get_index_rowid_from_record` read `weight` instead of `rowid`
  - Broken cancellation in cursor merge: `HashableRow` hashes values only, losing rowid context, causing duplicate old/new rows in in-transaction reads

- **Discovered three additional correctness gaps**: schema entry optimization causing optimizer to emit OpSeekRowid against indexes, REFRESH undefined for ORDER BY matviews, and non-recursive matviews using `circuit.execute()` which ignores ORDER BY/LIMIT entirely

- **Performed quick validation** on secondary claims (eq_only seek behavior, deterministic group rowids, schema reload) and identified a missing `BTreeCursor::new_with_index_info` constructor and `IndexInfo::new_from_index` losing nulls_order metadata

- **Reorganized work into five phased slices**, each ending with a passing test, deliberately separating single-column basics, multi-column/nulls/desc, limit/count, recursive safety, and testing — avoiding the "ship everything at once" trap that caused bugs

**Decisions:**

- **Slice-based delivery** over monolithic implementation to catch each layer's correctness before building on it
- **Storage→logical decoupling** explicit in cursor layer rather than implicit in layout — makes invariants visible
- **Schema safety**: use `has_rowid: !view.has_order_by` to prevent optimizer misuse

**Status:** Plan v5 rewritten with integrated critique; ready for implementation starting with single-column ORDER BY end-to-end slice (DDL flag → IndexInfo builder → cursor wiring + storage permutation → green test).
