## ReactiveEngine Migration + BuilderServices Trait Implementation

Completed the major refactor of the reactive layer to eliminate the god-object `RenderPipeline` and consolidate reactive capabilities into a narrowly-scoped `BuilderServices` trait.

### What Happened

- **Defined `BuilderServices` trait** in `reactive.rs` with 5 methods: `get_block_data`, `resolve_profile`, `compile_to_sql`, `start_query`, `widget_state`. Implemented on `ReactiveEngine` with proper delegation to `FrontendSession`.

- **Rewrote `RenderContext`** to hold `Arc<dyn BuilderServices>` instead of `RenderPipeline`. Removed `session()`, `engine()`, `runtime_handle()` accessors. Deleted entire `RenderPipeline` struct.

- **Updated shared builders** (`render_interpreter.rs`, `block_ref.rs`, `transclude.rs`) to call `ctx.services.*()` instead of `ctx.session()` / `ctx.engine()` chains.

- **Fixed GPUI frontend** (`lib.rs`, `entity_registry.rs`): removed `make_interpret_fn` + `OnceLock<RenderPipeline>`. Used simpler pattern: `interpret_fn` closure captures a `Weak<ReactiveEngine>` to break the circular reference, upgraded on-demand during reactive updates.

- **Updated frontends** (TUI, Ply, Blinc): replaced `AppState`/`BlockWatchRegistry` with `Arc<ReactiveEngine>` directly. Updated render function signatures to accept engine instead of pipeline.

- **Added GPUI-specific helpers** to `GpuiRenderContext` with accessor methods for `session()` and `runtime_handle()` that delegate through the engine, allowing GPUI builders to dispatch mutations and run async work.

- **Verified PBT integration tests** still compile (already using correct patterns with `RenderContext::headless`).

### Challenges

- **Circular dependency in ReactiveEngine**: Engine needs an `interpret_fn` callback during construction, but the callback needs the engine as `BuilderServices`. Solved by storing a `Weak<Self>` inside the engine and upgrading it lazily during reactive signal updates—avoids the `Arc` cycle at construction time.

- **GPUI builders need mutation dispatch**: GPUI-specific builders like `operation_button` call `ctx.session().execute_operation()` for mutations. The narrow `BuilderServices` trait doesn't include this. Added dual-path: `BuilderServices` for read-only operations (resolve_profile, get_block_data), pass `FrontendSession` directly through `GpuiRenderContext` for mutations.

### Decisions

- **Trait-based capability scoping**: Builders (shared + GPUI-specific) see only what they need via `BuilderServices`. Prevents accidental coupling to `ReactiveEngine` internals. This is a capability-based security pattern that scales to multiple frontend implementations.

- **Method signatures require `&Arc<Self>`**: `interpret()`, `snapshot()`, `watch()` take `self: &Arc<Self>` to pass the engine itself as `Arc<dyn BuilderServices>` to child contexts. This is idiomatic Rust for trait-object passing without storing `Weak` pointers.
