## Fixed UiWatcher command channel lifecycle across all frontends

The `BlockRenderCache` was dropping the `WatcherCommand` sender prematurely, killing structural CDC for nested blocks. This bug existed in similar forms across all frontends (GPUI, Blinc, WaterUI, Ply). The root cause was that `watch_ui()` returned a raw tuple `(Receiver, Sender)`, forcing every caller to manually keep both ends alive—an easy invariant to break.

**What happened:**

- Diagnosed the initial bug in `holon-frontend/src/render_context.rs:70`: `_cmd_tx` was dropped at the end of the `match` arm after spawning the CDC listener, closing the channel and causing the UiWatcher to exit
- Moved `_cmd_tx` into the spawned async task to keep it alive alongside the receiver
- Identified the same pattern bug in GPUI `main.rs`, Blinc `main.rs`, and WaterUI—all dropped the command sender prematurely
- Extracted the pattern into a new `WatchHandle` struct in `holon-api/src/streaming.rs` that owns both channel ends
- Updated `watch_ui()` API to return `WatchHandle` instead of a tuple, making the lifetime invariant enforced by the type system
- Refactored all five frontends (GPUI, Blinc, Ply, Dioxus, WaterUI) and Flutter FFI to use `WatchHandle`, adding `into_parts()` method for Flutter's split-channel pattern
- Updated all integration tests to work with the new API
- Updated the error message in `ui_watcher.rs` to remove the misleading "Dart side" reference

**Decisions:**

- `WatchHandle` owns both channel ends rather than splitting them at the API boundary, making lifetime management explicit and correct by default
- Provided `into_parts()` for special cases (Flutter FFI) that need to forward the receiver to a `StreamSink`

**Status:** All crates compile cleanly. The structural CDC bug is fixed across all frontends, and future misuse of the API is prevented by the type system.
