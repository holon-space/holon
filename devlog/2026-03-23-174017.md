## Unified Config Architecture: Merged HolonConfig and InstanceConfig

Settings were being lost from `~/.config/holon/holon.toml` because two separate structs (`HolonConfig` and `InstanceConfig`) both read/wrote the same file without coordinating.

### What happened

- **Root cause analysis**: `InstanceConfig::save()` serialized only its own fields (`hooks`, `todoist`, `ui`, `preferences`), clobbering keys from `HolonConfig` like `[orgmode] root_directory` and `db_path` on every UI setting change
- **Identified duplication**: Both structs had overlapping fields (`hooks`, `todoist`, `ui`) with no architectural reason to be separate — `HolonConfig` used `clap::Parser` for startup config loading, while `InstanceConfig` was added later as a simpler serde struct for runtime mutations
- **Merged into single source of truth**: Moved `InstanceConfig` logic into `HolonConfig`:
  - Replaced `TodoistPreferences` stub with full `TodoistConfig` from `holon-todoist` 
  - Added `preferences: HashMap` field for runtime storage
  - Implemented `load_runtime()` / `save_runtime()` using read-merge-write pattern
  - Moved `WidgetState` struct from `instance_config.rs`
- **Simplified `FrontendSession`**: Removed dual `config_state` field; `holon_config: Mutex<HolonConfig>` now handles both startup and runtime configuration
- **Removed `instance_config.rs` entirely** and updated dependent code (`frontend_module.rs`, GPUI frontend)

### Decisions

- **Single config file, single struct ownership**: `HolonConfig` is now the sole reader/writer of `holon.toml`, eliminating the possibility of field loss through deserialization
- **Kept `UiConfig` shape** (with `Option<bool>` for `glass_background`) instead of changing to `UiSettings`, to avoid cascading changes through GPUI frontends

### Status

All holon crates compile cleanly. `HolonConfig` is now the unified config struct; runtime preference changes are persisted safely without losing other keys.
