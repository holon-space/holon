# holon-dioxus-web

Browser frontend for Holon. Renders the Holon ViewModel tree with
[Dioxus](https://dioxuslabs.com/) 0.7. The entire backend (engine, Turso,
reactive system, parsers) runs in a dedicated Web Worker compiled to
`wasm32-wasip1-threads`; the main thread only owns the DOM and a thin RPC
bridge.

## Quick start

```bash
cd frontends/dioxus-web
npm install                         # installs `ws` for the dev server
node serve.mjs --build              # one-shot trunk build + serve
# or:
node serve.mjs --watch              # trunk watch + serve
```

Open <http://127.0.0.1:8765/>. You should see "ready (≈1.3 s)" and a
"Welcome" entry in the left sidebar.

### Prerequisites

- `trunk` 0.21+ (`brew install trunk`)
- Node 20+
- `wasm32-unknown-unknown` and `wasm32-wasip1-threads` Rust targets:
  ```bash
  rustup target add wasm32-unknown-unknown wasm32-wasip1-threads
  ```
- `@napi-rs/cli` 3.1+ — already vendored in
  `frontends/holon-worker/node_modules`

### Building the worker wasm

The worker wasm is **not** checked in. It's produced by `napi build` and
served directly out of `target/`:

```bash
cd ../holon-worker
EMNAPI_LINK_DIR="$(pwd)/node_modules/emnapi/lib/wasm32-wasi-threads" \
  ./node_modules/.bin/napi build \
  --features browser --profile dev --platform \
  --target wasm32-wasip1-threads --no-js \
  --manifest-path ./Cargo.toml --output-dir .
```

`napi`'s `copyArtifact` step always errors out at the end (looks for
`target/.../dev/` instead of `target/.../debug/`). **That error is
harmless** — `serve.mjs` reads from `target/` directly. Just reload the
page after the build finishes.

`--profile release-official` (LTO + `opt-z`) produces a much smaller wasm
but takes ~15 min. Use `dev` during iteration.

### Wasm sizes by profile

| profile            | size   | use case            |
| ------------------ | ------ | ------------------- |
| `dev` / `debug`    | ~640 MB | local iteration only |
| `release`          | ~?      | not currently built |
| `release-official` | ~21 MB  | what you'd ship     |

`serve.mjs` picks whichever build is newest by mtime. Override with
`HOLON_WORKER_WASM=/abs/path node serve.mjs` if you need to force a
specific build. If no wasm exists, the page returns a 404 with the exact
napi command to run.

## Architecture

```
┌────────────────────────┐         postMessage          ┌──────────────────────┐
│  Dioxus page (main)    │  ─────────────────────────▶  │  holon-worker Web    │
│  holon-dioxus-web      │                              │  Worker (wasi-threads)│
│                        │  ◀───  {kind, handle, json}  │                       │
│  ┌─────────────────┐   │                              │  BackendEngine +      │
│  │ WorkerBridge    │   │                              │  ReactiveEngine       │
│  │ (bridge.rs)     │   │                              │  (current-thread rt)  │
│  │   onmessage ──▶ │   │                              │                       │
│  │   call(kind, a) │   │                              │                       │
│  │   on_snapshot   │   │                              │                       │
│  └─────────────────┘   │                              └──────────────────────┘
│                        │
│  BRIDGE (thread_local) │
│  view_model: Signal    │     typed ViewModel deserialized at the boundary
│  RenderNode            │     (no JSON probing on the main thread)
│  ├─ BlockRefNode       │◀── each owns its own watch subscription
│  │  └─ inner_vm Signal │
│  └─ EditableTextNode   │
│     └─ EditorCell      │
└────────────────────────┘
```

All Holon crates live inside the worker. The Dioxus side speaks JSON over
`postMessage` and never imports a Holon Rust type from the engine — only
the typed `ViewModel` enum from `holon-frontend`.

### Why a Web Worker?

`wasm32-wasip1-threads` requires `SharedArrayBuffer`, which means the
page must be cross-origin isolated (`COOP: same-origin`,
`COEP: require-corp`). It also blocks the main thread on every wasm
call, which is forbidden on the main thread. Putting the engine in a
Worker solves both problems and gives us a clean RPC boundary.

### Render pipeline

The renderer is generated by the shared
`holon_macros::builder_registry!` macro — the same one GPUI uses. Each
`ViewKind` variant has a one-file builder under `src/render/builders/`
with the signature:

```rust
pub fn render(field_a: &T, field_b: &T, ctx: &DioxusRenderContext) -> Element
```

The macro dispatches on the `ViewKind` variant, calls the matching
builder, and returns the `Element`. Adding a new widget = adding a new
file in `builders/` whose function arg names match the variant fields.

`DioxusRenderContext` is currently zero-sized — Dioxus doesn't need the
GPUI bounds registry, focus scope, or handles. It exists so the macro has
a stable type to thread through and so cross-cutting state (theme,
preferences) has a home if needed.

## File layout

```
frontends/dioxus-web/
├── Cargo.toml          # wasm32-only, in workspace `exclude` list
├── serve.mjs           # combined trunk + worker dev server (port 8765)
├── package.json        # npm deps for serve.mjs (`ws`)
├── README.md           # you are here
├── index.html          # trunk entry
└── src/
    ├── main.rs         # boot, root layout watch, tick pump
    ├── bridge.rs       # WorkerBridge RPC + snapshot routing
    ├── editor.rs       # uncontrolled contenteditable + flat-offset cursor
    └── render/
        ├── mod.rs      # EntityContext, DioxusRenderContext
        └── builders/
            ├── mod.rs            # builder_registry! invocation
            ├── prelude.rs        # per-builder re-exports
            ├── util.rs           # value_to_display helper
            ├── block_ref.rs      # owns per-cell watch subscription
            ├── editable_text.rs  # uses EntityContext to mutate
            ├── …                 # ~31 widget builders
```

The seed that populates the in-memory database on boot lives in
`../holon-worker/src/seed.rs` — the worker can't pull
`holon-orgmode` (which depends on `notify`, no wasm backend), so the
default layout is hand-written SQL. Edit that file to change what shows
up on first load.

## Dev server (`serve.mjs`)

| URL prefix                        | served from                              |
| --------------------------------- | ---------------------------------------- |
| `/`                               | `dist/` (Trunk-built Dioxus)             |
| `/web/*`                          | `../holon-worker/web/`                   |
| `/node_modules/*`                 | `../holon-worker/node_modules/`          |
| `/holon_worker.wasm32-wasi.wasm`  | newest of `../holon-worker/target/wasm32-wasip1-threads/{debug,release,release-official}/holon_worker.wasm` |
| `/mcp`, `/mcp/*`                  | HTTP proxy → native MCP relay on `:3002` |
| `/mcp-hub`                        | WebSocket bridge native relay ↔ browser  |

All responses carry `COOP: same-origin` + `COEP: require-corp` so
`SharedArrayBuffer` works.

`PORT=9000` and `RELAY_PORT=4000` env vars override the defaults.

## Browser MCP relay

Lets Claude Code call MCP tools against the **in-browser WASM engine** instead of a
local native instance. The full call path:

```
Claude Code
  → http://localhost:PORT/mcp          (serve.mjs proxies to RELAY_PORT)
     → holon-mcp (relay mode, RELAY_PORT)
        → WebSocket /mcp-hub?role=native
           ↕ serve.mjs hub
        ← WebSocket /mcp-hub?role=browser  (Dioxus page)
           → bridge.call("engineMcpTool", …)
              → holon-worker WASM → real in-browser Turso DB
```

Tool schemas stay exclusively in `frontends/mcp/src/tools.rs` — `BrowserRelayServer`
calls the existing `tool_router_ui() + tool_router_backend()` associated functions and
never duplicates `#[tool]` annotations.

```bash
    --features browser --profile release-official --platform \
```


### Startup sequence

```bash
# Terminal 1 — Trunk build + dev server
cd frontends/dioxus-web
PORT=8766 node serve.mjs --build      # or --watch for live rebuild

# Terminal 2 — native relay (browser relay mode)
HOLON_BROWSER_RELAY_URL=ws://127.0.0.1:8766/mcp-hub \
RELAY_PORT=3002 \
cargo run -p holon-mcp

# Terminal 3 — open browser, wait for [mcp-hub] browser connected, then test
open http://127.0.0.1:8766
```

Watch Terminal 1 for:
```
[mcp-hub] native connected
[mcp-hub] browser connected
```

Both sides reconnect automatically on restart (no page reload or relay restart needed).

### Unsupported tools in browser mode

These tools return a `"not supported in browser worker"` error — they require
`LoroDocumentStore` or GPUI APIs not wired into the worker:

- `inspect_loro_blocks`, `diff_loro_sql`, `list_loro_documents`
- `read_org_file`, `render_org_from_blocks`

All other tools (SQL/PRQL queries, UI introspection, operations, commands, Petri-net
ranking) work against the live in-browser DB.

## Wire format lockdown

`holon_api::Value` is `#[serde(untagged)]`. Primitives round-trip as
bare JSON: `Value::String("x") → "x"`, `Value::Integer(42) → 42`,
`Value::Float(3.5) → 3.5`, `Value::Null → null`. There is **NO**
`{"Text": …}` or `{"String": …}` wrapping.

Lockdown tests live in `crates/holon-api/src/lib.rs`:

- `tests::value_serde_wire_format_is_untagged`
- `tests::value_map_params_round_trip_untagged`

Both fail loudly if anyone adds `#[serde(tag = "…")]` to `Value`.

## Gotchas

1. **Don't `cargo check --workspace`.** Both `frontends/dioxus-web` and
   `frontends/holon-worker` are in the workspace `exclude` list because
   they're wasm32-only. Use:
   ```bash
   cd frontends/dioxus-web && cargo check --target wasm32-unknown-unknown
   ```

2. **Workspace inheritance is broken inside excluded packages.** An
   excluded package can't use `foo.workspace = true`. `Cargo.toml`
   pins `tracing`, `futures`, `serde_json` directly and explains why.

3. **The worker runs on a single dedicated current-thread tokio
   runtime.** Nothing advances unless someone calls `block_on`. The
   tick pump in `main.rs::App` drives it at ~60 fps by calling
   `engineTick(10)` in a loop. Without this, subscriptions emit their
   initial value and then go silent.

4. **`contenteditable` body must be empty in the RSX.** Dioxus diffing
   stomps native cursor state on every render if you put `"{content}"`
   in the body. `EditorCell` uses `use_effect` + `set_text_content`
   instead, and skips the overwrite entirely when the element is
   focused.

5. **napi-rs on emnapi/wasi delivers TSFN tuples as JS arrays.**
   `worker-entry.mjs` unwraps single-element arrays before routing. If
   you change the `(String,)` tuple in `backend::watch_view::tsfn`, the
   worker-entry unwrap code needs to change too.

6. **Worker console ≠ page console.** The Dioxus page cannot see
   `[wasm] …` stdout/stderr from the worker. `bridge.rs` forwards
   worker `error` events but not regular `console.log`. For worker
   tracing during debugging, navigate to
   <http://127.0.0.1:8765/web/index.html>, which hosts the worker
   directly.

7. **Don't put display-bound metadata in `properties` JSON blindly.**
   The `block` table has real columns for `id`, `parent_id`, `name`,
   `content`, `content_type`, `source_language`, `source_name`,
   `created_at`, `updated_at`, `sort_key`. PRQL queries that filter on
   any of these are filtering on the column, not on `properties[name]`.
   Properties JSON is only for things that aren't first-class.

8. **The MCP-hub WebSocket errors in the page console are
   pre-existing.** They come from the Dioxus boot path trying to
   connect a browser-side mcp-hub client that the dev server doesn't
   accept yet. Don't chase these when triaging — they have nothing to
   do with the rendering pipeline.

## Known limitations

- **Layout always degraded.** The title bar shows `[layout: degraded
  — AvailableSpace=None in worker]`. The worker has no viewport
  dimensions, so container queries like `if_space(600.0, …)` always
  take the fallback branch. Fix: pipe page viewport into the worker
  via `engineSetViewport(w, h)` on mount + resize.

- **Editor `Enter` key is silently swallowed.** `editor.rs::EditorCell`
  prevents default but doesn't dispatch a block split. A real
  implementation should emit a `block.split` operation.

- **Debounced dispatch + blur racing.** If the user types, blurs, and
  the 50 ms debounce hasn't flushed yet, the next snapshot can clobber
  pending text. Fix: flush debounce on blur.

- **`engineExecuteSql` returns `()`** instead of an affected-row count.
  `DbHandle::query` doesn't expose `changes()`; needs a new accessor
  if we want "X rows inserted" UI.

- **Dev wasm is 640 MB.** Fine for localhost but `release-official`
  (~21 MB) is what you'd ship.
