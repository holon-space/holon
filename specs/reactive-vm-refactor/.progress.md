## Learnings
- Holon's persistent-node + `Mutable<T>` architecture maps closest to futures-signals/haalka and Floem -- per-node signal ownership, not VDOM diffing
- No exact prior art for "macro-generated builder -> macro-generated InterpretFn closure" transition; closest is haalka's ECS self-describing entities
- Template switching via shared `Mutable<RenderExpr>` broadcast is novel -- most frameworks (SolidJS, Leptos, Floem) destroy+recreate child nodes on template change
- Signal diamond problem: simultaneous data+expr changes fire InterpretFn twice (benign but wasteful); `map_ref!` would batch but adds complexity
- `set_template()` only changes RenderExpr, NOT CollectionVariant -- layout-changing mode switches (table->tree) still need full rebuild via stable_cache_key change
- GPUI's view_mode_switcher builder currently bypasses reactive pipeline (full interpret+slot replacement on click instead of set_template)
- Shadow builders with structural side effects (block_ref, expand_toggle, columns, live_query) cannot be eliminated -- only the props-extraction portion can become InterpretFn
- view_mode_switcher click handler needs ReactiveView reference -- three options: node field, render context, or click closure capture; simplest is click closure capture to avoid bloating ReactiveViewModel for all nodes
- Engine caches (expand_state_cache, view_mode_cache) have exactly 1 consumer each in ReferenceState -- ui_state() reads both, expand_toggle writes expand, view_mode_switcher writes mode. Removal needs ui_state() bridge analysis
- subscribe_props_signals clears ALL watchers on VecDiff::Replace but does nothing for InsertAt/Push -- new items get no props watcher until next Replace
- NEXT_VIEW_ID at reactive_view.rs:157 confirmed dead -- zero references outside definition
- Cleanup step (Epic 6) is safe to do first -- zero dependencies, removes noise before structural work
- Epics 2+3 (resolve_props + macro refactor) are tightly coupled -- resolve_props defines the function signature that generate_interpret_fn must produce

- GPUI VMS click handler can walk `slot_content.collection` to find the `Arc<ReactiveView>` — no new fields needed on ReactiveViewModel
- `ui_state()` `is_expanded` entry has zero production consumers in entity profiles — safe to remove with expand_state_cache
- `view_mode_cache` becomes unnecessary once `set_template()` is wired — the template_mutable IS the active mode state
- `push_down_children` already preserves `expanded: Option<Mutable<bool>>` across structural rebuilds — cache was a safety net, now redundant
- spacer is classified as props_only despite setting LayoutHint::Fixed — LayoutHint is consumed at build time by the parent, not reactively
- ~13 builders (not 15-20 as estimated in research) are genuinely props_only eligible after careful analysis
- `widget_builder!` macro's `generate_extraction` + `generate_auto_body` together = the InterpretFn body. Can split with no semantic changes.
- For intra-variant detection, comparing the top-level function name of the mode template expression suffices — no need for deep analysis
- The `item_template` passed to `set_template()` must be extracted FROM the collection expression (e.g. table's item_template arg), not the collection expression itself
- `generate_resolve_props_body` shares extraction logic with `generate_extraction` but tracks positional_idx independently — Collection/Expr params don't consume positional slots
- For auto-body widgets, `build` delegates entirely to `resolve_props_from_args` (no separate extraction) to avoid duplicate unused variable warnings
- For custom-body widgets, `build` keeps full `generate_extraction` (including Collection/Expr) + user body unchanged; `resolve_props_from_args` is a separate sibling function
