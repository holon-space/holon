### 2. CLEAN-UP: `is_template_arg` replacement (Step 7 of plan)

The hardcoded `is_template_arg()` in `crates/holon-api/src/render_eval.rs:369` lists template arg names manually. The new `is_template_arg_from_metas()` in `widget_meta.rs` can replace it — BUT only for builders that declare params (auto-body mode, 8 builders). The 20 raw-mode builders have empty param lists in WIDGET_META.

**Two paths:**
- **(A) Hybrid:** Keep `is_template_arg()` as fallback, merge with metas. Template args like `item_template`, `item`, `header`, `sortkey`, `parent_id`, `states` are inherently template args regardless of which builder uses them.
- **(B) Enrich raw builders:** Add param declarations to raw-mode WIDGET_META. This requires extending the `widget_builder!` macro to accept `raw(params(...))` syntax, or manually adding `WIDGET_META` with params.

**Recommendation:** Path (A) — the hardcoded list is small and stable. Merge it with metas for extra coverage, keep as definitive source.

WaterUI also has its own duplicate `is_template_arg()` in `frontends/waterui/src/render/interpreter.rs:126` that should be deleted in favor of the one in `holon-api`.

## Bigger Architectural Work Remaining

### 5. Lazy Expansion Backend (Step 5 of plan — NOT started)

The frontend-facing types are in place (`LazyChildren`, `CollectionId`, `UiEvent::CollectionUpdate`), but the backend mechanism doesn't exist yet:

**What's missing:**
- `ExpansionContext` struct in `ui_watcher.rs` — stores the template + data rows + render context per `CollectionId`
- `WatcherCommand::ExpandRange { collection_id, offset, count }` — new command variant
- `expand_range()` handler in `UiWatcher::run()` — interprets template for the requested row window, sends `CollectionUpdate` event
- `CollectionId` allocation — a counter in `UiWatcher` that mints new IDs during shadow interpretation

**When this matters:** Only when a single watch produces hundreds+ of items. Currently everything is fully materialized. This is an optimization, not a correctness issue.

### 6. Frontend Migration: ViewModel as Universal Contract (Step 6 — NOT started)

This is the big architectural payoff. Currently:
```
RenderExpr ──→ shadow interpreter ──→ ViewModel ──→ Flutter (via FFI)
RenderExpr ──→ GPUI interpreter   ──→ Div          (parallel, duplicated)
RenderExpr ──→ TUI interpreter    ──→ TuiWidget    (parallel, duplicated)
```

Target:
```
RenderExpr ──→ shadow interpreter ──→ ViewModel ──→ Flutter (via FFI)
                                                  ──→ GPUI (ViewModel → Div)
                                                  ──→ TUI  (ViewModel → TuiWidget)
```

**Current frontend state:**
| Frontend | Uses RenderInterpreter? | Produces | Migration effort |
|---|---|---|---|
| **Flutter** | No (consumes ViewModel via FFI) | Already done | — |
| **GPUI** | Yes (`RenderInterpreter<Div, BoundsRegistry>`) | `Div` | Medium — rewrite builders as `ViewModel → Div` mappers |
| **TUI** | Yes (`RenderInterpreter<TuiWidget>`) | `TuiWidget` | Medium — same pattern |
| **Blinc** | No (ad-hoc `interpret()` function) | `Div` | Medium — replace ad-hoc interpreter |
| **Ply** | No (ad-hoc) | `PlyWidget` | Medium |
| **WaterUI** | No (ad-hoc + duplicated `resolve_args()`) | `AnyView` | Medium + delete duplicate code |

**How to migrate one frontend (e.g., GPUI):**
1. In the GPUI crate, consume `ViewModel` instead of calling `RenderInterpreter::interpret()`
2. The GPUI `builders/` directory becomes a set of `fn render(node: &ViewModel, ctx: &GpuiCtx) -> Div` functions
3. `builder_registry!` generates a `render_node()` dispatch that matches `NodeKind` variants → typed render calls
4. Remove GPUI's `RenderInterpreter<Div>` and arg extraction — all arg extraction is now in the shadow interpreter
5. Wire `expand_range()` to scroll events in collection builders (for lazy expansion)

**The plan's Step 4 describes this:** `builder_registry!` should grow a `dispatch: NodeKind` mode that generates:
```rust
pub fn render_node(node: &ViewModel, ctx: &GpuiCtx) -> Div {
    match &node.kind {
        NodeKind::Text { content, bold } => text::render(content, *bold, ctx),
        NodeKind::List { children } => list::render(children, ctx),
        _ => fallback::render(node, ctx),
    }
}
```

**Recommendation:** Migrate GPUI first (it's the most actively developed desktop frontend), then TUI. Blinc/Ply/WaterUI can follow later or be deprecated.

### 7. `builder_registry!` Auto-generates `all_widget_metas()` (Step 3 of plan — partial)

Currently `all_widget_metas()` is manually maintained in `shadow_builders/mod.rs`. Once all builders use `widget_builder!`, `builder_registry!` can scan for `WIDGET_META` consts and auto-generate the function. This requires extending `builder_registry.rs` to emit:
```rust
pub fn all_widget_metas() -> Vec<&'static holon_api::WidgetMeta> {
    vec![&badge::WIDGET_META, &block::WIDGET_META, ...]
}
```

Since all 28 builders already export `WIDGET_META`, this is straightforward.

### 8. NodeKind Generation from `widget_builder!` Signatures (Step 3 of plan — NOT started)

The plan envisions `builder_registry!` generating the `NodeKind` enum from builder signatures, eliminating the hand-maintained enum in `view_model.rs`. Currently:
- `NodeKind` is hand-written with ~35 variants
- Shadow builders use backward-compatible constructors (`ViewModel::leaf()`, `::collection()`, etc.)

**To generate NodeKind:**
1. `builder_registry!` reads each builder module's `WIDGET_META`
2. Generates `NodeKind` variants from param types
3. Generates typed constructors
4. Shadow builders produce `NodeKind::Badge { label }` directly instead of `ViewModel::leaf("badge", Value::String(label))`

**This is lower priority** — the current hand-maintained enum works fine, and the backward-compat constructors bridge the gap. Pursue this if/when adding new widgets becomes frequent enough that maintaining the enum manually is painful.
