# Property-Based Testing (PBT) Strategy

Holon uses property-based testing as the primary quality gate. The PBT generates random sequences of operations (mutations, queries, navigation) and verifies invariants after each step.

## Golden Rule

**Never add new PBT tests. Only use `general_e2e_pbt.rs`.**

If a bug isn't reproduced by the PBT, the fix is to make the PBT and production **more similar** (same code path, same data flow), so the PBT can reproduce it. This keeps testing coverage concentrated.

## State Machine Pattern

The PBT uses `proptest`'s state machine API:

```
proptest state machine:
  state: PbtPhaseState (before/after setup, ready to mutate)
  transitions: Vec<E2ETransition>  (generated by proptest)
  invariants: checked after each transition
```

`run_phased_pbt()` drives the state machine. Each step:
1. Apply `E2ETransition` to both real system (`E2ESut`) and `ReferenceState`
2. Check all invariants against both
3. Fail fast on first invariant violation

## ReferenceState

The in-memory reference model is the ground truth. It's a pure Rust struct with no async or database. After each transition, the real system's state is compared against it.

The reference model tracks:
- All blocks with their fields
- Document hierarchy
- Navigation state (current focus, history, regions)
- Active watch specs per region
- Render expressions per render source block

This design means the reference model is the executable specification.

## Invariants (inv1–inv10)

Invariants checked after each transition:

| Invariant | What it checks |
|-----------|---------------|
| inv1–5 | Block structure: parent-child consistency, no dangling references, sort keys valid |
| inv6 | Navigation state matches reference model |
| inv7 | Org round-trip: write org → re-parse → blocks match original |
| inv8 | Displayed blocks match `expected_focus_root_ids()` from reference |
| inv9 | Query results match reference model evaluation |
| inv10a | Error count in ViewModel is zero |
| inv10b | (retired) |
| inv10c | Error count hard assert |
| inv10d | Root widget matches expected `RenderExpr` |
| inv10e | Entity ID set from display ⊆ reference model IDs |
| inv10f | Decompiled data matches query data from reference model |

## Transition Budget

To prevent state explosion, transitions have budgets:
- `StartApp` — exactly once, always first
- Structural mutations (create, delete, indent) — max N per run
- Write operations — limited count
- Navigation — limited count

`transition_budgets.rs` (behind `otel-testing` feature) tracks per-transition counts for OpenTelemetry export.

## Phased Execution

`pbt_setup()` → `pbt_step() × N` → `pbt_teardown()`

`pbt_step_confirm()` — waits for the system to stabilize after a mutation (CDC to propagate, matviews to update) before checking invariants.

`run_pbt_with_driver_sync_callback` — variant that invokes a callback (e.g., GPUI user driver) after each step, for UI-integrated testing.

## Always Tee Before Filtering

```bash
# Correct: preserve all output for cross-reference
cargo nextest run general_e2e_pbt 2>&1 | tee /tmp/pbt.log | grep FAIL

# Wrong: filtering directly loses the full context
cargo nextest run general_e2e_pbt 2>&1 | grep FAIL
```

Non-deterministic tests generate different UUIDs each run. Without `tee`, you can't cross-reference a failure with its context.

## Regression Files

Failed proptest cases are saved to `.pbt-regressions` files:
- `crates/holon/src/api/loro_backend_pbt.pbt-regressions`
- `crates/holon/src/api/sync_pbt.pbt-regressions`

Proptest replays these on every run before generating new cases. This prevents regressions from being reintroduced.

## Running Tests

```bash
# Full PBT run
cargo nextest run -p holon-integration-tests general_e2e_pbt 2>&1 | tee /tmp/pbt.log

# With coverage
cargo llvm-cov --test general_e2e_pbt -p holon-integration-tests --html --output-dir target/coverage-report
```

Test runner config: `Nextest.toml`. Notable: `test-threads = 1` for all integration tests (real database).
